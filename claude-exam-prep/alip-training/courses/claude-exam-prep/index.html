Agentic Architecture & Orchestration

Domain 1 — The heaviest domain. Master loops, multi-agent coordination, hooks, and session management.

What is an Agentic Loop?

An agentic loop is the pattern that turns Claude from a single-response chatbot into an autonomous agent that can take multi-step actions in the real world.

Without an agentic loop, Claude answers one question and stops. With an agentic loop, Claude can look up a customer, check their order, decide to issue a refund, and escalate to a human — all in a single conversation, making decisions at each step based on what it learns.

Without Agentic Loop

User asks a question → Claude responds → Done. One turn. No tools. No follow-up actions. Just text in, text out.

With Agentic Loop

User asks a question → Claude calls a tool → sees the result → calls another tool → reasons → decides → acts → responds when truly done.

The Big Idea

The agentic loop is your code, not Claude's. You write the loop that sends requests to Claude, inspects the response, executes any tools Claude asks for, feeds results back, and repeats. Claude decides what to do; your loop handles how it gets done. The stop_reason field in every response is the signal that drives everything — it tells your loop whether to keep going or stop.

Spawning Subagents

The Task tool spawns subagents. The coordinator's allowedTools must include "Task". Each subagent gets an AgentDefinition with its own system prompt and tool restrictions.

Subagents do not automatically get the coordinator's conversation history. You must explicitly pass all relevant context in the subagent's prompt. This is the #1 gotcha on the exam.

When subtasks are independent (no subagent needs another's output), emit multiple Task tool calls in a single coordinator response to run them in parallel. Sequential execution for independent tasks wastes time.

Enforcement vs. Guidance

Context: This is about building agents with the Claude Agent SDK

The Claude Agent SDK is Anthropic's framework for building autonomous agent applications — customer support bots, research systems, code review tools. When you build an agent, you define its tools, system prompt, and behavior. This slide is about a key architectural decision in YOUR agent: how do you enforce business rules?

You have two options: put the rule in the prompt (a suggestion Claude usually follows) or enforce it in your code via SDK hooks (a gate Claude cannot bypass). The question is: when does each approach make sense?

When must you use programmatic hooks vs. prompt instructions?

Programmatic Hooks (Deterministic)

These are functions in YOUR agent code (Agent SDK) that intercept tool calls

  • PostToolUse hook — your code that runs after every tool call, can transform or block results
  • Blocks policy-violating actions (e.g., refunds > $500) before they execute
  • Prerequisite gates — your code blocks process_refund until get_customer has completed
  • 100% compliance rate — it's code running in your application, not a suggestion to Claude
  • Redirects to escalation workflows when triggered

Prompt Instructions (Probabilistic)

These are instructions in the system prompt that Claude tries to follow

  • System prompt says "always verify customer before processing refunds"
  • Few-shot examples showing correct tool ordering
  • Works most of the time, but has a non-zero failure rate (exam says 12% skip rate)
  • Good for soft preferences: style, tone, response format
  • Not sufficient when errors have financial or safety consequences
Exam Rule of Thumb: If non-compliance means money lost, wrong customer, or safety risk → use hooks (your code). If non-compliance means suboptimal but harmless output → prompt instructions are fine.

Anti-Patterns to Recognize

Remember: With the Agent SDK, You're Writing Application Code

These anti-patterns happen in your Python/TypeScript code — the application you build on top of the Claude API. When the exam mentions stop_reason, that's a property on the API response object:

response = client.messages.create(...)
# response.stop_reason is a string the API returns:
# "tool_use" → Claude wants to call a tool (keep looping)
# "end_turn" → Claude is done (stop the loop)

# CORRECT: check the API field
if response.stop_reason == "tool_use": ...

# WRONG: parse the assistant's text looking for "I'm done"
if "completed" in response.content[0].text: ... # fragile!

This is YOUR code making these decisions. The anti-patterns below are mistakes developers make when writing their agentic loop, coordinator logic, or session management code.

The exam tests whether you can identify these mistakes in production scenarios.

🚫
NL Parsing for Termination
Your loop code checks response.content[0].text for "I'm done" instead of checking response.stop_reason. The API gives you a reliable signal — use it.
🚫
Arbitrary Iteration Caps
Your loop code does for i in range(10) as the primary stop. The model decides when it's done via stop_reason == "end_turn".
🚫
Narrow Decomposition
Your coordinator code splits "creative industries" into only visual arts subtopics, missing music, writing, and film entirely.
🚫
Assuming Context Inheritance
Your code spawns a subagent via Task without passing context in the prompt, expecting it to "know" what the coordinator knows. It doesn't — subagents are isolated.
🚫
Prompt-Only Enforcement
Your agent's system prompt says "always verify customer first" but you wrote no PostToolUse hook to enforce it. For financial operations, prompts alone fail ~12% of the time.
🚫
Resuming Degraded Sessions
Using --resume on a session where Claude is already giving vague, inconsistent answers instead of starting fresh with a structured summary.

Task Decomposition Strategies

Prompt Chaining (Fixed Sequential)

When: Predictable, multi-aspect workflows
Step 1: Extract → Step 2: Validate → Step 3: Enrich → Step 4: Format

Fixed sequence of steps, each feeding into the next. Good for code reviews, multi-step validations, data pipelines.

Dynamic Adaptive Decomposition

When: Open-ended investigation tasks
Map structure → Find hotspots → New subtasks
based on findings → Adapt plan
as you go

Subtasks emerge from discoveries. The plan adapts at each step based on what's found.

Per-File + Integration Pass

When: Large multi-file code reviews
File A File B File C ... local analysis (parallel)
↓ results feed into
Cross-file integration pass data flow, interfaces

Avoids attention dilution. Each file gets full focus, then a separate pass catches cross-file issues.

Iterative Refinement Loops

When: Synthesis quality matters
Search + Analyze → Synthesize → Evaluate gaps ↺ loop until sufficient

Coordinator evaluates synthesis, re-delegates for gaps, re-synthesizes. Loops until coverage is good enough.

Session State Management

--resume <name>
Continue a specific prior conversation. Good when prior context is mostly valid and you need to pick up where you left off. Inform about specific file changes for targeted re-analysis.
fork_session
Create independent branches from a shared analysis baseline. Good for comparing approaches — e.g., testing two refactoring strategies from the same codebase understanding.
Fresh Start + Summary
Start a new session with a structured summary injected. Best when context has degraded — vague answers, "typical patterns" instead of specifics. More reliable than resuming stale state.
Crash Recovery via State Persistence

For long-running multi-agent workflows, persist progress to disk so you can recover from crashes. This is a design pattern, not a specific file format — use whatever your code reads easily (JSON, YAML, MD, SQLite, etc.).

// manifest.json — machine-readable
{
  "completed": ["auth.py", "users.py"],
  "pending": ["orders.py"]
}
# progress.md — human + machine readable
## Completed
- [x] auth.py (converted 10:23)
- [x] users.py (converted 10:25)
## Pending
- [ ] orders.py

The pattern: Each agent writes progress to a known location after completing work. On restart, the coordinator reads it to skip already-completed tasks. Without it, the coordinator starts from scratch, reprocessing everything.

Tool Design & MCP Integration

Domain 2 — Design effective tools, structure error responses, configure MCP servers, and choose the right built-in tools.

5 Task Statements • 6 Practice Problems • Covers the Developer Productivity Tools Scenario

What is Tool Design & MCP?

When Claude-powered agents need to interact with the real world — looking up customers, querying databases, calling APIs — they do it through tools. Tool Design is how you define those interfaces so Claude knows what's available and how to use each one correctly. MCP (Model Context Protocol) is the standard for connecting external services to Claude as tool providers.

Without good tool design, agents pick the wrong tool, get confusing errors they can't recover from, or drown in too many options. The difference between a reliable agent and a frustrating one often comes down to how well its tools are described, scoped, and organized.

Without Good Tool Design

Vague tool descriptions, generic error messages, 18 tools dumped on one agent. Claude guesses which tool to use and frequently guesses wrong.

With Good Tool Design

Clear descriptions with examples and boundaries, structured errors with retry guidance, each agent scoped to 4-5 relevant tools. Claude reliably picks the right tool.

The Big Idea

Tools are how agents act in the world. The quality of your tool descriptions, error responses, and tool distribution directly determines how reliably your agent behaves. MCP servers are how you connect these tools — project-level for shared team tooling, user-level for personal experiments.

Effective Tool Descriptions

Tool descriptions are the #1 lever for fixing tool selection. Improve them before adding infrastructure.

Minimal Description (Before)

"name": "get_customer"
"description": "Retrieves customer information"

"name": "lookup_order"
"description": "Retrieves order details"

No input formats, no boundary explanations, no examples. The model can't tell them apart reliably.

Detailed Description (After)

"name": "lookup_order"
"description": "Retrieves order details
by order ID (format: #12345 or
12345). Use when user asks about
order status, shipping, items, or
history. Do NOT use for customer
profile info — use get_customer."

Input formats, example queries, boundary explanations, and explicit "do NOT use for..." guidance.

When tools are being confused, improve descriptions first. Descriptions are the cheapest, most direct fix. Don't add routing layers, few-shot examples, or consolidate tools until you've exhausted this approach.

Structured Error Responses

MCP's isError flag communicates failures. But how you structure the error determines if the agent can recover.

Context: This is about YOUR MCP tool code

When you build an MCP server, you define the error response format in your tool's handler function. The isError field is part of the MCP protocol's CallToolResult — you set it to true in your server code to tell Claude the tool call failed. Everything else in the response body (error category, retry guidance, partial results) is a structure you design.

Transient
isRetryable: true
Timeouts, rate limits, temporary network failures. Agent should retry with backoff or try a narrower query.
Validation
isRetryable: false
Invalid input format, missing required fields, type mismatches. Agent should fix the input and re-call.
Business Logic
isRetryable: false
Policy violations (refund > limit), insufficient balance, business rule failures. Agent should explain to user or escalate.
Permission
isRetryable: false
Unauthorized access, insufficient privileges, expired tokens. Agent should request elevation or route differently.

Generic Error (Anti-pattern)

{
  "status": "error",
  "message": "Operation failed"
}

Agent has no idea if it should retry, fix input, or give up entirely.

Structured Error (Best Practice)

{
  "status": "error",
  "errorCategory": "transient",
  "isRetryable": true,
  "attemptedQuery": "AI impact...",
  "partialResults": ["3 articles"],
  "alternatives": ["narrower query"]
}

Agent can decide: retry, narrow scope, try alternative, or proceed with partial results.

What isError Looks Like in MCP

// In your MCP server's tool handler, you return a CallToolResult:

// Structured error (agent CAN recover):
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "{\"errorCategory\": \"transient\", \"isRetryable\": true,
      \"attemptedQuery\": \"AI impact on healthcare 2024\",
      \"partialResults\": [\"3 articles retrieved before timeout\"],
      \"alternatives\": [\"retry with narrower query\", \"use cached results\"]}"
  }]
}

// Generic error (agent is BLIND):
{
  "isError": true,
  "content": [{"type": "text", "text": "Operation failed"}]
}
Critical Distinction

Access failures are NOT empty results. A failed query (timeout, auth error) is fundamentally different from a successful query that returned zero matches. Never mask errors as empty success in your tool handler code — the agent won't attempt recovery.

tool_choice Options

Context: This is YOUR code talking to the Claude API

When you build an agent with the Claude Agent SDK, your application sends messages to the Claude API via client.messages.create(...). tool_choice is a parameter on that API call — it tells Claude how it's allowed to respond. This is not related to MCP. MCP is where tools come from (the plumbing); tool_choice is how Claude uses them (the rules).

Your Python/TS app → Claude API (tool_choice goes here) → Claude responds with tool_use or text
# In your Python application code:
response = client.messages.create(
  model="claude-sonnet-4-20250514",
  messages=[...],
  tools=[...],
  tool_choice={"type": "auto"} # or "any", or {"type": "tool", "name": "extract_data"}
)
tool_choice: "auto"
Model decides whether to call a tool or return text. Default behavior.
Guarantees: Nothing — may return text instead of calling a tool
tool_choice: "any"
Model MUST call a tool. Good for guaranteeing structured output.
Guarantees: A tool WILL be called (structured output)
tool_choice: {type: "tool", name: "..."}
Forces the model to call a specific tool by name.
Guarantees: That EXACT tool will be called
Scoped Tool Access Principle

Give each agent only role-relevant tools plus limited cross-role tools for high-frequency needs. When a subagent needs limited cross-role capabilities, provide a scoped tool for the common case rather than full access to another agent's toolset.

MCP Server Configuration

Context: Claude Code CLI Configuration

This slide is about configuring MCP servers for Claude Code (the CLI/IDE tool), not the Agent SDK. The .mcp.json and ~/.claude.json files tell Claude Code which MCP servers to connect to when it starts up. If you are building an Agent SDK application, you configure MCP servers programmatically in your Python/TypeScript code instead.

Two levels of configuration — project-scoped (shared) and user-scoped (personal). Never commit secrets.

Project-Level: .mcp.json

  • Lives in the project root directory
  • Version controlled — shared with all team members
  • Contains shared team tooling (GitHub, CI/CD, etc.)
  • Available to anyone who clones the repo
// .mcp.json (project root)
{
  "mcpServers": {
    "github": {
      "command": "gh-mcp-server",
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

User-Level: ~/.claude.json

  • Lives in the user's home directory
  • NOT version controlled — personal only
  • For personal/experimental MCP servers
  • Other team members don't see or use these
// ~/.claude.json (user home)
{
  "mcpServers": {
    "jira-experimental": {
      "command": "jira-mcp",
      "env": {
        "JIRA_TOKEN": "${JIRA_TOKEN}"
      }
    }
  }
}
Environment Variable Expansion

Use ${VARIABLE_NAME} syntax in MCP config for credentials. Each developer sets the actual value in their local environment. Never hardcode tokens in .mcp.json — they would be committed to version control. All tools from all configured MCP servers are discovered at connection time and available simultaneously.

MCP Resources vs. Tools

MCP provides two distinct primitives. Match the primitive to the operation type.

MCP Resources (Read-Only Browsing)

Content catalogs the agent can browse without tool calls. Passive consumption of available data.

  • Issue summaries and ticket metadata
  • Database schemas and table definitions
  • Project template catalogs
  • Documentation hierarchies
  • Configuration inventories
Operation: READ
No state changes. Reduces exploratory tool calls.

MCP Tools (Actions That Modify State)

Explicit actions with parameters, confirmation, and side effects. Active operations that change systems.

  • Create project from template
  • File a new issue or ticket
  • Deploy a service to staging
  • Process a refund transaction
  • Update database records
Operation: CREATE / UPDATE / DELETE
Modifies state. Requires explicit parameters.
Decision Rule: Reading/browsing available data = Resource. Creating/updating/deleting = Tool. A template catalog is a resource; creating a project from a template is a tool.

What is Claude Code Configuration?

Claude Code is Anthropic's CLI tool that puts Claude directly in your development workflow. Configuration is how you teach it your team's conventions, create reusable commands, and integrate it into automated pipelines. Think of it as setting up Claude as a team member who knows your codebase standards.

Without proper configuration, every developer gets a generic Claude that doesn't know your testing conventions, coding standards, or deployment patterns. With it, Claude automatically applies the right rules to the right files and your CI pipeline gets structured, actionable feedback.

Without Configuration

Every developer manually explains conventions each session. Claude applies React patterns to API code. CI pipeline hangs waiting for interactive input.

With Configuration

The Big Idea

Claude Code configuration is about encoding your team's knowledge so Claude applies it automatically. CLAUDE.md files create a hierarchy of instructions (user > project > directory), slash commands share team workflows, path-specific rules target conventions by file type, and the -p flag bridges the gap to CI/CD automation.

Custom Commands & Skills

A command is a reusable prompt you invoke with a slash — like /review or /deploy. Instead of typing the same instructions every time ("review this PR for security issues, check for SQL injection, XSS..."), you save them as a markdown file and invoke with one slash. Commands are just .md files in a commands/ directory — the key question for the exam is where you put them.

Project-Scoped Commands

  • .claude/commands/ in the repository
  • Version controlled — shared via git
  • Available to all devs who clone/pull
  • Team-standard workflows: /review, /deploy
  • Updates propagate automatically

User-Scoped Commands

  • ~/.claude/commands/ in home directory
  • Not version controlled — personal only
  • Must be manually created per developer
  • Personal shortcuts: /my-lint, /quick-test
  • Updates don't propagate to others

Skills

A skill is a more powerful version of a command. While commands are simple markdown prompts, skills live in .claude/skills/ (or ~/.claude/skills/ for personal ones) and have a SKILL.md file with YAML frontmatter that controls how the skill runs — whether it's isolated, what tools it can access, and what input it requires. What is Prompt Engineering for Structured Output?

When you need Claude to produce reliable, machine-readable output — extracting data from documents, generating code review findings, processing invoices — you need more than conversational prompts. This domain covers the techniques that turn Claude from a helpful assistant into a precision data processing tool.

Without these techniques, you get inconsistent formats, hallucinated data in required fields, reviews that miss real bugs while flagging false positives, and no way to process documents at scale. The gap between a demo and a production system is almost entirely in how you engineer the prompts and validate the output.

Without These Techniques

Vague review criteria, inconsistent extraction formats, fabricated values for missing data, one-shot processing that can't scale, single-pass reviews that miss bugs in large PRs.

With These Techniques

Explicit criteria with few-shot examples, JSON schemas with nullable fields, validation-retry loops, batch processing for scale, multi-pass reviews that catch what single passes miss.

The Big Idea

Production-grade output requires four layers: explicit criteria (tell Claude exactly what to flag), few-shot examples (show the pattern, don't just describe it), tool_use with JSON schemas (guarantee structure), and validation loops (catch what slips through). Each layer catches what the previous one misses.

Vague instructions produce noisy output. Explicit criteria tell the model exactly what to flag and what to skip.

Few-Shot Prompting

"Few-shot" means giving Claude a small number of worked examples (typically 2-4) inside your prompt before asking it to process real data. The term comes from machine learning: "zero-shot" = no examples, "few-shot" = a few examples, "many-shot" = lots of examples. In practice, it's like showing a new employee: "Here's how we handle this type of case — see these three examples? Now do the same for this new one."

Why it matters: when you write detailed instructions but Claude's output is still inconsistent (wrong format, different handling of edge cases each time), examples fix what words can't. Claude sees the exact input-to-output pattern and generalizes to novel cases it hasn't seen. The exam tests this as the go-to technique when prose descriptions fail.

Structured Output via tool_use

When you ask Claude a question normally, it responds with free-form text. That's fine for conversation, but useless when your application needs to parse the response — extract an invoice total, populate a database field, or feed data into another system. Structured output means forcing Claude to respond in a specific JSON format that your code can reliably parse every time.

The mechanism is called tool_use: you define a fake "tool" with a JSON schema describing the exact structure you want, and Claude responds by "calling" that tool with the data filled in. It's not really calling anything — it's a trick to force structured output. The JSON schema guarantees no syntax errors (missing brackets, wrong types), but it does NOT prevent semantic errors (putting a city name in a zip code field). That's what validation loops are for.

Normally when your app calls the Claude API, it sends one request and waits for a response — that's synchronous. But what if you need to process 500 invoices overnight? Sending them one at a time is slow and expensive. The tradeoff: no guaranteed latency. Results come back within a 24-hour window, not in seconds. So batch is great for overnight jobs but terrible for anything a developer is waiting on.

What is Context Management?

Every Claude conversation has a context window — a finite amount of information Claude can hold in working memory at once. Context Management is the discipline of keeping the right information available at the right time, especially in long conversations, multi-agent systems, and large codebase explorations.

Without context management, agents forget critical details mid-conversation, research systems lose source attribution, escalation decisions are based on gut feelings instead of criteria, and 97% accuracy hides the fact that one document type fails 22% of the time. These are the reliability problems that sink production deployments.

Without Context Management

Agent forgets customer details after 5 tool calls. Research report cites statistics without sources. Escalation based on sentiment analysis. '97% accurate' masks 78% accuracy on handwritten invoices.

With Context Management

Persistent case facts block preserves key data. Structured claim-source mappings through synthesis. Explicit escalation criteria with few-shot examples. Stratified accuracy tracking by document type.

The Big Idea

Context management is about fighting entropy. As conversations grow, information degrades — specifics become vague, sources lose attribution, and the model falls back on general knowledge instead of what it actually found. Every technique in this domain (fact extraction, trimming, scratchpads, coverage annotations) exists to keep critical information sharp and accessible.

Preserving Critical Information

What "Conversation History" Actually Is

In the Claude API, "conversation history" is the messages array you pass in each API call. Every user message, assistant response, and tool call result gets appended to this array. As the array grows, it consumes more of the context window. In Claude Code (the CLI tool), this happens automatically as you interact — but the same principle applies: the conversation grows with each exchange, and older information can get lost or deprioritized.

Long conversations lose precision. Tool outputs bloat context. Key facts get buried.

The "Lost in the Middle" Effect

Models process the beginning and end of long inputs reliably, but findings from middle sections may be missed.

BEGINNING — processed well
MIDDLE — may be missed / deprioritized
END — processed well

Error Propagation Across Multi-Agent Systems

In a multi-agent system, subagents will fail — timeouts, service outages, rate limits. The exam tests three anti-patterns: generic errors that hide context ("search unavailable" — but was it a timeout or a service outage?), silent suppression (returning empty results as success so the coordinator thinks nothing was found), and terminating the entire workflow because one subagent failed.

Anti-Patterns

🚫
Generic Error Message
In your subagent's error handler: returning "Search unavailable" — hides what was attempted, what partially worked, and what alternatives exist. Coordinator is blind.
🚫
Silent Suppression
In your subagent's return value: returning empty results as success. Coordinator can't distinguish "nothing found" from "search failed." Valid empty results look identical to errors.
🚫
Workflow Termination
In your coordinator's error handling: killing the entire multi-agent workflow because one subagent failed. Other subagents' results are lost. Partial progress is discarded.
Key Distinction

Distinguish access failures (timeouts needing retry) from valid empty results (successful query, no matches). Subagents should implement local recovery for transient failures and only propagate errors they cannot resolve.

Pydantic Validation

Pydantic is a Python library for data validation using type hints. In the exam context, it is the second layer of defense after JSON schema validation from tool_use.

The Two-Layer Defense

JSON schemas via tool_use eliminate syntax errors, but Pydantic catches semantic errors — values that are valid JSON but wrong.

Layer 1: tool_use + JSON Schema
Prevents syntax errors — missing brackets, wrong types, missing required fields. Claude's tool_use guarantees structurally valid JSON.
Layer 2: Pydantic Validation
Catches semantic errors — a city name in a zip code field, line items that don't sum to the stated total, a date in the past for a future appointment.
Layer 3: Validation-Retry Loop
On failure, append specific Pydantic error messages to the prompt and retry. The model sees exactly what went wrong ("zip_code: value is not a valid zip code") and self-corrects.
Exam Trap

The exam may test whether you know that tool_use alone does not prevent semantic errors. An answer like "tool_use with JSON schema prevents all data errors" is incorrect. You need Pydantic (or equivalent validation) as a second layer for semantic correctness.

Tool Naming & Splitting

Tool Naming: Eliminate Ambiguity

The tool name itself is part of the description that helps Claude choose. Ambiguous names cause misrouting.

Ambiguous Names

  • analyze_content — "Analyzes content"
  • analyze_document — "Analyzes a document"
  • Near-identical descriptions. Claude picks randomly between them.

Specific Names

  • extract_web_results — "Extracts structured data from web search results"
  • summarize_pdf_document — "Generates a summary of a PDF document"
  • Names alone tell Claude which to use.

When to Split vs Consolidate

Split When...

  • A tool does too many things and Claude can't predict what it will return
  • Example: analyze_document → split into extract_data_points, summarize_content, verify_claim_against_source
  • Each tool has a single, predictable purpose

Consolidate When...

  • Two tools overlap and Claude picks the wrong one
  • Example: get_customer and lookup_customer doing the same thing → keep one, remove the other
  • Redundant tools create confusion, not flexibility
The Rule

Split when a tool does too many things and Claude can't predict what it'll return. Consolidate when two tools overlap and Claude picks the wrong one.

Key Decision Frameworks

The recurring rules that apply across multiple domains. Internalize these and you can reason through any scenario.

Hooks > Prompts
When compliance must be deterministic — financial operations, identity verification, policy enforcement. Prompts have a non-zero failure rate; hooks are code.
Descriptions First
When tools are confused or misrouted — improve tool descriptions before adding routing layers, classifiers, or infrastructure. Cheapest, most direct fix.
Plan Mode vs Direct Execution
Plan mode for complex multi-file architectural changes with multiple valid approaches. Direct execution for well-scoped single-file fixes with clear scope.
Few-Shot Examples
When prose instructions produce inconsistent output or ambiguous cases need consistent handling. Examples demonstrate judgment better than descriptions.
Nullable Fields
When source data may be absent. Required fields for optional data force the model to fabricate. Allow null to represent “not found in source.”
Retry = Structural Errors Only
Retry-with-feedback fixes format/ordering errors where the model has the data but made a mistake. It cannot conjure absent information — make the field nullable instead.
Sync API vs Batch API
Sync (real-time) for blocking workflows like pre-merge checks. Batch API (50% savings, up to 24h) for overnight reports, weekly audits, nightly jobs.
Per-File + Integration Pass
For large multi-file reviews. Analyze each file individually (avoids attention dilution), then run a cross-file pass for data flow and interface consistency.
Fresh Start with Summary > Resume
When context has degraded (vague answers, “typical patterns” instead of specifics). Start fresh with structured summary — don’t resume stale state.
Ask for Identifiers, Don’t Select Heuristically
When multiple matches are returned (e.g., 3 customers named “Sarah Johnson”). Ask for email, phone, or card to disambiguate — never pick “most recent.”
