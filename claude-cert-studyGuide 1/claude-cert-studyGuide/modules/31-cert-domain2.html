Cert Domain 2: Tool Design & MCP Integration

Master the 18% of the certification exam covering tool descriptions, structured errors, MCP configuration, tool distribution, and built-in tool selection. This module provides deep-dive content, hands-on exercises, and a realistic practice exam.

⏱ ~75 min ★ Expert 🎯 3 Interactive Exercises 📋 8-Question Practice Exam

1. Domain Overview

18% of Exam Weight

Domain 2 accounts for 18% of your certification score, making it the third-largest domain on the exam. It tests your ability to design tools that Claude can use effectively, handle errors gracefully, configure MCP servers correctly, and choose the right built-in tools for each task.

The domain is divided into 5 task areas, each targeting a specific competency. Understanding these areas helps you allocate study time proportionally.

The 5 Task Areas

1 Tool Descriptions
2 Structured Errors
3 Tool Distribution
4 MCP Integration
5 Built-in Tools
ⓘ

Exam strategy: Tool description questions are the most common in this domain. If you can write excellent descriptions and identify description flaws, you'll handle roughly 40% of this domain's questions.

What Makes This Domain Challenging

Unlike domains that test conceptual knowledge, Domain 2 is intensely practical. Exam questions present you with real-world scenarios -- a tool that's being misrouted, an error response that's missing context, an MCP config that's not loading -- and ask you to diagnose or fix the problem. Rote memorization won't help. You need hands-on familiarity with how Claude interprets tool descriptions, how error metadata shapes retry behavior, and how MCP server scoping determines which tools are available.

⚠

Common trap: Many candidates confuse "tool distribution" (project-scoped vs user-scoped) with "tool_choice" settings (auto/any/specific). The exam tests both, and they are distinct concepts. Distribution controls who sees the tool; tool_choice controls when Claude picks it.

2. Tool Description Excellence

Tool descriptions are the single most important factor in whether Claude uses a tool correctly. Claude reads tool descriptions the way a developer reads API documentation -- it makes routing decisions, parameter choices, and error-handling assumptions based entirely on what you write.

The 5 Rules of Tool Descriptions

Every tool description should answer these 5 questions. Missing any one of them creates an opportunity for misrouting or misuse.

1

What the Tool Does

Identity

Start with a clear, one-sentence statement of the tool's purpose. Use active voice. Be specific about the domain it operates in. Avoid generic language like "handles data" or "manages resources."

Good vs Bad // Bad: vague, could be anything "Manages customer data" // Good: specific domain + action "Retrieves a customer's billing history from Stripe, returning a list of invoices with amounts, dates, and payment status."
2

When to Use It

Routing

Explicitly state the conditions under which this tool should be selected. This is the most common source of misrouting -- Claude cannot differentiate between two similar tools if the "when" is ambiguous.

Example "Use this tool when the user asks about past payments, invoice history, or billing charges. Do NOT use for updating payment methods -- use update_payment_method instead."
3

Input Format & Constraints

Parameters

Document each parameter's type, format, valid range, and any special encoding requirements. JSON Schema handles type validation, but Claude needs human-readable context for semantic correctness.

Parameter Description "customer_id: The Stripe customer ID (starts with 'cus_'). Required. If the user provides an email instead, use lookup_customer first." "date_range: ISO 8601 date range. Max span is 12 months. If not provided, defaults to last 30 days."
4

Output Shape

Response

Describe what the tool returns, including the structure, key fields, and any pagination behavior. Claude needs this to plan its next steps before the tool even runs.

Output Description "Returns { invoices: Invoice[], hasMore: boolean, cursor: string }. Each Invoice includes id, amount (in cents), currency, status ('paid'|'open'|'void'), and created_at. Results are paginated; pass cursor to get the next page."
5

Edge Cases & Gotchas

Safety

Document known failure modes, rate limits, and surprising behaviors. This is what separates a "good enough" description from an excellent one.

Edge Cases "Returns an empty array (not an error) if the customer has no billing history. Rate-limited to 100 requests per minute. If the customer was deleted, returns { error: 'customer_archived' } -- this is NOT a permission error; suggest the user contact support."

Common Pitfalls

⚠

Ambiguous descriptions cause misrouting. When two tools have overlapping descriptions, Claude picks the one whose description best matches the user's phrasing -- not the one you intended. The fix is always to sharpen the "when to use" section, adding explicit negative constraints ("do NOT use for...").

System prompt keyword sensitivity: Claude gives higher weight to keywords that appear in both the system prompt and a tool description. If your system prompt says "always verify customer identity before billing lookups," and your get_billing_history tool description mentions "customer identity verification," Claude will route to it more aggressively. This is useful when intentional, dangerous when accidental.

Renaming and Splitting Tools

Sometimes the best fix for misrouting is not editing the description but renaming or splitting the tool. A tool named manage_customer that handles both reads and writes is inherently ambiguous. Splitting it into get_customer_profile and update_customer_profile gives Claude two distinct routing targets. The exam frequently tests this pattern -- look for questions where the "fix" involves restructuring the tool surface rather than editing the description.

Pitfall Symptom Fix
Vague description Claude uses the tool for unrelated tasks Add "when to use" and negative constraints
Missing output shape Claude calls the tool but can't parse the response Document return fields and types
Overlapping tools Claude alternates between two tools for the same task Differentiate "when" sections or merge tools
Overloaded tool Claude passes wrong parameters for the sub-action Split into single-purpose tools
Missing edge cases Claude retries on valid empty results, thinking it failed Document empty/null as valid outcomes

3. Error Handling Mastery

The exam expects you to design error responses that give Claude enough context to decide its next step without human intervention. A bare {"error": "failed"} is the worst possible response -- Claude cannot determine if it should retry, try a different tool, or inform the user.

Error Taxonomy

Every error belongs to one of four categories. Knowing the category determines the correct response strategy.

↻

Transient Errors

Retryable

Temporary failures caused by network issues, rate limits, or service unavailability. Claude should retry after a delay, ideally with exponential backoff.

  • HTTP 429 (Too Many Requests)
  • HTTP 503 (Service Unavailable)
  • Network timeouts
  • Database connection pool exhaustion
✗

Validation Errors

Fix Input

The tool was called with invalid parameters. Claude should fix the input and retry -- never retry with the same parameters.

  • Missing required field
  • Invalid format (e.g., email without @)
  • Value out of range
  • Type mismatch
ⓘ

Business Logic Errors

Inform User

The request is technically valid but violates a business rule. Claude should explain the constraint to the user and suggest alternatives.

  • Insufficient account balance
  • Duplicate record already exists
  • Feature not available on the user's plan
  • Approval required from a different role
🔒

Permission Errors

Escalate

The caller lacks authorization. Claude should not retry and should inform the user about the required permissions. Never expose internal authorization details.

  • HTTP 401 (Unauthorized) / 403 (Forbidden)
  • Expired or revoked tokens
  • IP restriction blocks
  • Role-based access control denials

Structured Error Metadata

The exam tests your ability to construct error responses with 5 key metadata fields. Each field serves a specific purpose in Claude's decision-making loop.

Ideal Error Response { "isError": true, "errorCategory": "validation", "isRetryable": false, "description": "The customer_id 'usr_123' is not a valid Stripe customer ID. Stripe customer IDs start with 'cus_'. Did you mean to use lookup_customer with the user ID first?", "recoveryGuidance": "Call lookup_customer with the user ID to get the Stripe customer ID, then retry this call." }
Field Type Purpose
isError boolean Distinguishes errors from valid empty results
errorCategory string enum Routes Claude to the correct recovery strategy
isRetryable boolean Tells Claude whether retrying with the same input makes sense
description string Human-readable explanation that Claude can relay to the user
recoveryGuidance string Specific next steps Claude should take to resolve the issue

Local vs Propagated Errors

Local errors occur within the tool itself (bad input, failed database query). These should be returned as structured error responses to Claude with full metadata.

Propagated errors come from downstream services (a third-party API returns 500). The tool should catch these, translate them into the structured format, and add context about which downstream service failed. Never pass raw third-party error messages to Claude -- they often contain internal details that confuse rather than help.

💡

Exam tip: A frequent question pattern asks you to distinguish between an access failure and a valid empty result. For example, a search tool returning zero results is not an error -- it's {"isError": false, "results": []}. But a search tool returning 403 because the user lacks access IS an error -- {"isError": true, "errorCategory": "permission"}. The isError flag is the critical discriminator.

4. MCP Server Patterns

The Model Context Protocol (MCP) standardizes how Claude connects to external tools and data sources. The exam tests your understanding of MCP configuration, scoping, and server selection.

Project vs User Configuration

MCP servers can be registered at two levels, and the exam frequently tests which level is appropriate for a given scenario.

Aspect Project: .mcp.json User: ~/.claude.json
Location Project root directory User's home directory
Scope Available to all team members working on this project Available only to this user, across all projects
Version controlled Yes -- committed to the repository No -- personal configuration
Best for Project-specific databases, internal APIs, domain tools Personal productivity tools, private API keys, global utilities
Env variables Expanded at runtime from the user's environment Same expansion behavior

Configuration Structure

.mcp.json (Project level) { "mcpServers": { "database": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-postgres"], "env": { "DATABASE_URL": "${DB_CONNECTION_STRING}" } }, "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } } } }

Environment variable expansion: The ${VARIABLE_NAME} syntax is resolved at runtime from the current shell environment. This means sensitive values like tokens never appear in the config file -- only variable references. The exam tests whether you can identify when env expansion will fail (e.g., the variable is not set in the user's environment).

Tool Description Enhancement

MCP servers often provide minimal tool descriptions. You can enhance them by wrapping the server's tools in your own MCP server or by adding description overrides in your configuration. The exam tests this as a strategy for improving tool routing when using community MCP servers.

Community Servers vs Custom Servers

🌐

When to Use Community Servers

  • Standard protocols with well-defined schemas (PostgreSQL, GitHub, Slack)
  • Read-heavy operations where built-in descriptions are adequate
  • Rapid prototyping -- get something working in minutes
  • The community server has strong maintenance and issue tracking
🔧

When to Build Custom Servers

  • Proprietary internal APIs with domain-specific semantics
  • You need enhanced tool descriptions for better routing
  • Complex authorization flows (multi-tenant, row-level security)
  • You want to expose resources alongside tools to reduce exploratory calls

Exposing Resources to Reduce Exploratory Calls

MCP resources are read-only data that Claude can access without calling a tool. By exposing schema definitions, enum lists, or configuration values as resources, you eliminate the need for Claude to make "discovery" tool calls just to understand the data model.

Resource Example // Instead of Claude calling get_table_schema("users") // every time, expose it as a resource: { "resources": [ { "uri": "schema://tables/users", "name": "Users Table Schema", "description": "Column definitions for the users table", "mimeType": "application/json" } ] }
💡

Exam tip: Questions about reducing token usage in multi-step workflows frequently have "expose data as MCP resources" as the correct answer. Resources are loaded once into context, while tool calls consume tokens on every invocation.

5. Built-in Tool Selection

Claude Code ships with 7 built-in tools. The exam tests your ability to choose the right tool for a given task and to combine them in efficient multi-step workflows.

Decision Tree: Grep vs Glob vs Read

These three tools form the core of codebase understanding. Choosing the wrong one wastes context tokens and adds unnecessary latency.

🔍

Grep -- Search Inside Files

Content Search

Use when you need to find files containing specific text, patterns, or code constructs. Returns matching lines with file paths and line numbers.

  • Best for: "Find all files that import X," "Where is function Y defined?"
  • Supports: Full regex, file type filtering, glob filtering, context lines
  • Output modes: files_with_matches (default), content, count
📁

Glob -- Find Files by Name

File Discovery

Use when you need to find files by their name, extension, or path pattern. Returns matching file paths sorted by modification time.

  • Best for: "Find all .tsx files in src/," "Is there a config.yaml?"
  • Supports: Standard glob patterns (**, *, ?)
  • Does NOT search: Inside file contents
📄

Read -- View File Contents

File Viewing

Use when you need to see the actual content of a known file. Supports offset/limit for large files and can read images, PDFs, and notebooks.

  • Best for: "Show me line 42 of main.py," "What does the config look like?"
  • Supports: Line offset/limit, PDF page ranges, image viewing
  • Caution: Reading large files without offset/limit wastes tokens

Edit vs Read+Write

Use Case Tool Why
Change one function signature Edit Surgical replacement, no risk of destroying unrelated code
Rename a variable everywhere in one file Edit (replace_all) Atomic rename with a single tool call
Create a brand new file Write File does not exist yet; Edit requires an existing file
Complete file rewrite (80%+ changed) Read then Write When most lines change, a full rewrite is cleaner than many edits
Add 3 lines to the middle of a file Edit Targeted insertion; Write would need to reproduce the entire file

Incremental Codebase Understanding

The most token-efficient workflow for understanding unfamiliar code follows a three-step pattern. The exam frequently presents scenarios where the wrong ordering wastes context.

  1. Grep -- Find entry points

    Start with a targeted Grep for the function name, class name, or error message. This gives you file paths and line numbers without loading full file contents.

  2. Read -- Examine key files

    Use Read with offset/limit to view only the relevant sections of the files Grep identified. Avoid reading entire files when you only need 20 lines around the match.

  3. Trace imports -- Follow the chain

    Use Grep again to trace imports and dependencies from the files you've read. This builds understanding incrementally rather than loading everything at once.

⚠

Anti-pattern: Reading every file in a directory "just in case" before doing anything. This fills the context window with irrelevant code. Always start with Grep to narrow down which files matter.

Practice Exam: Domain 2

This 8-question practice exam simulates the certification format for the Tool Design & MCP Integration domain. You need at least 6 correct to pass. Your score is saved to localStorage.

Domain 2 Practice Exam

8 questions -- 6 correct to pass. Answer all questions before submitting.

Question 1 of 8
A team reports that Claude frequently calls the search_documents tool when users ask about document metadata (author, creation date), even though a separate get_document_metadata tool exists. What is the most likely cause and best fix?
  • The search_documents tool is too slow, so Claude avoids get_document_metadata
  • The search_documents description lacks negative constraints; add "Do NOT use for metadata lookups -- use get_document_metadata for author, dates, or properties"
  • Delete the search_documents tool and merge its functionality into get_document_metadata
  • Set tool_choice to force get_document_metadata
Question 2 of 8
A tool returns {"error": "Rate limit exceeded"} without any metadata. Claude retries 5 times and then tells the user the feature is broken. What structured error response would fix this behavior?
  • {"isError": true, "errorCategory": "transient", "isRetryable": true, "description": "Rate limit exceeded. Wait 30 seconds before retrying.", "recoveryGuidance": "Retry after a 30-second delay."}
  • {"error": "Rate limit exceeded", "retry": true}
  • {"isError": true, "errorCategory": "permission", "isRetryable": false}
  • Simply increase the rate limit on the server side
Question 3 of 8
You want the entire development team to have access to a PostgreSQL MCP server, but each developer has their own database credentials stored in environment variables. Where should the MCP configuration go?
  • Each developer's ~/.claude.json with hardcoded credentials
  • The project's .mcp.json using ${DB_CONNECTION_STRING} environment variable expansion
  • A shared ~/.claude.json on a network drive
  • Hardcode the connection string directly in .mcp.json
Question 4 of 8
You need to find where the calculateTax function is defined in a large TypeScript project, then understand how it works. Which tool sequence is most token-efficient?
  • Read every .ts file in the project, then search for the function
  • Glob for *.ts files, then Read each one
  • Grep for "calculateTax" to find the definition file/line, then Read with offset/limit to examine just that section, then Grep for imports to trace dependencies
  • Use the Agent tool to search the entire codebase
Question 5 of 8
In a multi-agent system, a sub-agent's tool returns an error. What should the sub-agent do with the error before passing results back to the orchestrator?
  • Pass the raw error directly to the orchestrator without modification
  • Silently ignore the error and return an empty result
  • Attempt local recovery if possible; if not, translate the error into the structured format with context about what the sub-agent was trying to do, then return it to the orchestrator
  • Retry the tool call indefinitely until it succeeds
Question 6 of 8
A database MCP server requires Claude to call get_table_schema before every query to understand the columns. This wastes tokens on repeated schema lookups. What is the best optimization?
  • Cache the schema in a CLAUDE.md file
  • Expose table schemas as MCP resources so they are loaded into context once, eliminating repeated tool calls
  • Remove the get_table_schema tool entirely
  • Hardcode all column names into the tool descriptions
Question 7 of 8
A developer configures an MCP server in their personal ~/.claude.json for a project-specific internal API. A new team member joins and cannot access the tool. What is the root cause?
  • The new team member's Claude Code version is outdated
  • The MCP server is in user-level config (~/.claude.json) instead of project-level (.mcp.json), so only the original developer has it
  • MCP servers cannot be shared between team members
  • The API requires a VPN connection
Question 8 of 8
You want Claude to always use a specific verification tool before executing any write operation, regardless of what the user asks. What is the best approach?
  • Set tool_choice to the verification tool for all API calls
  • Include an instruction in the system prompt AND add "Must call verify_permissions first" to every write tool's description
  • Remove all write tools and only provide the verification tool
  • Hope that Claude figures it out from context
