Claude Code Deep Dive

Master the advanced configuration, workflows, and integration patterns that transform Claude Code from a helpful assistant into a precision-tuned development partner. Learn to configure CLAUDE.md hierarchies, build custom commands, leverage Plan Mode, and integrate Claude into CI/CD pipelines.

⏱ ~90 min 📚 Intermediate 🎓 Developers & Architects 🎯 4 Interactive Exercises

1. CLAUDE.md Hierarchy

The CLAUDE.md file is how you give Claude Code persistent instructions. But one file is rarely enough for a real project. Claude Code supports a three-level hierarchy that lets you layer instructions from broad personal preferences down to directory-specific conventions.

The Three Levels

Global (~/.claude/CLAUDE.md)

Your personal preferences that apply to every project. Think of these as your coding personality: preferred naming conventions, communication style, tools you always want approved. This file lives in your home directory and is never committed to any repository.

Project (root CLAUDE.md or .claude/CLAUDE.md)

Team conventions shared via git. This is where you define the project's tech stack, coding standards, testing requirements, and architectural patterns. Every developer on the team sees these instructions. You can place this at the repo root as CLAUDE.md or inside the .claude/ directory.

Directory (subdirectory CLAUDE.md)

Scoped rules for specific parts of the codebase. A backend/CLAUDE.md might enforce Python conventions, while frontend/CLAUDE.md enforces TypeScript strict mode. These override project-level rules for files in their scope.

Inheritance and Override Rules

The hierarchy follows a most-specific-wins model:

  • Directory-level overrides project-level for matching files
  • Project-level overrides global for the current project
  • Global-level provides the baseline defaults
  • Non-conflicting rules from all levels are merged together

The @import Syntax

For complex configurations, you can split your CLAUDE.md into multiple files using the @import directive. This keeps each file focused and manageable.

CLAUDE.md # Project Configuration ## Tech Stack React 18 + TypeScript + Vite for frontend FastAPI + SQLAlchemy for backend ## Import modular rules @import .claude/rules/testing.md @import .claude/rules/api.md @import .claude/rules/frontend.md

The .claude/rules/ Directory

Beyond @import, Claude Code automatically loads rule files from .claude/rules/. Each file can include YAML frontmatter to scope its rules to specific file paths.

.claude/rules/testing.md --- path: **/*.test.* --- # Testing Standards - Always use describe/it blocks for test organization - Prefer test factories over inline fixture data - Every test must have an assertion - Use meaningful test names that describe the behavior
.claude/rules/api.md --- path: src/api/** --- # API Conventions - Use RESTful naming: plural nouns for resources - Always return proper HTTP status codes - Include request validation with Pydantic models - Add OpenAPI docstrings to every endpoint

Visual Hierarchy

Here is how the full hierarchy looks in a monorepo: 

~/.claude/CLAUDE.md (global -- your personal preferences)
  |
  +-- project-root/
      |
      +-- CLAUDE.md (project-level -- shared via git)
      |
      +-- .claude/
      |   +-- CLAUDE.md (alternative project location)
      |   +-- rules/
      |       +-- testing.md (path: **/*.test.*)
      |       +-- api.md (path: src/api/**)
      |       +-- frontend.md (path: src/ui/**)
      |
      +-- backend/
      |   +-- CLAUDE.md (directory-level -- Python rules)
      |
      +-- frontend/
          +-- CLAUDE.md (directory-level -- TypeScript rules)
💡

Monorepo tip: In a monorepo with packages that have different conventions (e.g., a Python service and a React app), put a CLAUDE.md in each package directory. This way Claude automatically follows the right conventions depending on which part of the codebase it is working in.

2. Settings.json Mastery

While CLAUDE.md provides natural-language instructions, settings.json controls Claude Code's operational behavior: which tools are auto-approved, which are forbidden, the default model, and conversation limits.

File Locations

  • Global: ~/.claude/settings.json -- applies to all projects
  • Project: .claude/settings.json -- committed to git, shared with team

Project settings merge with (and can override) global settings, similar to the CLAUDE.md hierarchy.

Key Settings

permissions.allow

An array of glob patterns for tool calls that should be auto-approved without asking. This is how you skip the "approve?" prompt for safe operations.

settings.json (permissions.allow) { "permissions": { "allow": [ "Read", "Glob", "Grep", "Bash: npm test*", "Bash: npm run lint*", "Bash: git status", "Bash: git diff*", "Bash: git log*" ] } }

permissions.deny

An array of patterns that Claude is never allowed to execute, regardless of allow rules. Deny always wins over allow.

settings.json (permissions.deny) { "permissions": { "deny": [ "Bash: rm -rf *", "Bash: git push --force*", "Bash: git reset --hard*", "Bash: DROP TABLE*", "Bash: curl * | bash" ] } }

model

Set the default model for the session. Useful when you want a faster model for simple tasks or a more capable model for complex architecture decisions.

maxTurns

Limits the number of agentic turns Claude can take per response. Useful in CI/CD to prevent runaway sessions. The default is essentially unlimited in interactive mode, but you should set this to a reasonable number (e.g., 20-50) for automated pipelines.

Full Example Configuration

~/.claude/settings.json { "permissions": { "allow": [ "Read", "Glob", "Grep", "Bash: git status", "Bash: git diff*", "Bash: git log*", "Bash: npm test*", "Bash: npm run *", "Bash: python -m pytest*" ], "deny": [ "Bash: rm -rf /*", "Bash: git push --force*", "Bash: git reset --hard*", "Bash: sudo *" ] }, "model": "claude-sonnet-4-20250514", "maxTurns": 50 }
⚠

Security first: The deny list is your safety net. A deny pattern always takes precedence over allow. If a tool call matches both an allow and a deny pattern, it is denied. Use deny patterns to protect against destructive operations, especially in production-adjacent environments.

3. Slash Commands

Slash commands are shortcuts you type directly in the Claude Code prompt. They fall into two categories: built-in commands that come with Claude Code, and custom commands you define for your project or personal use.

Built-in Commands

Command Purpose When to Use
/help Show available commands and usage tips When you need a quick reference
/clear Clear the conversation history When starting a new task in the same session
/compact Summarize the conversation to free context When context is getting full but you want continuity
/status Show session info (model, token usage, settings) When you want to check context usage
/doctor Diagnose configuration issues When Claude is behaving unexpectedly
/review-pr Review the current PR or a specified PR number After you or a teammate open a PR
/commit Generate a commit message and commit staged changes After making changes you want to save

Custom Commands

You can create your own slash commands by adding markdown files to the commands directory:

  • Project commands: .claude/commands/ -- shared with the team via git
  • Personal commands: ~/.claude/commands/ -- only for you

Each file becomes a command with the same name as the file (minus the .md extension).

.claude/commands/deploy.md # Deploy to Staging Run the following deployment steps in order: 1. Run all tests: `npm test` 2. Build the project: `npm run build` 3. Run the staging deploy script: `./scripts/deploy-staging.sh` 4. Verify the deployment by checking the health endpoint 5. Report the results If any step fails, stop and report the error. Do not continue.
.claude/commands/test-suite.md # Run Comprehensive Test Suite Execute all test categories and report results: 1. Unit tests: `npm run test:unit` 2. Integration tests: `npm run test:integration` 3. Lint check: `npm run lint` 4. Type check: `npx tsc --noEmit` Summarize: total tests run, passed, failed, and any warnings.

After creating these files, you can use them as /deploy and /test-suite in your Claude Code session.

💡

Command arguments: Custom commands can accept arguments. Whatever you type after the command name is passed as context. For example, /deploy production would include "production" as part of the instruction context.

4. Skills Deep Dive

Skills are SKILL.md files that encode domain knowledge and tool instructions that Claude can reference when working on related tasks. Unlike CLAUDE.md (which is always loaded), skills use lazy loading: Claude only reads a skill when the current task is relevant to it.

What Makes a Skill

A skill consists of two parts:

  • Frontmatter: YAML metadata that describes the skill, when to use it, and how to load it
  • Body: Markdown instructions and context that Claude follows when the skill is activated

Skill Anatomy

.claude/skills/code-review.md --- name: Code Review Standards description: Enforces team code review standards and checklist tags: [review, quality, standards] operations: [review, audit, check] context: fork allowed-tools: [Read, Grep, Glob] argument-hint: File path or PR number to review --- # Code Review Checklist When reviewing code, check the following: ## Security - No hardcoded secrets, API keys, or credentials - Input validation on all user-facing endpoints - SQL queries use parameterized statements ## Code Quality - Functions are under 50 lines - No duplicate code blocks (DRY) - Clear, descriptive variable and function names - Complex logic has explanatory comments ## Testing - New code has corresponding test coverage - Edge cases are tested - Tests follow the Arrange-Act-Assert pattern ## Performance - No N+1 database queries - Large lists use pagination - Expensive operations are cached where appropriate

Key Frontmatter Options

Option Purpose Example
context: fork Runs in an isolated sub-agent to avoid polluting the main session Heavy research or review tasks
allowed-tools Restricts which tools the skill can use [Read, Grep, Glob] for read-only skills
argument-hint Describes what argument the user should provide "File path or PR number"
operations Keywords that trigger skill activation [deploy, release, publish]
tags Categorization for skill discovery [testing, quality, ci]

Lazy Loading

Skills are not loaded into context at startup. Instead, Claude reads the frontmatter of available skills and only loads the full body when it determines a skill is relevant to the current task. This is critical for context management -- you can have dozens of skills without wasting context on irrelevant ones.

ℹ

Skills vs. CLAUDE.md: Use CLAUDE.md for rules that should always apply (coding standards, project conventions). Use skills for specialized knowledge that is only needed sometimes (deployment procedures, specific API integration guides, review checklists).

5. Plan Mode

Plan Mode is a two-phase workflow that separates planning from execution. Instead of Claude immediately making changes, it first explores the codebase, creates a detailed plan, and waits for your approval before executing anything.

When to Use Plan Mode

Use Plan Mode Skip Plan Mode
Complex changes touching 5+ files Simple single-file bug fixes
Architectural refactors Well-understood, small changes
Unfamiliar codebase Adding a new test for existing code
Database migrations Updating a string or config value
When you want to review the approach first When you know exactly what needs to change

The Workflow

Enter Plan Mode

Type /plan or describe your task and ask Claude to plan it. Claude switches to exploration mode, using Read, Grep, and Glob to understand the codebase without making changes.

Claude Generates a Plan

Claude produces a numbered list of steps with file-level granularity: which files to modify, what changes to make, and in what order. Complex plans may include dependency analysis and risk assessment.

You Review and Refine

Read through the plan. You can ask Claude to modify steps, add steps, remove steps, or change the approach entirely. This is your checkpoint before any code is touched.

Approve and Execute

Once satisfied, tell Claude to execute the plan. It works through each step, making the actual code changes. If it encounters unexpected issues, it stops and reports back.

Explore Subagents

During the planning phase, Claude can spawn explore subagents to research specific aspects of the codebase without consuming the main conversation's context. For example, it might spawn a subagent to map all database models while the main agent continues analyzing the API layer.

💡

Pro tip: After Claude generates a plan, ask it to identify the riskiest step. This helps you focus your review on the most critical changes and decide if those steps need extra caution or manual verification.

6. Headless & CI/CD Integration

Claude Code is not just an interactive tool. With headless mode, you can embed it into automated pipelines for code review, test generation, documentation, and more.

The -p (Print) Flag

The -p flag runs Claude Code in non-interactive mode. You provide a prompt, Claude processes it, and the result is printed to stdout. No interactive session, no approval prompts (tools run within allowed permissions).

Shell # Simple one-shot usage claude -p "Explain what this project does in 3 sentences" # Pipe input to Claude git diff | claude -p "Review this diff for bugs"

Structured Output

For machine-parseable output, use --output-format json and optionally --json-schema to enforce a specific structure.

Shell # JSON output claude -p --output-format json \ "List all API endpoints with their methods" # Enforce schema claude -p --output-format json \ --json-schema '{"type":"object","properties":{"endpoints":{"type":"array"}}}' \ "List all API endpoints"

GitHub Actions Example: Automated Code Review

.github/workflows/claude-review.yml name: Claude Code Review on: pull_request: types: [opened, synchronize] jobs: review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 with: fetch-depth: 0 - name: Claude Code Review run: | git diff origin/main...HEAD | \ claude -p --output-format json \ "Review this diff. Report bugs, security issues, and suggestions. Be specific about file and line." env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

Session Isolation

In CI/CD pipelines, it is a best practice to use separate Claude instances for generation vs. review tasks. The reasoning context from generating code makes the same session less effective at objectively reviewing that code. Two separate -p calls ensure independent evaluation.

Shell # Instance 1: Generate tests claude -p "Generate unit tests for src/auth.py" > new-tests.py # Instance 2: Review the generated tests (separate session) claude -p "Review new-tests.py for completeness and edge cases"
⚠

CI/CD safety: Always set maxTurns in your settings when running headless. Without it, a confused Claude session could loop indefinitely, burning API credits and compute time.

7. Git Workflows

Claude Code has deep git awareness. It reads your git status, understands diffs, can generate commit messages, create PRs, and even help resolve merge conflicts.

Git-Aware Operations

  • Commit messages: Use /commit and Claude reads the staged diff to generate a descriptive commit message that explains the "why" not just the "what"
  • PR creation: Claude can draft PR titles, descriptions, and test plans based on the full branch diff
  • PR review: /review-pr analyzes a PR for bugs, security issues, and style violations
  • Branch management: Claude can create feature branches, check for conflicts, and suggest rebase strategies

Handling Merge Conflicts

When you hit a merge conflict, Claude is exceptionally helpful. It can read the conflict markers, understand the intent of both sides, and resolve the conflict intelligently:

Prompt "I have merge conflicts after rebasing onto main. Resolve them, preferring the changes from my branch but incorporating any new additions from main."

Best Practices

  • Let Claude make changes, you review and commit. Claude proposes, you dispose. Review diffs before committing.
  • Use atomic commits. Ask Claude to commit logically related changes together, not everything at once.
  • Never auto-approve git push. Keep git push as a manual approval step, especially for shared branches.
  • Review before merging. Even if Claude created a PR, have a human review it before merging.
💡

Workflow pattern: Make changes with Claude, then run /commit to stage and commit with a well-crafted message. This keeps your git history clean without you having to write commit messages manually.
