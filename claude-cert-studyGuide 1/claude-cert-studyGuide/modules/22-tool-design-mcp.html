Tool Design & MCP Integration

Master the art of designing tools that Claude can use effectively, and learn how the Model Context Protocol (MCP) standardizes the connection between AI models and external capabilities.

⏱ ~90 min 📚 Advanced 🎓 Developers, Architects 🎯 4 Interactive Exercises

1. Tool Design Principles

Tools are how Claude interacts with the outside world. Every time Claude needs to read a file, query a database, call an API, or perform any action beyond generating text, it does so through a tool call. The quality of your tool design directly determines how effectively Claude can accomplish tasks.

Think of tools as the hands and eyes you give to an otherwise bodiless intelligence. If the hands are clumsy and the eyes are blurry, even the smartest mind will struggle. Great tool design is the difference between an AI assistant that frustrates users and one that delights them.

The Five Principles

⚙

1. Atomic Operations

Principle

Each tool should do one thing well. A tool called manage_users that can create, read, update, delete, and send emails to users is doing too much. Break it into create_user, get_user, update_user, delete_user, and send_user_email.

When tools do too many things, Claude must parse complex conditional logic to pick the right parameters. Atomic tools reduce ambiguity and increase reliability.

📄

2. Clear Naming

Principle

Tool names should be self-documenting verb-noun pairs. Claude reads tool names to decide which tool to use -- a clear name guides it toward the right choice.

  • Good: search_documents, get_weather_forecast, create_invoice
  • Bad: do_stuff, process, helper, run_action
📖

3. Exhaustive Descriptions

Critical

The tool description is the single most important field in your tool definition. The description IS the documentation for Claude. It reads this text to decide when and how to use the tool.

A good description answers: What does this tool do? When should it be used? When should it NOT be used? What are the side effects?

🔧

4. Typed Parameters

Principle

Use JSON Schema types, enums, and required fields to constrain what Claude sends. Typed parameters prevent entire classes of errors before they happen.

If a parameter accepts only "asc" or "desc", define it as an enum -- do not rely on Claude guessing the valid values from the description alone.

⚠

5. Error Clarity

Principle

Return structured errors, not generic "something went wrong" messages. When Claude receives a clear error, it can decide whether to retry, adjust parameters, or inform the user. A vague error leaves Claude guessing.

Bad: "Error: operation failed"

Good: "Validation error: 'email' must be a valid email address. Received: 'not-an-email'. This is not retryable -- fix the input."

Anti-Patterns to Avoid

⚠

The Swiss Army Knife tool: A single tool with a action parameter that changes its entire behavior. This forces Claude to understand complex branching logic and makes error handling unpredictable.

⚠

Missing parameter descriptions: If a parameter called mode has no description, Claude will guess what values are valid. It will sometimes guess wrong. Every parameter deserves a description.

⚠

Vague tool descriptions: "Handles data processing" tells Claude nothing. Instead: "Transforms raw CSV sales data into aggregated monthly summaries grouped by region. Use when the user needs sales analytics. Input must be a valid CSV file path."

2. JSON Schema for Tools

Every tool definition includes a JSON Schema that describes its parameters. This schema is what Claude reads to understand what arguments a tool accepts, what types they must be, and which ones are required.

The Type System

Type JSON Schema Example Value Use Case
string "type": "string" "hello world" Names, descriptions, queries, IDs
number "type": "number" 3.14 Prices, scores, measurements
integer "type": "integer" 42 Counts, limits, page numbers
boolean "type": "boolean" true Flags, toggles, yes/no options
array "type": "array" ["a", "b"] Lists of tags, IDs, items
object "type": "object" {"key": "val"} Nested config, filters, metadata
null "type": "null" null Explicit absence (rare in tools)

Enums: Restricting Valid Values

Enums are one of the most powerful features for tool design. They tell Claude exactly which values are valid, eliminating guesswork entirely.

JSON Schema { "sort_order": { "type": "string", "enum": ["asc", "desc"], "description": "Sort direction. 'asc' for oldest first, 'desc' for newest first." } }

Required Fields

The required array specifies which parameters Claude MUST provide. Omitting a required field is an error. Make parameters required when the tool cannot function without them.

Descriptions on Every Parameter

⚠

This is critical. Claude reads parameter descriptions to understand what to pass. A parameter without a description is a parameter Claude will guess about. Every. Single. Parameter. Gets. A. Description.

Complete Example: CRM Query Tool

Complete Tool Schema { "name": "search_contacts", "description": "Search the CRM for contacts matching the given criteria. Returns up to 'limit' results sorted by the specified field. Use this tool when the user asks about customers, contacts, or account holders. For company-level queries, use search_companies instead.", "input_schema": { "type": "object", "properties": { "query": { "type": "string", "description": "Free-text search query. Searches across name, email, and notes fields." }, "status": { "type": "string", "enum": ["active", "inactive", "lead", "churned"], "description": "Filter by contact status. Omit to include all statuses." }, "limit": { "type": "integer", "description": "Maximum number of results to return. Default is 10, max is 100.", "default": 10, "minimum": 1, "maximum": 100 }, "sort_by": { "type": "string", "enum": ["name", "created_at", "last_activity"], "description": "Field to sort results by. Defaults to relevance when a query is provided, or 'created_at' otherwise." } }, "required": ["query"], "additionalProperties": false } }

The additionalProperties: false Pattern

Setting additionalProperties: false in your schema tells Claude (and the runtime) that only the explicitly defined properties are allowed. This prevents Claude from inventing extra parameters your tool cannot handle. It is a best practice for every tool schema.

💡

Pro tip: If your tool accepts nested objects, apply additionalProperties: false at every level of nesting, not just the top level.

3. The Model Context Protocol (MCP)

The Model Context Protocol (MCP) is a standard protocol for connecting AI models to external tools and data sources. Think of it as a USB standard for AI -- just as USB lets any device connect to any computer, MCP lets any tool server connect to any AI client that supports the protocol.

The Client-Server Architecture

MCP uses a client-server model:

  • MCP Client -- The AI application (Claude Code, Claude.ai, or your custom application) that wants to use external tools.
  • MCP Server -- Your tool provider that exposes capabilities to the client.

Communication happens over a transport layer. Two transport types are supported:

Transport How It Works Best For Example
stdio Client launches server as a subprocess; communicates via stdin/stdout Local tools, development, personal servers A Python script that queries your local database
HTTP/SSE Client connects to a remote HTTP endpoint; server pushes events via Server-Sent Events Remote services, shared team servers, cloud APIs A company-wide API gateway MCP server

When a client connects to a server, they perform capability negotiation -- the server tells the client what it can do (which tools, resources, and prompts it exposes), and the client can then use those capabilities.

What MCP Servers Expose

An MCP server can expose three types of capabilities:

🔧

Tools

Functions

Functions that Claude can call to perform actions. These are the most common capability. Examples: querying a database, sending an email, creating a file, calling an API.

📂

Resources

Data Catalogs

Data that Claude can browse without making function calls. Resources are like a catalog or directory. Examples: a list of database tables, file listings, issue trackers, documentation indexes. Claude reads these to understand what data is available.

📝

Prompts

Templates

Pre-built prompt templates that guide Claude's behavior for specific workflows. Examples: a "code review" prompt template, a "data analysis" template, a "bug report" template. These help standardize how Claude approaches recurring tasks.

Why MCP Matters

Before MCP, every AI application had to build custom integrations with every tool. If you wanted Claude to use your database, you wrote Claude-specific code. If you wanted ChatGPT to use it too, you wrote ChatGPT-specific code. This did not scale.

MCP standardizes the interface. Build one MCP server, and any MCP-compatible client can use it. This means your tool investment is portable, sharable, and future-proof.

💡

Key insight: MCP is to AI tools what REST is to web APIs -- a standard protocol that lets independently developed systems work together without custom glue code.

4. Building MCP Servers

Building an MCP server is straightforward with the right library. The most popular approach is Python with FastMCP, which provides a decorator-based API similar to Flask or FastAPI.

Python with FastMCP

Python - FastMCP Server from fastmcp import FastMCP import json # Create the MCP server mcp = FastMCP("my-tools") @mcp.tool() def search_database(query: str, limit: int = 10) -> str: """Search the product database. Args: query: Search keywords to match against product names and descriptions. limit: Maximum number of results to return (default 10, max 100). """ results = db.search(query, limit) return json.dumps(results) @mcp.tool() def get_product_by_id(product_id: str) -> str: """Get detailed product information by its unique ID. Args: product_id: The unique product identifier (e.g. 'PROD-12345'). """ product = db.get(product_id) if not product: return json.dumps({"error": "Product not found"}) return json.dumps(product) # Run the server (uses stdio transport by default) mcp.run()

FastMCP automatically generates the JSON Schema from Python type hints and docstrings. The @mcp.tool() decorator registers the function as an MCP tool.

Node.js MCP Server

TypeScript - MCP Server import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js"; import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js"; import { z } from "zod"; const server = new McpServer({ name: "my-tools", version: "1.0.0" }); server.tool( "search_database", "Search the product database by keywords.", { query: z.string(), limit: z.number().default(10) }, async ({ query, limit }) => { const results = await db.search(query, limit); return { content: [{ type: "text", text: JSON.stringify(results) }] }; } ); const transport = new StdioServerTransport(); await server.connect(transport);

Exposing Resources

Resources give Claude contextual awareness of what data is available, without requiring tool calls. This is especially useful for database schemas, file directories, or configuration options.

Python - Exposing Resources @mcp.resource("schema://tables") def get_database_schema() -> str: """List all database tables and their columns.""" tables = db.get_schema() return json.dumps(tables, indent=2)

5. MCP Configuration in Claude Code

Claude Code supports two levels of MCP configuration: project-level (shared with the team) and user-level (personal).

Project-Level: .mcp.json

Place this file in your project root and commit it to version control. Every team member who opens this project in Claude Code will automatically connect to these MCP servers.

.mcp.json { "mcpServers": { "my-database": { "command": "python3", "args": ["mcp_server.py"], "env": { "DB_URL": "${DATABASE_URL}" } }, "search-engine": { "command": "npx", "args": ["@company/search-mcp"] } } }

User-Level: ~/.claude.json

For personal MCP servers that should not be shared with the team. This is the right place for servers that use your personal API keys or access your personal tools.

~/.claude.json { "mcpServers": { "my-notes": { "command": "python3", "args": ["/Users/me/tools/notes_mcp.py"], "env": { "NOTES_API_KEY": "${NOTES_API_KEY}" } } } }

Environment Variable Expansion

Use ${VAR_NAME} syntax to reference environment variables. This keeps secrets out of committed files. The variable is expanded at runtime from the user's shell environment.

When to Use Project vs User Level

Criteria Project (.mcp.json) User (~/.claude.json)
Shared with team? Yes (committed to git) No (personal only)
Uses team resources? Yes (shared DB, APIs) No (personal tools)
Contains secrets? No (use ${VAR} expansion) No (use ${VAR} expansion)
Example servers Project DB, internal APIs, CI tools Personal notes, calendar, custom scripts
ⓘ

Note: Never put raw API keys or passwords in either config file. Always use environment variable expansion (${VAR_NAME}) and set the actual values in your shell profile or a .env file that is gitignored.

6. Error Handling for Tools

Error handling in tools is fundamentally different from error handling in regular application code. In a tool, the consumer of the error is an AI model, not a human or another program. Claude needs enough information in the error response to make intelligent recovery decisions.

The isError Flag

MCP responses include an isError boolean that tells Claude the tool call failed. Without this flag, Claude may interpret error text as a successful result.

Error Taxonomy

⏰

Transient Errors

Retryable

Temporary failures that may succeed on retry: timeouts, rate limits, service temporarily unavailable. Claude should wait and retry.

❌

Validation Errors

Fix Input

The input was malformed or invalid: wrong type, missing required field, invalid email format. Claude should fix the input and retry with corrected parameters.

📜

Business Errors

Redirect

The operation violates a business rule: refund exceeds limit, user already exists, insufficient balance. Claude should inform the user or try an alternative workflow.

🔒

Permission Errors

Escalate

The caller lacks authorization: missing API key, expired token, insufficient role. Claude should inform the user that elevated access is needed.

Structured Error Response

MCP Error Response { "content": [ { "type": "text", "text": "Rate limit exceeded. You have made 100 requests in the last minute. Retry after 30 seconds." } ], "isError": true, "_meta": { "errorCategory": "transient", "isRetryable": true, "retryAfter": 30 } }

Why Generic Errors Hurt

When a tool returns "Error: operation failed", Claude has no basis for making a recovery decision. Should it retry? Should it change the input? Should it tell the user? It does not know. The result is either a confusing error message passed to the user, or a wasted retry that was never going to succeed.

Structured errors with categories, retryable flags, and human-readable explanations let Claude make intelligent decisions autonomously, which is the entire point of agentic AI.

7. Tool Distribution & Testing

Once you have built an MCP server, you need to distribute it to users and ensure it works reliably. This section covers the distribution and testing lifecycle.

Local Plugins for Development

During development, point Claude Code directly at your local MCP server using the .mcp.json configuration. Changes to your server code are picked up immediately (if using stdio transport, restart Claude Code or re-connect).

Community MCP Servers

The MCP ecosystem is growing rapidly. Published MCP servers are available via:

  • npm packages: npx @company/mcp-server -- Node.js MCP servers installed and run via npx
  • pip packages: uvx package-name -- Python MCP servers installed via pip/uvx
  • Docker images: For isolated, reproducible server environments
  • GitHub repositories: Clone and run from source

Testing Tools

✅

Manual Testing

Use the MCP Inspector to interactively test each tool with sample inputs. Verify outputs match expectations, edge cases are handled, and error responses are well-structured.

🔧

Mock Claude Calls

Write scripts that simulate how Claude would call your tools. Test sequences of tool calls that mimic real workflows -- not just individual calls in isolation.

🚀

Integration Tests

Connect your MCP server to Claude Code in a test project and run real prompts. Verify that Claude selects the right tools, passes correct parameters, and handles errors gracefully.

Debugging Tool Selection

When Claude picks the wrong tool, the problem is almost always in the tool descriptions. Common causes:

  • Overlapping descriptions: Two tools that sound like they do the same thing. Differentiate them clearly.
  • Too vague: A description that could apply to multiple tools. Be specific about the tool's domain and use case.
  • Missing "when NOT to use" guidance: Tell Claude when the tool is inappropriate, not just when it is appropriate.
  • Too many similar tools: If you have 5 search tools, Claude may struggle to pick the right one. Consider consolidating or adding very clear differentiation in the descriptions.
💡

Debugging technique: Ask Claude "Why did you choose [tool X] instead of [tool Y]?" in the same conversation. Claude will explain its reasoning, revealing which parts of the description influenced its decision.

Debugging: Why Claude Picks the Wrong Tool

Follow the 4-step diagnostic to fix a tool selection problem.

1
The Problem

You built a custom search_docs MCP tool that searches your internal knowledge base using semantic search. But when you ask Claude Code to "find documentation about our authentication flow," Claude uses the built-in Grep tool instead.

This keeps happening. Claude never selects your custom tool. Why?

# What you asked: "Find documentation about our authentication flow" # What Claude did: Grep --pattern "authentication" --type "md" # (exact text search in markdown files) # What you wanted: search_docs --query "authentication flow" # (semantic search in knowledge base)
2
Check the Tool Description

Let us look at the current tool description. Here is what your search_docs tool exposes:

"name": "search_docs", "description": "Search documents."

Now compare that to Grep's built-in description:

"name": "Grep", "description": "A powerful search tool... Supports full regex syntax... Use for search tasks."

The problem is clear: Your description says "Search documents" -- but so does Grep, effectively. Claude sees two tools that both search, and it defaults to the one with the richer, more informative description: Grep.

3
Improve the Description

Rewrite the description to clearly differentiate your tool from Grep. Explain what it does, when to use it, and how it differs:

"name": "search_docs", "description": "Search the internal knowledge base by semantic meaning, returning ranked results with relevance scores. Use INSTEAD of Grep when searching for concepts, topics, or questions rather than exact text patterns. This tool understands natural language queries and finds conceptually related documents even when the exact keywords don't appear in the text. Input: a natural language query. Returns: ranked document excerpts with titles and relevance scores."

Key improvements: explains semantic vs exact search, says when to use it INSTEAD of Grep, describes input and output format.

4
Re-test and Verify

After updating the description, restart Claude Code and test again:

# Same request: "Find documentation about our authentication flow" # Claude's new behavior: search_docs --query "authentication flow" # Returns: 3 ranked results with relevance scores # 1. "OAuth2 Authentication Architecture" (0.94) # 2. "JWT Token Refresh Flow" (0.87) # 3. "SSO Integration Guide" (0.82)

Claude now correctly selects your semantic search tool for concept-based queries, while still using Grep for exact text pattern matching. The description made the difference.

💡

Takeaway: When Claude picks the wrong tool, always start by examining the tool description. In the vast majority of cases, that is where the problem lives. A great description is the single highest-leverage improvement you can make.
