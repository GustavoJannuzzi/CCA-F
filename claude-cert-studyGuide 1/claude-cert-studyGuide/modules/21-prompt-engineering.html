Prompt Engineering Lab

Master the art and science of crafting effective prompts for Claude. From explicit criteria and few-shot examples to structured output patterns and multi-pass review -- learn the techniques that transform vague requests into precise, reliable AI interactions.

⏱ ~90 min 📚 Intermediate 🎯 3 Interactive Exercises 👥 Developers, Architects & Consultants

1. The 5 Core Prompting Techniques

Prompt engineering is not guesswork. There are five core techniques that, when applied systematically, transform unreliable AI responses into consistent, high-quality output. These techniques compound -- the best prompts combine multiple approaches, and understanding each individually lets you know exactly which lever to pull when results fall short.

Here is an overview of each technique and when it delivers the most value.

1

Explicit Criteria

Use when: You need predictable, consistent output quality

Tell Claude exactly what to include, what to exclude, how to format, and what quality means. "Summarize this" is a wish. "Summarize in 3 bullets, each under 20 words, focusing on financial impact" is a specification.

2

Few-Shot Examples

Use when: Output format is nuanced or non-obvious

Show Claude 2-3 examples of input/output pairs. This is far more effective than describing the format in words. Include one "edge case" example to establish how Claude should handle ambiguity.

3

Chain-of-Thought

Use when: The problem requires multi-step reasoning

Ask Claude to reason step by step before answering. This dramatically improves accuracy on math, logic, debugging, and analysis tasks. For complex problems, use Claude's Extended Thinking mode.

4

Role Assignment

Use when: You need domain-specific depth or a particular communication style

Give Claude a specific persona: "You are a senior database architect with 15 years of PostgreSQL experience." This focuses the model's knowledge distribution and adjusts tone, vocabulary, and depth.

5

Structured Output

Use when: Your code needs to parse the response programmatically

Define the exact format -- JSON schema, XML tags, markdown template. For guaranteed structure, use tool_use with a JSON Schema definition. Never rely on Claude "usually" returning the right format.

💡

The power is in combination. The best prompts typically use 3-4 of these techniques together. For example: a role assignment + explicit criteria + structured output + one example. The next sections break down each technique in depth.

Combined Example: All 5 Techniques in One Prompt

Here is a real-world prompt that combines all five techniques. Notice how each technique plays a specific role:

5 Techniques Combined # Technique 4: Role Assignment You are a senior security engineer specializing in web application security (OWASP Top 10). # Technique 1: Explicit Criteria Analyze this HTTP endpoint for security vulnerabilities. Evaluate ONLY these categories: 1. Injection (SQL, XSS, command injection) 2. Broken Authentication 3. Sensitive Data Exposure 4. Missing Rate Limiting For each vulnerability found, provide: - Severity: CRITICAL / HIGH / MEDIUM / LOW - CWE number (if applicable) - Specific line reference - Exploitation scenario (one sentence) - Remediation code If a category has no issues, state "No issues found." Do not invent vulnerabilities. # Technique 5: Structured Output Return your analysis as JSON matching this schema: { "vulnerabilities": [{ "category": "string", "severity": "CRITICAL|HIGH|MEDIUM|LOW", "cwe": "CWE-xxx or null", "line": "number", "description": "string", "exploitation": "string", "remediation": "string (code)" }], "overall_risk": "CRITICAL|HIGH|MEDIUM|LOW", "summary": "string (max 50 words)" } # Technique 2: Few-Shot Example Example output for a SQL injection vulnerability: { "category": "Injection", "severity": "CRITICAL", "cwe": "CWE-89", "line": 15, "description": "User input concatenated into SQL query", "exploitation": "Attacker can extract entire database", "remediation": "Use parameterized queries" } # Technique 3: Chain-of-Thought Before producing the JSON, think through each endpoint parameter and trace how user input flows through the code to identify taint paths. Code to analyze: [paste code here]

This prompt leaves almost nothing to chance. The role focuses Claude's knowledge. The criteria define the scope. The structure ensures parseability. The example shows the expected format. The CoT instruction ensures thorough analysis.

2. Explicit Criteria -- The Foundation

Explicit criteria is the single most impactful technique in your prompt engineering toolkit. It is the difference between "Claude gave me something okay" and "Claude gave me exactly what I needed." Every underperforming prompt can be improved by adding more specific criteria.

Why Vague Prompts Fail

When you write "Summarize this article," you are leaving every decision to Claude: How long should it be? What format? What should it emphasize? Should it include quotes? What about statistics? Each of those decisions is a coin flip that might not land the way you want.

Compare these two prompts:

Vague Prompt Summarize this article.
Criteria-Driven Prompt Summarize this article in 3 bullet points. For each bullet: - State the key finding or decision (max 20 words) - Include the supporting data point or statistic - Note the source/speaker if attributed Focus on financial impact. Ignore editorial opinions. If a statistic is approximate, prefix with "~".

The second prompt leaves almost nothing to chance. Claude knows the length, format, focus, what to include, what to exclude, and how to handle edge cases.

The Criteria Framework

Every effective criteria-driven prompt addresses four dimensions:

WHAT to Include (and Exclude)

Be specific about what information should appear in the output. Equally important: tell Claude what to leave out. "Focus on technical risks. Do not include marketing language or competitive positioning." Exclusions prevent Claude from padding the response with irrelevant content.

HOW to Format It

Define the structure (bullets, table, JSON), length (word count, number of items), and style (formal, conversational, technical). "Return as a markdown table with columns: Risk, Likelihood, Impact, Mitigation." The more structural detail you provide, the less variance in the output.

Quality Standards

Set explicit quality bars. "Every claim must reference the specific section of the document it comes from." "Do not speculate -- only include information explicitly stated in the input." "If you are less than 80% confident in a classification, mark it as 'UNCERTAIN'."

Edge Case Handling

Tell Claude what to do when information is missing, ambiguous, or contradictory. "If no deadline is specified, write 'TBD' rather than guessing." "If two sources conflict, present both with a note about the discrepancy." This is the dimension most people forget -- and where most prompt failures occur.

Criteria Across Domains

Domain Example Criteria
Code Review Categorize issues as: Bug / Performance / Style / Security. Show the line number and a fix for each.
Data Extraction Return JSON. Use null for missing fields. All dates in ISO 8601. Monetary values as numbers without currency symbols.
Document Summary 3 bullets, max 20 words each, focusing on action items. Note the responsible person and deadline.
Classification Exactly one category from the provided list. Include confidence (high/medium/low) and one-sentence reasoning.
Risk Assessment Likelihood x Impact matrix. Sorted by risk score descending. Include specific mitigation steps, not generic advice.

Building Criteria Iteratively

You rarely get criteria perfect on the first try. The most effective workflow is iterative:

  1. Start with a minimal prompt and observe what Claude produces.
  2. Identify the gaps -- where did the output diverge from your expectations?
  3. Add criteria for each gap -- format rules, inclusion/exclusion lists, edge case handlers.
  4. Test again with the same input. Did the new criteria fix the issue?
  5. Repeat until the output is consistent across 3-5 runs.
🛈

The 3-run test: Run your prompt 3 times with the same input. If the outputs are structurally consistent (same format, similar length, same types of information), your criteria are working. If they vary significantly, you need more explicit criteria in the areas where they diverge.

Complete Example: Criteria-Driven Code Review

Here is a full, production-quality criteria-driven prompt for code review. Notice how every aspect of the expected output is specified.

Production Criteria-Driven Prompt Review this pull request diff for a Python FastAPI backend. Evaluate across these 4 categories (in order): ## 1. Correctness - Logic errors that would cause incorrect behavior - Unhandled exceptions that would crash the process - Race conditions in async code - Missing await keywords on async functions ## 2. Security - SQL injection (raw string interpolation in queries) - Path traversal (user input in file paths) - Missing authentication/authorization checks - Sensitive data exposure in logs or error messages - Hardcoded secrets or credentials ## 3. Performance - N+1 query patterns (queries inside loops) - Missing database indexes for frequent query patterns - Unbounded result sets (no LIMIT clause) - Blocking I/O in async handlers ## 4. Maintainability - Functions longer than 50 lines - Missing type annotations on public functions - Unclear variable names (single letters, abbreviations) - Duplicated logic that should be extracted For each issue: 1. Quote the exact line(s) from the diff 2. Explain the risk in one sentence 3. Rate severity: CRITICAL / HIGH / MEDIUM / LOW 4. Provide a corrected code snippet If no issues are found in a category, state: "No issues found in [category]." Do NOT invent issues. Only report problems you can demonstrate with specific code references. At the end, provide: - Overall assessment: APPROVE / REQUEST_CHANGES / BLOCK - Summary: 2-3 sentences explaining your recommendation

This prompt is long, but every line serves a purpose. The review will be structured, consistent, and actionable. Compare this to "Review this PR" -- which could produce anything from a one-liner to a novel.

3. Few-Shot Prompting

Few-shot prompting means providing Claude with 2-3 examples of the input/output pattern you want, directly in the prompt. Instead of describing the desired output format, you show it. This is one of the most reliable techniques for getting consistent, correctly-formatted responses.

When to Use Few-Shot

  • Output format is not obvious -- when you need a specific JSON structure, classification schema, or custom format that Claude would not default to.
  • Classification tasks -- sentiment analysis, ticket routing, priority assignment. Examples establish the decision boundary far better than descriptions.
  • Nuanced judgment calls -- when the "right answer" depends on contextual factors that are hard to describe in rules.
  • Tone or style matching -- when you need output that matches an existing voice (e.g., your company's writing style).

How Many Examples?

2-3 examples is the sweet spot. One example is often ambiguous -- Claude might latch onto incidental features. Two examples establish a pattern. Three examples with an edge case solidify it. More than three rarely helps and wastes context tokens.

⚠

Example quality matters more than quantity. Three carefully-chosen examples (covering common case, edge case, and negative case) outperform ten sloppy ones. Make sure your examples are correct -- Claude will faithfully reproduce errors in your examples.

Include Negative Examples

Negative examples -- showing Claude what not to do -- are surprisingly powerful. They define the boundary of acceptable output and prevent common failure modes.

Few-Shot with Negative Example Classify customer feedback into exactly ONE category. Categories: BUG, FEATURE_REQUEST, PRAISE, COMPLAINT, QUESTION Example 1 (common case): Input: "The app crashes when I upload a PDF" Output: {"category": "BUG", "confidence": "high"} Example 2 (edge case -- mixed signals): Input: "I love the new dashboard, but it would be even better with dark mode" Output: {"category": "FEATURE_REQUEST", "confidence": "medium"} Note: Even though there is praise, the primary intent is requesting a feature. Example 3 (what NOT to do): Input: "How do I reset my password?" WRONG: {"category": "COMPLAINT", "confidence": "low"} CORRECT: {"category": "QUESTION", "confidence": "high"} Reason: Asking for help is not a complaint.

Example: Full Classification Prompt

Production-Ready Few-Shot Prompt Classify this customer feedback. Return JSON with these fields: - category: exactly one of BUG | FEATURE_REQUEST | PRAISE | COMPLAINT | QUESTION - confidence: high | medium | low - sentiment: 1 (very negative) to 5 (very positive) - summary: max 15 words Example: Input: "Your app crashes every time I try to upload a PDF. This is so frustrating!" Output: {"category": "BUG", "confidence": "high", "sentiment": 1, "summary": "App crashes on PDF upload, user frustrated"} Example: Input: "Just wanted to say the new search feature is incredible. Saves me hours every week!" Output: {"category": "PRAISE", "confidence": "high", "sentiment": 5, "summary": "User loves new search feature, significant time savings"} Example: Input: "Can you add Slack integration? Our team lives in Slack and switching tools is painful." Output: {"category": "FEATURE_REQUEST", "confidence": "high", "sentiment": 2, "summary": "Requests Slack integration due to workflow friction"} Now classify this feedback: [paste feedback here]

Anti-Pattern: Too Many Examples

A common mistake is including 8-10 examples "to be thorough." This backfires for two reasons:

  • Token waste: Each example consumes context tokens. Ten examples might use 2,000+ tokens that would be better spent on actual input data.
  • Overfitting: With many examples, Claude can start matching surface patterns (keyword similarity) rather than understanding the underlying classification logic. Three diverse examples encourage generalization.
Number of Examples When to Use Risk
0 (zero-shot) Simple, well-understood tasks. Claude already knows the format. Format variance across runs
1 The format is unusual but the logic is simple. Claude may latch onto incidental features of the single example
2-3 (sweet spot) Classification, extraction, generation with specific format. Minimal -- this is the recommended range
4-6 Highly nuanced judgment where boundaries are subtle. Increased token cost; diminishing returns
7+ Almost never. Consider fine-tuning or few-shot with retrieval instead. Token waste, overfitting, reduced input capacity

Choosing Your Examples

Select examples that maximize coverage across the decision space:

  1. Example 1: The common case. Pick the most frequent input type. This anchors Claude's default behavior.
  2. Example 2: The edge case. Pick an input that is ambiguous or sits at the boundary between two categories. This teaches Claude how to handle uncertainty.
  3. Example 3: The negative case (optional). Show what the output should NOT look like. This is especially valuable for preventing common misclassifications.
💡

Pro tip: Use real data. If you are building a classification system, pull your examples from actual production data rather than inventing synthetic inputs. Real data captures the messiness and ambiguity that Claude will encounter in practice.

4. Chain-of-Thought & Extended Thinking

Chain-of-Thought (CoT) prompting asks Claude to reason through a problem step by step before giving a final answer. This is not a trick -- it fundamentally changes how the model processes the problem, allocating more computation to each reasoning step rather than jumping directly to a conclusion.

The Simplest CoT Trigger

Adding "Think step by step" to the end of a prompt is the simplest form of CoT. It works because it forces Claude to generate intermediate reasoning tokens before the final answer. Each reasoning token conditions the next, creating a chain of logical dependencies.

Without CoT A store has 3 shelves. Each shelf holds 4 boxes. Each box contains 6 red balls and 2 blue balls. If I remove all blue balls from the top shelf, how many balls remain in the entire store?
With CoT A store has 3 shelves. Each shelf holds 4 boxes. Each box contains 6 red balls and 2 blue balls. If I remove all blue balls from the top shelf, how many balls remain in the entire store? Think step by step before giving your final answer.

When CoT Helps

  • Math and arithmetic -- multi-step calculations where intermediate results matter.
  • Logic puzzles -- constraint satisfaction, deduction, syllogisms.
  • Multi-step reasoning -- "Given A and B, what follows? Given that, what else follows?"
  • Debugging -- tracing through code execution to find where behavior diverges from expectation.
  • Analysis -- weighing pros and cons, comparing multiple options, risk assessment.

When CoT Hurts

⚠

CoT is not always better. For simple factual recall ("What is the capital of France?"), simple classification, or straightforward formatting tasks, CoT adds latency and token cost without improving accuracy. Use it selectively.

Extended Thinking

Claude's Extended Thinking mode is a built-in, more powerful version of CoT. When enabled, Claude allocates a separate "thinking" budget before responding. The thinking is more thorough than a simple "think step by step" instruction and is especially effective for:

  • Complex architecture decisions -- weighing trade-offs across multiple dimensions.
  • Long-form code generation -- planning the structure before writing.
  • Multi-file refactoring -- understanding cascading impacts before making changes.
  • Ambiguous requirements -- identifying and resolving ambiguities before proceeding.

Example: Debugging with CoT

CoT Debugging Prompt This function should return the average of a list of numbers, but it sometimes returns incorrect results. ```python def calculate_average(scores): total = sum(scores) return total / len(scores) ``` Trace through the execution for these inputs and identify the bug: 1. scores = [80, 90, 70] (expected: 80.0) 2. scores = [] (expected: 0.0) 3. scores = None (expected: error message) Think step by step for each case. Then provide a fixed version that handles all three cases correctly.

CoT vs Direct Response: Side-by-Side

The difference is dramatic on problems that require reasoning. Here is what happens when Claude answers the ball-counting problem from above with and without CoT:

Without CoT

Claude's response: "There are 88 balls remaining in the store."

This answer is wrong. Claude jumped to a conclusion without working through the steps. The actual answer requires careful counting: 3 shelves x 4 boxes x 8 balls = 96 total, minus 4 boxes x 2 blue balls = 8 removed, leaving 88. In this case the answer happens to be 88, but without showing the work, there is no way to verify the reasoning -- and on slightly more complex problems, the error rate climbs significantly.

With CoT

Claude's response:

"Let me work through this step by step:

1. Total boxes: 3 shelves x 4 boxes = 12 boxes
2. Balls per box: 6 red + 2 blue = 8 balls per box
3. Total balls: 12 boxes x 8 balls = 96 balls
4. Blue balls on top shelf: 4 boxes x 2 blue = 8 blue balls removed
5. Remaining: 96 - 8 = 88 balls

There are 88 balls remaining in the store."

The CoT version not only gets the right answer but shows its work, making it easy to verify and debug. If the answer were wrong, you could pinpoint exactly which step went astray.

Structured CoT: Beyond "Think Step by Step"

For complex problems, you can structure the reasoning process itself. Instead of generic "think step by step," give Claude a specific reasoning framework:

Structured CoT for Architecture Decisions Should we use a message queue or direct API calls for our microservice communication? Analyze this decision using this framework: ## 1. Requirements Analysis - What are the throughput requirements? - What are the latency requirements? - Is ordering important? - Is exactly-once delivery required? ## 2. Evaluate Option A: Message Queue - Pros (with specific technical reasoning) - Cons (with specific technical reasoning) - Best suited when: [conditions] ## 3. Evaluate Option B: Direct API Calls - Pros (with specific technical reasoning) - Cons (with specific technical reasoning) - Best suited when: [conditions] ## 4. Recommendation - State your recommendation - Justify it against the requirements from step 1 - Note what would change your recommendation
🛈

When to use Extended Thinking vs manual CoT: Use Extended Thinking (API parameter) when the problem is genuinely complex and you want Claude to reason deeply without consuming visible output tokens for thinking. Use manual CoT ("think step by step") when you want to see the reasoning as part of the output -- useful for debugging, auditing, and building trust with stakeholders.

5. Structured Output Patterns

When your code needs to parse Claude's response programmatically, you need guaranteed structure -- not "Claude usually returns JSON." This section covers the patterns for ensuring Claude's output matches your schema every time.

JSON Output with tool_use

The most reliable way to get structured JSON from Claude is to define a tool with a JSON Schema and ask Claude to call it. Combined with tool_choice: "any", this guarantees that Claude will return valid JSON matching your schema.

Guaranteed JSON via tool_use // Define a tool with your desired output schema tools: [{ name: "extract_invoice_data", description: "Extract structured data from an invoice", input_schema: { type: "object", properties: { invoice_number: { type: "string" }, date: { type: "string", format: "date" }, vendor_name: { type: "string" }, line_items: { type: "array", items: { type: "object", properties: { description: { type: "string" }, quantity: { type: "number" }, unit_price: { type: "number" }, total: { type: "number" } }, required: ["description", "quantity", "unit_price", "total"] } }, total: { type: "number" } }, required: ["invoice_number", "vendor_name", "total"] } }] // Force Claude to call the tool tool_choice: { type: "any" }

XML Tags for Parsing

XML tags provide lightweight structure without requiring tool definitions. They are especially useful for separating reasoning from answers, or for multi-part responses.

XML Tag Pattern Analyze this code for security vulnerabilities. Structure your response using these XML tags: <thinking> Your analysis process (I will use this for logging) </thinking> <vulnerabilities> <vuln severity="high|medium|low"> <description>What the vulnerability is</description> <line>Line number</line> <fix>How to fix it</fix> </vuln> </vulnerabilities> <summary> One-paragraph executive summary </summary>

Markdown Templates

For human-readable output, define the exact heading structure and bullet format. Claude follows markdown templates very reliably.

Markdown Template Generate a technical design document using this template: ## Overview [1-2 sentences describing the feature] ## Requirements - [Bulleted list of requirements] ## Technical Approach ### Architecture [Describe the high-level architecture] ### Data Model | Field | Type | Description | |-------|------|-------------| | ... | ... | ... | ### API Endpoints #### `METHOD /path` - **Request:** [describe] - **Response:** [describe] ## Risks and Mitigations | Risk | Likelihood | Mitigation | |------|-----------|------------| | ... | ... | ... |

The Validation-Retry Loop

Even with tool_use, sometimes you need additional validation beyond what JSON Schema can express (business logic, cross-field dependencies, etc.). The validation-retry pattern handles this:

  1. Claude generates output -- using your tool or structured prompt.
  2. Your code validates -- check against schema, business rules, cross-field constraints.
  3. If invalid, send back with specific errors -- include the original output AND the specific validation failures.
  4. Claude fixes and retries -- with the error context, Claude almost always fixes the issue.
  5. Max 3 retries -- if it fails after 3 attempts, flag for human review.
⚠

Always include the specific validation error. "Try again" is useless. "The 'date' field '2024-13-01' is invalid -- month cannot be 13. Please use a valid ISO 8601 date." tells Claude exactly what to fix.

Validation-Retry in Practice

Validation-Retry Implementation async function extractWithRetry(input, maxRetries) { maxRetries = maxRetries || 3; var attempt = 0; var lastError = null; var lastOutput = null; while (attempt < maxRetries) { var prompt = attempt === 0 ? buildInitialPrompt(input) : buildRetryPrompt(input, lastOutput, lastError); var response = await claude.generate({ prompt: prompt, tools: [extractionTool], tool_choice: { type: "any" } }); var data = response.tool_calls[0].input; var validationResult = validate(data); if (validationResult.valid) { return data; // Success! } lastOutput = data; lastError = validationResult.errors; attempt++; } // All retries exhausted -- flag for human review throw new Error("Extraction failed after " + maxRetries + " attempts: " + lastError.join(", ")); } function buildRetryPrompt(input, failedOutput, errors) { return "Your previous extraction had errors.\n\n" + "Original input:\n" + input + "\n\n" + "Your previous output:\n" + JSON.stringify(failedOutput, null, 2) + "\n\n" + "Validation errors:\n" + errors.map(function(e) { return "- " + e; }).join("\n") + "\n\n" + "Please fix these specific errors and try again."; }

Choosing the Right Output Pattern

Pattern Best For Guarantees Tradeoff
tool_use + JSON Schema API integrations, data pipelines Valid JSON, correct field types Requires tool definition; adds complexity
XML tags Multi-part responses, reasoning + answer separation Parseable sections; flexible content within tags No schema validation; content within tags is unstructured
Markdown template Human-readable documents, reports Consistent structure; easy to render Hard to parse programmatically; headings may vary
Prompt-only JSON Quick prototyping; low-stakes applications Usually correct format No guarantee; may include markdown fences or extra text
💡

Rule of thumb: If your code will parse the output (JSON.parse, XML parser, regex), use tool_use. If a human will read the output, use markdown templates. If you need both (code parses it, then displays to human), use tool_use for extraction and render the parsed data in your UI.

How the Quality Scorer Works

The scorer evaluates your prompt across 5 dimensions, each scored 0-100:

  • Clarity (0-100): Length, specificity, detail level. Longer, more specific prompts score higher. Presence of words like "exactly," "specific," "for each" boost the score.
  • Structure (0-100): Does the prompt define an output format? Keywords like "JSON," "table," "bullet," "template," "format," or Markdown syntax (##, |) indicate structural intent.
  • Context (0-100): Does it provide background? Role assignments ("you are"), domain context ("given that"), and analytical verbs ("analyze," "evaluate") score higher.
  • Examples (0-100): Are there few-shot examples? Presence of "Example:", "Input:", "Output:", "e.g." indicates examples. Multiple examples score higher. Negative examples ("wrong," "incorrect") earn bonus points.
  • Constraints (0-100): Are there explicit quality rules? Words like "must," "never," "always," "max," "limit," "only," "exactly," "if missing" indicate constraints.

The Overall Score is the average of all 5 dimensions, mapped to a letter grade: A (80+), B (65-79), C (50-64), D (35-49), F (below 35).

6. Tool Descriptions -- The Art of Tool Design

If you are building applications that use Claude's tool use capability, the quality of your tool descriptions is the number one factor in whether Claude selects the right tool. A well-described tool gets selected correctly 95%+ of the time. A poorly-described tool causes misrouting, hallucinated parameters, and user frustration.

The 5 Rules of Effective Tool Descriptions

1

Explain WHAT the Tool Does

The tool name alone is not enough. "search_database" could mean full-text search, SQL query execution, or vector similarity search. The description should make the tool's purpose unambiguous: "Executes a read-only SQL query against the PostgreSQL analytics database. Returns results as an array of row objects."

2

Explain WHEN to Use It

Differentiate this tool from similar tools. "Use this tool when the user asks about historical data or trends. For real-time metrics, use the 'get_live_dashboard' tool instead." Without this guidance, Claude will guess when tools overlap.

3

Define Input Formats with Examples

Do not just say "query: string." Say "query: A SQL SELECT statement. Example: 'SELECT name, revenue FROM customers WHERE region = 'EMEA' ORDER BY revenue DESC LIMIT 10'." Examples in descriptions are just as powerful as examples in prompts.

4

Describe Output Format

"Returns a JSON object with 'results' (array of rows), 'row_count' (integer), and 'execution_time_ms' (integer). If no results match, returns an empty array -- not null." This prevents Claude from making incorrect assumptions about the return format.

5

Note Edge Cases and Limitations

"Maximum 1000 rows per query. Queries that take longer than 30 seconds will timeout. This tool only has read access -- INSERT, UPDATE, and DELETE statements will return an error." These boundaries prevent Claude from attempting impossible operations.

Common Pitfall: Overlapping Descriptions

⚠

The #1 cause of tool misrouting is overlapping descriptions. If you have both "search_knowledge_base" and "search_documents" and their descriptions sound similar, Claude will randomly pick one. Fix this by making the WHEN clause explicit: "search_knowledge_base: For FAQ-style questions where the answer exists in pre-curated articles." vs "search_documents: For research queries that need to scan raw uploaded documents."

A/B Testing Tool Descriptions

Treat tool descriptions like you treat code: test, measure, iterate. Log which tool Claude selects for each query type. If misrouting exceeds 5%, revise the description. Small wording changes can produce large accuracy improvements.

Logging Tool Selection for A/B Testing // Log every tool selection for analysis function logToolSelection(query, selectedTool, expectedTool) { var entry = { timestamp: new Date().toISOString(), query: query, selected: selectedTool, expected: expectedTool, correct: selectedTool === expectedTool, description_version: toolDescriptionVersion }; analytics.log("tool_selection", entry); } // After collecting 100+ entries, analyze accuracy // If accuracy < 95% for a tool, revise its description // Focus on the WHEN clause -- that is where most // misrouting originates

  • Differentiation: Does it explain when to use this tool vs similar alternatives?

7. Multi-Pass Review Patterns

A common misconception is that Claude can effectively review its own output within the same session. While Claude can self-critique, the review is biased by the reasoning context that produced the original output. The model tends to justify its earlier decisions rather than critically re-evaluate them. Multi-pass patterns solve this.

Solution 1: Two-Pass Generation

How It Works

Pass 1 (Generate): Claude produces an initial output -- code, document, analysis, etc.

Pass 2 (Critique and Improve): In the same session or a new one, ask Claude to review the output with a specific rubric. "Review this code for: (1) edge cases not handled, (2) performance issues, (3) security vulnerabilities. For each issue, explain the risk and provide a fix."

This works best when the critique rubric is different from the generation criteria, forcing Claude to evaluate from a new perspective.

Solution 2: Independent Reviewer

How It Works

Send the output to a fresh Claude instance (new conversation, no shared context) for review. The reviewing instance does not retain the reasoning that produced the original output, so it is not biased toward justifying earlier decisions.

This is more effective at catching subtle issues because the reviewer approaches the output cold -- the same way a human code reviewer would.

Solution 3: Specialized Passes

How It Works

Pass 1 (Local Analysis): Review each file or component individually. Focus on correctness within that unit.

Pass 2 (Cross-file Integration): Review how the components interact. Look for interface mismatches, missing error propagation, inconsistent naming, and integration gaps.

This pattern avoids attention dilution -- when Claude tries to review everything at once, it spreads its attention too thin and misses issues that would be obvious in a focused review.

💡

When to use which pattern: Two-pass is sufficient for most tasks. Use an independent reviewer when accuracy is critical (production code, legal documents, financial analysis). Use specialized passes for large codebases (10+ files) where a single review would exceed context limits.

Implementation Example

Independent Reviewer Pattern // Step 1: Generate code in Session A const generatedCode = await claude.generate({ prompt: "Write a function that validates email addresses..." }); // Step 2: Review in Session B (fresh context) const review = await claude.generate({ prompt: `Review this code. You did NOT write this code. Evaluate it objectively for: 1. Correctness (does it handle all valid RFC 5322 emails?) 2. Security (ReDoS vulnerability in regex?) 3. Edge cases (empty string, null, unicode, very long input) 4. Performance (is the regex efficient?) Code to review: \`\`\` ${generatedCode} \`\`\` For each issue, show the problematic line and a fix. If the code is correct for a category, say "No issues found." ` });

When to Use Each Pattern

Pattern Best For Cost (API calls) Effectiveness
Two-Pass Most tasks: drafts, code, analysis 2 calls (same session) Good -- catches obvious errors
Independent Reviewer Critical output: production code, legal, finance 2 calls (separate sessions) Very good -- catches subtle bias
Specialized Passes Large scope: 10+ files, full codebase reviews N+1 calls (one per unit + integration) Excellent -- avoids attention dilution

The Attention Dilution Problem

When Claude reviews a large codebase in a single pass, it divides its attention across all files simultaneously. A subtle bug in file 7 of 20 may get less scrutiny than the same bug would receive if file 7 were reviewed in isolation. This is not a model limitation -- it is a fundamental property of how attention mechanisms work across large contexts.

The specialized passes pattern directly addresses this by ensuring each file gets full attention during its local analysis pass. The integration pass then focuses exclusively on cross-cutting concerns, where the context of inter-file relationships is what matters most.

Specialized Passes for Large Codebase Review // Pass 1: Review each file individually var fileReviews = []; for (var i = 0; i < files.length; i++) { var review = await claude.generate({ prompt: "Review this single file for correctness,\n" + "security, and performance issues.\n\n" + "File: " + files[i].name + "\n" + files[i].content }); fileReviews.push({ file: files[i].name, review: review }); } // Pass 2: Cross-file integration review var integrationReview = await claude.generate({ prompt: "Given these individual file reviews,\n" + "identify cross-file issues:\n" + "- Interface mismatches between modules\n" + "- Inconsistent error handling patterns\n" + "- Missing error propagation\n" + "- Naming inconsistencies\n\n" + fileReviews.map(function(r) { return "## " + r.file + "\n" + r.review; }).join("\n\n") });
⚠

Cost consideration: Multi-pass patterns multiply your API costs. Use two-pass for general quality improvement. Reserve independent reviewer and specialized passes for high-stakes output where the cost of an error exceeds the cost of additional API calls.

Key Takeaways

Before you take the quiz, here is a summary of the most important concepts from this module:

1. Explicit criteria is the highest-leverage technique

Every underperforming prompt can be improved by adding more specific criteria. Define what to include, how to format it, quality standards, and edge case handling. This single technique accounts for more improvement than all others combined.

2. Few-shot examples: 2-3 is optimal

Show, do not describe. Choose examples that cover the common case, an edge case, and optionally a negative case. Quality matters more than quantity. Pull from real data when possible.

3. Chain-of-Thought is not always better

Use CoT for multi-step reasoning, math, debugging, and analysis. Skip it for simple factual recall and classification. Extended Thinking is the more powerful alternative for genuinely complex problems.

4. Use tool_use for guaranteed structured output

JSON Schema + tool_choice: "any" guarantees valid JSON matching your schema. Combine with a validation-retry loop for business logic that JSON Schema cannot express.

5. Tool descriptions determine tool selection accuracy

Explain what the tool does, when to use it (vs alternatives), input formats with examples, output format, and edge cases. The #1 cause of misrouting is overlapping descriptions.

6. Multi-pass review catches what self-review misses

An independent Claude instance is less biased than self-review. For large codebases, use specialized passes (local + integration) to avoid attention dilution.
