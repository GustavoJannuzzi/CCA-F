Multi-Agent Systems

Design, build, and operate systems where multiple AI agents collaborate to solve problems no single agent can tackle alone. Learn orchestration patterns, state management, and production-grade failure handling.

⏱ ~90 min 📚 Advanced 🎓 Architects 🎯 3 Interactive Exercises + Quiz

1. Why Multi-Agent?

A single AI agent is remarkably capable -- it can read files, write code, run tests, and reason through complex problems. But every single agent has hard limits. Understanding these limits tells you exactly when to reach for a multi-agent architecture.

The Limits of a Single Agent

  • Context window fills up. Even with 200K tokens, a single agent reading a large codebase, holding conversation history, and processing tool outputs will eventually run out of room. When context fills, the agent starts losing earlier information -- and quality degrades.
  • One agent cannot know everything. An agent that's excellent at React frontend work may struggle with Kubernetes deployment configs. Prompting a single agent with expertise in 10 different domains dilutes its performance in all of them.
  • Sequential bottleneck. A single agent processes one tool call at a time. If you need security review, performance analysis, and accessibility audit, a single agent does them one after another. Three agents do them simultaneously.
  • Blast radius. If a single agent hallucinates or makes an error deep into a 50-step workflow, all subsequent steps may build on that error. Multiple agents provide natural checkpoints and validation boundaries.

Benefits of Multi-Agent Architecture

Context Isolation

Each agent gets its own context window -- its own 200K tokens. A research agent can read 50 files without crowding the coordinator's context. When the research is done, only a concise summary flows back to the coordinator. This is the single most important benefit of multi-agent systems.

Specialization

An agent with a focused system prompt (e.g., "You are an expert security reviewer. Your only job is to find vulnerabilities.") consistently outperforms a generalist agent asked to "review this code for everything." Specialization allows shorter, more targeted prompts and more precise tool configurations.

Parallelism

Multiple agents can work simultaneously. Three parallel reviewers complete in the time of one. A research network that fans out to five specialists gathers information 5x faster in wall-clock time, even though total token usage is similar.

Fault Isolation

If the security reviewer agent fails, the performance reviewer and code quality reviewer continue independently. The coordinator can still synthesize partial results. In a single-agent system, one failure often derails the entire workflow.

The Fundamental Tradeoff

Multi-agent systems are not free. Every inter-agent communication adds latency and tokens. The coordinator must spend context summarizing tasks, dispatching work, and synthesizing results. There is always a coordination overhead vs. specialization benefit tradeoff.

💡

Rule of thumb: Use multi-agent when a task requires more than 3 distinct knowledge domains or more than 50K tokens of context. Below that threshold, a single well-prompted agent is simpler, cheaper, and often faster.

2. Orchestration Patterns

There are four major patterns for organizing multi-agent systems. Each has distinct strengths and is suited to different types of work. Understanding these patterns is the foundation for designing effective multi-agent architectures.

Pattern 1: Sequential Pipeline

Each agent passes its output to the next agent in a fixed order. The output of agent A becomes the input of agent B, whose output becomes the input of agent C.

Sequential Pipeline: A → B → C → D

Extract Agent
→
Validate Agent
→
Enrich Agent
→
Store Agent
  • Best for: Document processing, data transformation, staged analysis, ETL pipelines
  • Strengths: Predictable execution, easy to monitor, simple error handling
  • Weaknesses: Slowest overall -- total time = sum of all agents. No parallelism.
  • Example: Extract raw data from PDFs → Validate schema compliance → Enrich with external lookups → Store in database

Pattern 2: Parallel Fan-Out / Fan-In

A coordinator spawns multiple specialist agents simultaneously. They all work in parallel on different aspects of the same input. When all complete, their results are merged by a synthesizer.

Parallel Fan-Out / Fan-In

Coordinator
↓ fan-out
Security Reviewer
Performance Reviewer
Code Quality Reviewer
↓ fan-in
Synthesizer
  • Best for: Multi-aspect analysis, research, broad codebase exploration, code review
  • Strengths: Fast wall-clock time (limited by slowest agent, not sum). Each agent has full context for its domain.
  • Weaknesses: Higher total token cost. Synthesis step can be complex.
  • Example: Code review where Security + Performance + Style reviewers run simultaneously, then a Synthesizer combines their findings into a unified report.

Pattern 3: Hierarchical Delegation

A top-level coordinator delegates to team leads, who further delegate to workers. This creates a tree of management -- mirroring how large engineering teams actually operate.

  • Best for: Large-scale code generation, enterprise workflows, complex projects with multiple subsystems
  • Strengths: Scales to very large tasks. Each level can focus on its scope. Natural abstraction boundaries.
  • Weaknesses: Most complex to implement. Communication overhead at each level. Debugging requires tracing through the tree.
  • Example: Architect designs the system → Frontend Lead builds the UI components (delegating individual pages) while Backend Lead builds the API (delegating individual routes).

Pattern 4: Conditional Routing

A router agent inspects each incoming task and directs it to the most appropriate specialist. Different inputs take different paths through the system.

  • Best for: Customer support (route by topic), code review (route by language), document processing (route by type)
  • Strengths: Each specialist handles only what it's good at. Router can be a small, fast model (Haiku).
  • Weaknesses: Misrouting sends work to the wrong specialist. Needs a fallback for unrecognized categories.
  • Example: Incoming support ticket → Router classifies as billing/technical/account → Specialized agent handles it.
ⓘ

Combining patterns: Most production systems combine multiple patterns. A hierarchical system might use conditional routing at the top level and parallel fan-out within each branch. The patterns are building blocks, not mutually exclusive choices.

3. Task Decomposition Strategies

Before agents can collaborate, you need to break the task into pieces. How you decompose the work determines how well the multi-agent system performs. There are three approaches.

Fixed Decomposition

Predefined stages that always run in the same order, regardless of the input. The pipeline is designed once and executed many times.

Example: Document Processing Pipeline

Every document goes through: Parse → Classify → Extract Entities → Validate → Store. The stages never change, only the content varies.

  • Pro: Predictable execution time and cost. Easy to monitor and debug.
  • Con: Cannot adapt to unexpected findings. A simple document still goes through all stages.

Dynamic Decomposition

A coordinator agent analyzes the task and creates subtasks on the fly. The number of subtasks, their nature, and their dependencies are determined at runtime.

Example: "Build a feature for user onboarding"

The coordinator analyzes the request and decides it needs: database schema changes, API endpoint, email template, frontend form, and test suite. It creates five subtasks with the correct dependencies.

  • Pro: Adapts to the complexity of each specific task. Simple tasks get fewer agents.
  • Con: Harder to predict duration and cost. The coordinator can make poor decomposition decisions.

Hybrid: Fixed Phases, Dynamic Sub-tasks

Most production systems use this approach. The high-level phases are fixed (e.g., Plan → Build → Test → Deploy), but within each phase, the system dynamically decides what work is needed.

💡

The "agent-sized chunk" heuristic: Each subtask should be completable within 20-30 tool calls. If a subtask requires more, it should be decomposed further. If it requires fewer than 5, it's probably too granular and should be merged with another.

Dependency Analysis

After decomposing a task into subtasks, the critical question is: which tasks can run in parallel vs. which must be sequential?

  • Independent tasks: No shared inputs or outputs. Run in parallel. Example: Security review and performance review of the same code.
  • Dependent tasks: One task's output is another's input. Must be sequential. Example: Database schema must be created before API routes that reference those tables.
  • Partially dependent: Some outputs are shared. Can partially overlap. Example: Frontend can start with mock data while the backend API is being built, then integrate once both are done.
Strategy When to Use Predictability Adaptability
Fixed Well-understood, repeatable workflows High Low
Dynamic Novel, varying-complexity tasks Low High
Hybrid Production systems (most common) Medium Medium-High

4. State Management and Communication

The hardest part of multi-agent systems is not the individual agents -- it's the glue between them. How do agents share state? How do they communicate? How do you keep everything consistent?

Shared State Patterns

State File on Disk (JSON)

The simplest pattern. A single JSON file on disk that all agents read from and write to. The coordinator manages writes to prevent conflicts.

  • Pro: Simple, durable, human-readable, works with any tooling
  • Con: Needs locking for concurrent writes. Not suitable for real-time updates.

Event-Driven (WebSocket / SSE)

Agents publish events as they make progress. Other agents (and the UI) subscribe to these events. State is reconstructed from the event stream.

  • Pro: Real-time visibility. Agents can react to each other's progress.
  • Con: More complex infrastructure. Event ordering and replay need careful handling.

Database-Backed

Agents read from and write to a shared database. Transactions ensure consistency. Queries enable complex state lookups.

  • Pro: Most robust for production. Built-in concurrency control. Rich querying.
  • Con: Requires database infrastructure. More setup than a flat file.

Inter-Agent Communication

Hub-and-Spoke (Through the Coordinator)

Agents never communicate directly. All messages flow through the coordinator. Agent A sends results to the coordinator, which decides what to pass to Agent B.

When to use: Most systems. This is the default pattern. It keeps agents independent and gives the coordinator full control over information flow.

Via Shared Files

Agents read from and write to shared file locations. Agent A writes analysis-results.json, Agent B reads it. The file system is the communication channel.

When to use: Claude Code agents using the Agent tool. The spawned sub-agent can write files that the parent reads after it returns.

Via Database

Agents write their results to database tables. Other agents query those tables for input. This is hub-and-spoke with the database as the hub.

When to use: Production systems where durability and queryability matter more than simplicity.

Context Passing Best Practices

⚠

Critical rule: Never dump an agent's entire context to another agent. Summaries, not transcripts. Structured data, not raw text.

  • Pass structured data, not raw text. A JSON object with named fields is far more useful than a paragraph of natural language. Downstream agents can reliably extract what they need.
  • Include metadata. Every piece of data passed between agents should include: source (which agent produced it), confidence level, timestamp, and any caveats or limitations.
  • Summarize before passing. If Agent A reads 30 files and produces 40K tokens of analysis, summarize it to 2K tokens of key findings before passing to Agent B. This is the entire point of context isolation.
Structured Inter-Agent Message { "source_agent": "security-reviewer", "timestamp": "2025-01-15T14:30:00Z", "confidence": 0.92, "findings": [ { "severity": "high", "file": "src/auth/login.py", "line": 47, "issue": "SQL injection via unsanitized user input", "recommendation": "Use parameterized query" } ], "summary": "1 high-severity, 2 medium-severity issues found. No critical vulnerabilities." }

5. The Jumpstarter Pattern

The Jumpstarter pattern is a production-proven approach to multi-phase autonomous code generation. It has been used to generate over 50,000 lines of production code in a single run, with minimal human intervention.

How It Works

At the core is a state file -- .jumpstarter-state.json -- that tracks the entire workflow. A stop hook forces Claude to continue through all phases rather than stopping early. Each phase uses a specialized agent with deep expertise in its domain.

.jumpstarter-state.json { "current_phase": "api", "completed_phases": ["scaffold", "database"], "remaining_phases": ["api", "frontend", "tests", "qa"], "iteration": 3, "max_iterations": 15, "phase_results": { "scaffold": { "status": "complete", "files_created": 12 }, "database": { "status": "complete", "tables": 8 } } }

The Phase Agents

Scaffold Architect

Creates the entire directory structure, configuration files (package.json, tsconfig.json, vite.config.ts), and base templates. Establishes the project skeleton that all subsequent phases build upon.

Data Architect

Generates the database schema -- tables, relationships, indexes, and migrations. Uses the project specification to determine the data model. Produces SQL schema files and ORM models.

API Architect

Creates REST (or GraphQL) endpoints. Reads the database schema to understand available data, then generates routes, controllers, validation, and error handling. Follows patterns established by the scaffold.

Frontend Architect

Builds React components, pages, and state management. Reads the API endpoints to understand available data, then creates the UI that consumes them. Includes routing, forms, and data display.

QA Validator

Runs multi-phase testing: linting, type checking, unit tests, integration tests. Reads all generated code, verifies it compiles and passes tests. Reports any issues back for the coordinator to fix.

The Stop Hook Pattern

The key innovation is the stop hook. After each Claude Code session completes (or hits its context limit), the stop hook checks .jumpstarter-state.json. If there are remaining phases, it automatically restarts Claude with a fresh context window, pointed at the next phase. This allows the system to generate far more code than fits in a single context window.

⚠

Why this works: Each phase gets a clean 200K context window. The state file provides continuity between phases. The stop hook provides automation. The result: unlimited effective context for code generation, bounded only by the number of phases and the max_iterations safety valve.

6. Failure Handling and Recovery

In a multi-agent system, failures are not exceptional -- they are expected. Network calls fail, agents hallucinate, context windows overflow, and API rate limits trigger. Robust failure handling is what separates a demo from a production system.

Agent-Level Retries

The simplest recovery mechanism: if an agent fails, try again. But do it intelligently.

  • Retry 2-3 times maximum. If an agent fails 3 times on the same input, it is likely a systematic issue, not a transient failure.
  • Vary the approach on retry. Don't just re-run the same prompt. Simplify the task, provide more context, or use a different model.
  • Exponential backoff for rate limits. Wait 1s, then 2s, then 4s. Respect API rate limits.

Fallback Agents

If a specialist agent fails, try a generalist. A dedicated security reviewer might fail on an unusual codebase, but a general-purpose Opus agent can often produce a reasonable (if less targeted) analysis.

Fallback Pattern # Pseudocode: fallback agent strategy try: result = run_specialist_agent("security-reviewer", task) except AgentFailure: # Specialist failed -- fall back to generalist result = run_generalist_agent("opus-general", task) result.metadata["fallback"] = True

Graceful Degradation

When one agent in a parallel group fails, continue with partial results rather than failing the entire system. Three out of four reviewers completing is better than zero.

Example: Code Review with Partial Results

Security reviewer, performance reviewer, and style reviewer run in parallel. The security reviewer hits a rate limit and fails after retries. The coordinator still synthesizes results from the other two reviewers, noting that security review is missing. The final report includes a clear caveat: "Security review could not be completed -- manual review recommended."

Checkpoint Recovery

Save state after each successful phase. If the system crashes in phase 4, restart from the phase 4 checkpoint instead of re-running phases 1 through 3. This is exactly what .jumpstarter-state.json enables.

The max_iterations Safety Valve

Every multi-agent loop must have a hard upper bound on iterations. Without it, a confused coordinator can loop forever, burning tokens and producing nothing useful. Set max_iterations based on the expected complexity of the task, plus a comfortable margin.

Error Propagation

When an agent fails, the error context must flow up to the coordinator in a structured format.

Structured Error Report { "agent": "security-reviewer", "status": "failed", "error": "Rate limit exceeded after 3 retries", "attempts": 3, "partial_results": { "files_reviewed": 12, "issues_found": 2 }, "suggestion": "Retry in 60 seconds or use fallback agent" }

7. Cost and Performance Optimization

Multi-agent systems can be expensive if not designed carefully. The key optimization levers are model selection, parallelism, token budgets, and caching.

Model Selection Per Agent

Not every agent needs the most powerful model. Match the model to the task complexity.

Model Best For Cost Speed
Haiku Routing, classification, simple extraction, formatting Lowest Fastest
Sonnet Most coding work, analysis, research, writing Medium Fast
Opus Complex reasoning, architecture decisions, synthesis of conflicting findings Highest Slowest
💡

Common pattern: Use Haiku for the router agent, Sonnet for specialist workers, and Opus for the coordinator/synthesizer. This gives you the best quality where it matters most (synthesis) while keeping costs down for high-volume work (routing, individual analysis).

Parallel vs Sequential Cost

Parallel and sequential execution use the same total tokens. The difference is wall-clock time. Three parallel agents at 10K tokens each = 30K total. Three sequential agents at 10K tokens each = 30K total. But parallel finishes in ~10 seconds, while sequential takes ~30 seconds.

Token Budgets Per Agent

Set max_tokens on each agent to prevent runaway responses. An agent that should produce a 500-token summary does not need 4,096 output tokens. Tight budgets also encourage conciseness.

Caching Strategies

  • Prompt caching: If multiple agents share the same system prompt prefix, prompt caching reduces input token costs significantly.
  • Result caching: If the same file is analyzed by multiple agents, cache the file content so it is only read once.
  • Session caching: Reuse common context across invocations of the same agent on different inputs.

Monitoring

Track cost per agent, per phase, and per run. Over time, you'll identify which agents are the most expensive and whether they deliver proportional value. Common monitoring metrics:

  • Input tokens per agent per run
  • Output tokens per agent per run
  • Wall-clock time per phase
  • Retry count per agent
  • Total run cost (in dollars)
  • Success rate per agent
