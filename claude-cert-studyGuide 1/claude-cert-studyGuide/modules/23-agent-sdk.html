Agent SDK -- Building Agent Applications

Learn to build autonomous AI agents with the Claude Agent SDK. From simple one-shot queries to multi-turn conversations with custom tools, permission models, and safety guardrails -- this module covers everything you need to architect production agent systems.

⏱ ~90 min 📚 Advanced 🎓 Developers & Architects 🎯 4 Interactive Exercises

1. What is the Agent SDK?

The Claude Agent SDK (claude-agent-sdk) is a Python framework for building autonomous AI agents powered by Claude. It wraps the Claude Code CLI as a subprocess, giving you programmatic control over everything Claude Code can do -- reading files, writing code, running commands, and reasoning about complex tasks.

Where Claude Code is an interactive terminal experience, the Agent SDK lets you embed that same power into your own Python applications, CI/CD pipelines, automation scripts, and backend services. Think of it as the difference between using a calculator by hand and embedding a calculator into your software.

Key Concepts

⚙

Agents

Autonomous programs that use Claude to accomplish tasks. They can reason, plan, and execute multi-step workflows without constant human input.

🔧

Tools

Functions your agent can call -- both built-in (Read, Write, Bash) and custom tools you define via MCP or the @tool decorator.

☰

Orchestration

Managing multi-step workflows where the agent plans its approach, executes steps in sequence, and adapts based on results.

🛡

Guardrails

Safety constraints and permission models that control what an agent can and cannot do, protecting against unintended side effects.

Architecture Overview

The SDK communicates with Claude Code through a subprocess model. When you call query() or use ClaudeSDKClient, the SDK spawns a Claude Code process, sends your prompt and configuration, and receives structured responses. This architecture means your agent inherits all of Claude Code's capabilities -- its tool suite, its context management, and its reasoning ability.

Architecture Your Python App | v claude-agent-sdk (Python) | v Claude Code CLI (subprocess) | v Claude API (Anthropic) | v Tool Execution (Read, Write, Bash, Custom MCP Tools)
ⓘ

Prerequisite: You need Claude Code installed and authenticated before using the Agent SDK. The SDK shells out to the claude CLI, so it must be on your PATH and have a valid API key configured.

Installation

Terminal # Install the SDK pip install claude-agent-sdk # Verify installation python3 -c "from claude_agent_sdk import query; print('SDK ready')"

2. query() -- Single-Exchange Interactions

The query() function is the simplest entry point to the Agent SDK. It sends a single prompt to Claude, lets the agent work (potentially using multiple tools), and returns the final result. It is stateless -- each call is independent with no memory of previous calls.

Basic Usage

Python from claude_agent_sdk import query result = query( prompt="What files are in this directory?", options={ "system_prompt": "You are a helpful file explorer.", "permission_mode": "default", "allowed_tools": ["Read", "Glob", "Grep"], "cwd": "/path/to/project" } ) print(result.text)

Configuration Options

Option Type Description
system_prompt str Instructions that shape the agent's behavior and personality
permission_mode str "default", "acceptEdits", or "fullAuto" -- controls what the agent can do without asking
allowed_tools list Restrict which tools the agent can use (e.g., ["Read", "Grep"])
cwd str Working directory for the agent's file operations
model str Which Claude model to use (e.g., "claude-sonnet-4-20250514")
max_turns int Maximum number of reasoning/tool-use turns before the agent must respond

When to Use query()

  • One-shot tasks: "Analyze this log file and summarize the errors"
  • Simple Q&A: "What version of Node.js is configured in package.json?"
  • Script automation: Running as part of a CI/CD pipeline step
  • Batch processing: Looping over files, applying the same analysis to each

Practical Example: Automated Code Review

Python from claude_agent_sdk import query def review_pull_request(diff_text): """Automated code review for a pull request.""" result = query( prompt=f"Review this code diff for bugs, security issues, and style:\n\n{diff_text}", options={ "system_prompt": """You are a senior code reviewer. Focus on: 1. Security vulnerabilities 2. Logic errors 3. Performance issues 4. Code style consistency Provide specific, actionable feedback.""", "allowed_tools": ["Read", "Grep"], "max_turns": 10 } ) return result.text
💡

Pro tip: Use max_turns to prevent the agent from spending too long on a task. For simple analysis, 5-10 turns is usually sufficient. For complex multi-file tasks, you may need 20-30.

3. ClaudeSDKClient -- Multi-Turn Conversations

For ongoing conversations where you need to maintain state between messages, ClaudeSDKClient provides a stateful session. Unlike query(), the client remembers previous messages and maintains context across multiple exchanges.

Basic Usage

Python from claude_agent_sdk import ClaudeSDKClient client = ClaudeSDKClient(options={ "system_prompt": "You are a code review assistant.", "permission_mode": "acceptEdits" }) # First message response = client.send_message("Review the auth module for security issues.") print(response.text) # Follow-up (maintains conversation context) response = client.send_message("Now check the same module for performance issues.") print(response.text) # Clean up client.close()

query() vs. ClaudeSDKClient

Feature query() ClaudeSDKClient
State Stateless -- each call is independent Stateful -- context carries between messages
Use case One-shot tasks, automation Interactive sessions, iterative work
Memory No memory of previous calls Remembers all previous messages
Session New session every call Persistent session with session_id
Lifecycle Fire and forget Requires explicit close()

Session Management

Sessions can be paused and resumed using a session_id. This is useful for long-running workflows where you need to come back to a task later, or for building interactive applications where the user steps away and returns.

Python # Create a named session client = ClaudeSDKClient(options={ "system_prompt": "You are a refactoring assistant.", "session_id": "refactor-auth-module-v2" }) # Do some work... response = client.send_message("Identify all functions that need refactoring in auth.py") # Later, resume the same session client2 = ClaudeSDKClient(options={ "session_id": "refactor-auth-module-v2", "resume": True }) response = client2.send_message("Now refactor the first function you identified.")

Interrupts: Pausing for Human Input

The client supports interrupts -- pausing the agent mid-execution to get human input before continuing. This is essential for workflows where the agent might need clarification or approval at certain decision points.

⚠

Important: Always call client.close() when you are done with a session. The SDK spawns a subprocess that will continue running if not explicitly cleaned up. Use a try/finally block or a context manager pattern for safety.

4. Custom Tools with @tool


      

        The @tool decorator is how you extend your agent's capabilities beyond Claude Code's
        built-in tools. Any Python function decorated with @tool becomes an
        MCP tool that the agent can discover and call during its
        reasoning process.
      


      
Basic Tool Definition


      

        Python
        
from claude_agent_sdk import tool
import json

@tool
def search_jira(project_key: str, status: str = "open") -> str:
    """Search JIRA issues in a project.

    Args:
        project_key: The JIRA project key (e.g., "PROJ")
        status: Filter by status. Options: open, in_progress, done, all

    Returns:
        JSON string of matching issues with id, title, assignee, and priority
    """
    issues = jira_client.search(project_key, status)
    return json.dumps([{
        "id": i.key,
        "title": i.summary,
        "assignee": i.assignee,
        "priority": i.priority
    } for i in issues])


      
How It Works


      

        The SDK auto-generates a JSON Schema from your function's signature and docstring.
        This schema tells Claude what the tool does, what parameters it accepts, and what it returns. The
        quality of your docstring directly impacts how well Claude uses your tool.
      


      

        
The Anatomy of a Great @tool Function

        
    
          
  1. Descriptive function name -- the name itself tells Claude when to use it
    2. 
          
  2. Type-annotated parameters -- Python type hints become JSON Schema types
    3. 
          
  3. Default values -- optional parameters get default values
    4. 
          
  4. Rich docstring -- first line is the tool description, Args section describes each parameter
    5. 
          
  5. Return type hint -- tells Claude what to expect back
    6. 
          
  6. Structured return values -- JSON strings or plain text that Claude can parse
    7. 
        

      


      
Tool Annotations


      

        Annotations provide metadata that helps the permission system make smarter decisions about when
        to ask for human approval:
      


      

        

          
            

              Annotation
              Meaning
              Example
            

          
          
            

              readOnly=True
              Tool does not modify any state -- safe to auto-approve
              Database queries, API reads, search operations
            

            

              destructive=True
              Tool makes irreversible changes -- always ask for approval
              Deleting records, dropping tables, revoking access
            

            

              openWorld=True
              Tool accesses external systems outside the local environment
              API calls, sending emails, posting to Slack
            

          
        

      


      

        Python
        
@tool(annotations={"readOnly": True})
def get_user_profile(user_id: str) -> str:
    """Fetch a user's profile from the database.

    Args:
        user_id: The unique user identifier

    Returns:
        JSON string with user profile data
    """
    user = db.get_user(user_id)
    return json.dumps(user.to_dict())


@tool(annotations={"destructive": True})
def delete_user_account(user_id: str, reason: str) -> str:
    """Permanently delete a user account. This action is irreversible.

    Args:
        user_id: The unique user identifier
        reason: Reason for account deletion (for audit log)

    Returns:
        Confirmation message with deletion timestamp
    """
    result = db.delete_user(user_id, reason)
    return f"User {user_id} deleted at {result.timestamp}"


      

        💡
        

          
Best practice: Write your docstrings as if explaining the tool to a colleague
          who has never seen your codebase. Claude uses the docstring to decide when and how
          to call your tool. Vague docstrings lead to incorrect tool usage.

5. Human-in-the-Loop (HITL) Patterns


      

        Fully autonomous agents are powerful, but some decisions require human
        judgment. Human-in-the-Loop (HITL) patterns let your agent operate efficiently while
        pausing at critical decision points for human review and approval.
      


      
Why HITL Matters


      
    
        
  • High-stakes decisions: Deploying to production, deleting data, sending communications
    • 
        
  • Ambiguous situations: When the agent encounters conflicting requirements
    • 
        
  • Regulatory compliance: Actions that require documented human approval
    • 
        
  • Learning and trust-building: Gradually expanding agent autonomy as you gain confidence
    • 
      


      
Interrupt Patterns


      

        
Approval Gates

        
The agent pauses before executing destructive or high-impact actions. It presents what
        it plans to do and waits for explicit human approval before proceeding.

      


      

        
Confirmation Prompts

        
"Are you sure you want to delete 500 records?" -- The agent detects potentially dangerous
        operations and surfaces a confirmation dialog before continuing.

      


      

        
Review Checkpoints

        
The agent presents a complete plan (e.g., "I will modify 12 files in the following way...")
        and waits for human review of the plan before executing any changes.

      


      
Implementation with Permission Modes


      

        Python
        
from claude_agent_sdk import query

# The system prompt instructs the agent to pause before destructive actions
result = query(
    prompt="Deploy the latest build to staging",
    options={
        "permission_mode": "default",  # Will pause for confirmations
        "system_prompt": """Before any deployment action, present your plan
and wait for approval. Include:
1. What will be deployed
2. Which environment
3. Expected impact
4. Rollback plan"""
    }
)


      
The Permission Mode Continuum


      

        

          
            

              Mode
              File Edits
              Shell Commands
              Best For
            

          
          
            

              "default"
              Asks permission
              Asks permission
              Maximum safety, production environments
            

            

              "acceptEdits"
              Auto-approved
              Asks permission
              Development workflows, code generation
            

            

              "fullAuto"
              Auto-approved
              Auto-approved
              CI/CD pipelines with strict tool restrictions
            

          
        

      


      

        ⚠
        

          
Critical: Never use "fullAuto" without also restricting
          allowed_tools. An unrestricted agent in fullAuto mode can execute any shell
          command without asking -- this is powerful but dangerous if not properly scoped.

        

      

    


    
    

      
6. Safety and Guardrails


      

        Building safe agents is not optional -- it is a core architectural concern. The Agent SDK provides
        multiple layers of safety controls that work together to create a
        defense-in-depth strategy.
      


      
The Principle of Least Privilege


      

        Every agent should have the minimum set of permissions needed to accomplish its task. Do not
        give a code analysis agent the ability to write files. Do not give a documentation generator
        access to shell commands. Restrict the tool set to exactly what is needed.
      


      

        Python
        
# Read-only analysis agent -- cannot modify anything
result = query(
    prompt="Analyze this codebase for security vulnerabilities",
    options={
        "allowed_tools": ["Read", "Glob", "Grep"],  # No Write, Edit, or Bash
        "permission_mode": "default"
    }
)


      
Safety Layers


      

        

          

            
Permission Models

            
Choose the right permission mode for your use case. Start with "default" and
            only relax restrictions as you gain confidence in the agent's behavior.

          

        

        

          

            
Tool Restrictions

            
Use allowed_tools to create a whitelist of permitted tools. This is your
            primary defense against unintended side effects.

          

        

        

          

            
Sandboxing

            
Run agents in containers, virtual machines, or other isolated environments. If an agent
            goes rogue, the blast radius is contained to the sandbox.

          

        

        

          

            
Monitoring and Logging

            
Log every tool call, every decision, and every output. Use this audit trail for debugging,
            compliance, and improving agent behavior over time.

          

        

        

          

            
Rate Limiting

            
Prevent runaway behavior by limiting the number of tool calls, the execution time, or the
            number of files modified. The max_turns option is your first line of defense here.

          

        

      


      
The "Trust but Verify" Pattern


      

        This pattern balances efficiency with safety. Let the agent operate with reasonable autonomy for
        most operations, but validate critical actions through hooks and periodic human review.
      


      

        
How It Works

        
    
          
  1. Agent operates with acceptEdits -- auto-approves file changes for efficiency during development tasks
    2. 
          
  2. PostToolUse hooks validate critical operations -- custom logic inspects tool calls for dangerous patterns (e.g., modifying production configs, running rm -rf, accessing sensitive paths)
    3. 
          
  3. Periodic human review -- scheduled checkpoints where a human reviews the agent's recent actions and course-corrects if needed
    4. 
        

      


      

        Python
        
# Example: Restrict agent + monitor output
from claude_agent_sdk import query
import logging

logger = logging.getLogger("agent-audit")

result = query(
    prompt="Refactor the database module to use connection pooling",
    options={
        "permission_mode": "acceptEdits",
        "allowed_tools": ["Read", "Write", "Edit", "Glob", "Grep"],
        "max_turns": 25,
        "cwd": "/app/src"
    }
)

# Log the result for audit
logger.info("Agent task completed", extra={
    "result_length": len(result.text),
    "tool_calls": result.tool_call_count,
    "files_modified": result.files_modified
})


      

        💡
        

          
Pro tip: Start every new agent project with the most restrictive configuration
          possible. Only loosen restrictions after you have verified the agent behaves correctly in your
          specific context. It is much safer to add permissions than to remove them after an incident.
