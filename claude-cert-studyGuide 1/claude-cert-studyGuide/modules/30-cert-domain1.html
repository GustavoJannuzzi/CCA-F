2. Agentic Loop Mastery

The agentic loop is the fundamental execution pattern for all Claude-powered agents. Understanding how this loop works -- and more importantly, what can go wrong -- is essential for the exam.

The Loop Pattern

Every agentic system follows this core cycle:

  1. Send request to Claude with the user query (or continuation) and available tools
  2. Check stop_reason on the response to determine what Claude wants to do
  3. If "tool_use": execute the requested tool and append the result to the conversation
  4. If "end_turn": Claude is done -- exit the loop and return the response
  5. Repeat from step 1 with the updated conversation including tool results

Critical: stop_reason Values

stop_reason Meaning Action
"end_turn" Claude has finished and is ready to present its response Exit the loop, return response to the user
"tool_use" Claude wants to call one or more tools Execute the tool(s), append results, continue loop
"max_tokens" Output was truncated because max_tokens was reached May need a continuation request with higher limit

Model-Driven vs Pre-Configured

A critical concept for the exam: Claude decides which tool to call based on context. The agent loop does not follow a fixed sequence of tool calls. This is model-driven orchestration -- the model analyzes the situation and dynamically chooses the appropriate action at each step.

Anti-Patterns for the Exam

The exam specifically tests whether you can identify these common mistakes:

  • Anti-Pattern Parsing natural language for loop termination. Checking if Claude said "I'm done" or "Task complete" to decide whether to exit the loop. This is unreliable because Claude's phrasing varies and the model could use completion-like language while still intending to call more tools.
  • Anti-Pattern Arbitrary iteration caps as the sole exit condition. Setting max_iterations = 5 without checking stop_reason. This can terminate the loop before Claude finishes or waste iterations after Claude is done.
  • Anti-Pattern Using assistant text content as a completion indicator. Checking response.content[0].text instead of response.stop_reason. The text content may exist alongside tool_use blocks.
⚠

Exam Trap: If a question describes a loop that "works most of the time" by checking Claude's text for phrases like "Here is my final answer," this is ALWAYS the wrong approach. The stop_reason field is the only reliable signal.

Correct Agentic Loop (Python) import anthropic client = anthropic.Anthropic() messages = [{"role": "user", "content": query}] while True: response = client.messages.create( model="claude-sonnet-4-20250514", max_tokens=4096, tools=tools, messages=messages ) # The ONLY correct way to control the loop if response.stop_reason == "end_turn": break # Claude is done if response.stop_reason == "tool_use": # Execute each tool call and collect results tool_results = execute_tools(response.content) messages.append({"role": "assistant", "content": response.content}) messages.append({"role": "user", "content": tool_results})

3. Multi-Agent Orchestration for the Exam

The hub-and-spoke pattern is the primary multi-agent architecture tested on the certification exam. Understanding when and how to apply it is essential.

Hub-and-Spoke Architecture

Coordinator (Hub)

The coordinator agent manages ALL inter-agent communication. It acts as the central decision-maker:

  • Analyzes incoming queries and determines which subagents are needed
  • Routes tasks to appropriate subagents based on query complexity
  • Collects and synthesizes results from multiple subagents
  • Implements iterative refinement loops to fill gaps
  • Maintains observability by routing all communication through a single point

Subagents (Spokes)

Subagents are specialized workers with isolated context:

  • Each subagent has its own conversation history -- it does NOT inherit the coordinator's context
  • Subagents have restricted tool access based on their role
  • They receive explicit instructions and context from the coordinator
  • Results are returned to the coordinator for synthesis

Key Exam Concepts

  • Dynamic subagent selection: The coordinator must analyze query requirements and select which subagents to invoke. Not every query needs every subagent.
  • Parallel execution: Partition research across subagents to minimize duplication. When subagents have independent tasks, spawn them simultaneously.
  • Communication routing: ALL inter-agent communication goes through the coordinator. Direct peer-to-peer communication between subagents breaks observability.
  • Iterative refinement: The coordinator evaluates synthesis output for completeness. If gaps are found, it re-delegates targeted queries to specific subagents and re-synthesizes.
⚠

Common Exam Trap: Questions may present a system where subagents communicate directly with each other (peer-to-peer). This is almost always the wrong answer because it breaks observability and makes the system harder to debug and monitor.

4. Context Passing and Subagent Configuration

One of the most frequently tested areas is how context flows between agents. The exam assumes you understand that subagents have isolated context and requires you to design proper context passing strategies.

The Task Tool

The Task tool spawns subagents in the Claude Agent SDK. For coordinators to use this tool, their allowedTools configuration must explicitly include "Task".

Context Does NOT Inherit

⚠

Exam Trap: Subagents do NOT automatically see the parent coordinator's conversation history. If a search subagent found 5 research papers, and you want a synthesis subagent to use those findings, you must explicitly include those findings in the synthesis subagent's prompt. Simply assuming "the synthesis agent will know about the research" is wrong.

AgentDefinition Configuration

Each subagent is defined with specific configuration:

  • Description: What the agent does (used by the coordinator to decide which agent to invoke)
  • System prompt: Instructions, constraints, and persona for the subagent
  • Tool restrictions: Each subagent should only have access to tools it needs

Best Practices for Context Passing

  • Correct Pass complete findings in the prompt. Include all relevant data from prior agents directly in the subagent's task description.
  • Correct Use structured data formats. Separate content from metadata (source URLs, document names, timestamps) to preserve attribution and enable the subagent to cite its sources.
  • Correct Spawn parallel subagents. Emit multiple Task calls in one response to execute independent research tasks simultaneously.
  • Correct Design coordinator prompts with goals, not step-by-step procedures. Tell the coordinator what outcome you want, and let it decide the optimal delegation strategy.

5. Workflow Enforcement Patterns

This section covers one of the most critical distinctions on the exam: programmatic enforcement vs prompt-based guidance. Understanding when each approach is appropriate -- and when prompt-based guidance is insufficient -- separates passing from failing scores.

The Critical Distinction

Approach Reliability When to Use
Prompt-based guidance High but non-deterministic (95-99%) Preferences, style, soft constraints
Programmatic enforcement Deterministic (100%) Security, compliance, business rules

Prompt instructions have a non-zero failure rate for compliance. When you need guaranteed behavior -- such as verifying customer identity before issuing refunds -- you must use programmatic enforcement through hooks or prerequisite gates.

Prerequisite Gates

A prerequisite gate blocks a downstream tool call until a prior condition is met. For example, blocking process_refund until get_customer returns a verified customer ID:

Prerequisite Gate Pattern # PreToolUse hook that enforces identity verification def pre_tool_use(tool_name, tool_input, state): if tool_name == "process_refund": if not state.get("customer_verified"): return { "action": "block", "message": "Customer identity must be verified " "before processing refunds. Call " "get_customer first." } return {"action": "allow"}

Multi-Concern Request Decomposition

When a customer raises multiple issues in a single message (e.g., billing AND returns), the system must decompose the request into distinct items. Each item should be investigated independently -- potentially in parallel -- before synthesizing a unified resolution.

Structured Handoff for Escalation

When an agent needs to escalate to a human, it must provide a structured summary that includes:

  • Customer ID -- for the human agent to pull up the account
  • Root cause analysis -- what the AI agent determined about the issue
  • Actions already taken -- what steps the AI completed
  • Refund amount (if applicable) -- the calculated or requested amount
  • Recommended action -- the AI's suggestion for the human agent
★

Exam Key: If a question asks "what should the agent include in a human handoff?" and one option says "just the refund amount" while another says "customer ID, root cause, actions taken, and recommendation," the comprehensive option is always correct.

6. Hooks for Lifecycle Control

Hooks are the programmatic mechanism for intercepting and controlling agent behavior at specific points in the execution lifecycle. The exam tests three types of hooks:

Hook Types

PreToolUse Hooks

Execute before a tool call runs. Use cases:

  • Block policy-violating commands (e.g., rm -rf, DROP TABLE)
  • Enforce prerequisite conditions (identity verification before refund)
  • Add required parameters or headers to API calls
  • Redirect to alternative tools when primary tools are unavailable

PostToolUse Hooks

Execute after a tool call returns. Use cases:

  • Normalize data formats from different MCP tools (e.g., timestamps from Unix to ISO 8601)
  • Validate tool outputs against expected schemas
  • Strip sensitive information from tool results before Claude processes them
  • Run quality checks (e.g., eslint after code edits)

Stop Hooks

Execute when Claude signals end_turn. Use cases:

  • Force continuation when multi-phase work is incomplete (e.g., only 3 of 5 build phases done)
  • Check completion criteria against a state file or progress tracker
  • Append summary or reporting information before the session ends

When to Choose Hooks Over Prompts

The exam specifically asks you to distinguish between situations that need hooks versus prompt instructions:

  • Use Hooks when business rules require guaranteed compliance (financial operations, security boundaries, data retention policies)
  • Use Prompts when you want to guide behavior (code style preferences, response formatting, communication tone)
★

Exam Key: If the question involves money, security, or compliance, the answer is almost always hooks/programmatic enforcement. If it involves style, preferences, or soft guidelines, prompt instructions are acceptable.

7. Task Decomposition and Session Management

The final task areas in this domain test your ability to choose the right decomposition strategy and manage session state across complex workflows.

Decomposition Strategies

Strategy How It Works Best For
Fixed Sequential (Prompt Chaining) Predefined pipeline: Step A then Step B then Step C Well-understood workflows with predictable steps
Dynamic Adaptive The coordinator decides next steps based on results Open-ended research, complex debugging, unknown scope
Per-File + Integration Local passes on individual files, then cross-file integration Code review, large-scale refactoring, multi-file analysis

Code Review Example: Per-File + Integration

For a 500-line PR across 8 files, the recommended approach splits the review into:

  • Per-file local analysis passes: Each file is reviewed independently for file-specific issues (logic errors, style violations, single-file bugs). This avoids attention dilution.
  • Cross-file integration pass: A separate pass examines data flow between files, API contract mismatches, and consistency problems that only emerge when looking at multiple files together.

Session Management

The exam tests your understanding of session lifecycle management:

  • --resume <session-name>: Continue a named session where you left off. Useful for long-running tasks that span multiple work periods.
  • fork_session: Create parallel exploration branches from a shared baseline. Ideal for comparing two approaches without re-doing the initial analysis.
  • Fresh session with injected summaries: When tool results are stale (e.g., files have changed since the last session), start fresh but inject key summaries from prior work to avoid full re-exploration.
⚠

Exam Trap: The exam rewards candidates who match the approach to the task. Always choosing "dynamic adaptive" is wrong -- simple, well-understood tasks benefit from fixed pipelines. Always choosing "fixed sequential" is also wrong -- complex research tasks need adaptive decomposition. The correct answer depends on the specific scenario.

Resuming After Code Changes

When resuming a session after files have been modified:

  • Correct Inform the resumed session about specific file changes for targeted re-analysis
  • Correct If many files changed, start fresh with injected summaries instead of resuming
  • Anti-Pattern Assuming Claude will automatically detect what changed in files
