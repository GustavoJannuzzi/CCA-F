1. Domain Overview

Domain 5 of the Claude AI certification exam focuses on Context Management & Reliability and carries a 15% weight. This domain tests your ability to build AI systems that remain accurate over long conversations, degrade gracefully when errors occur, and know when to escalate to humans.

Exam Weight: Domain 5
15%
Context Preservation Escalation Patterns Error Propagation Codebase Exploration Confidence Calibration Information Provenance

Why This Domain Matters

Context management is the hidden complexity that separates toy demos from production systems. A chatbot that works perfectly for 5 turns may fail catastrophically at turn 50 when critical information has been pushed out of the context window. Similarly, a system that never escalates to humans will inevitably make confident-sounding errors on edge cases. This domain tests whether you can build systems that know what they know, preserve what matters, and fail gracefully.

The Six Task Areas

  • Conversation Context Preservation: Techniques for maintaining critical information across long conversations, including progressive summarization risks and the "lost in the middle" effect.
  • Escalation Patterns: When and how to hand conversations to humans. The exam focuses on explicit criteria for escalation (not sentiment-based), honoring human requests immediately, and proper handoff content.
  • Error Propagation: How to handle errors from tool calls, distinguish access failures from valid empty results, and annotate synthesized responses with coverage information.
  • Codebase Exploration: Maintaining accuracy during extended code exploration sessions, using scratchpad files, and delegating to subagents for verbose research.
  • Confidence Calibration: Field-level confidence annotations, stratified sampling for error detection, and handling temporal data (publication dates, filing dates).
  • Information Provenance: Tracking which claims come from which sources, distinguishing between stated facts and inferences, and mapping conclusions to their evidence.
ⓘ

Exam tip: This domain heavily tests your understanding of failure modes. Many questions present a system that appears to work but has a subtle reliability problem. Look for questions about what happens after 20+ conversation turns, what happens when one of three data sources fails, and what happens when the model encounters a case it has never seen before.

2. Context Preservation

Claude's context window is large but finite. In production systems -- especially multi-turn customer service agents, code exploration sessions, and research workflows -- conversations regularly approach or exceed context limits. The exam tests your understanding of what goes wrong and how to mitigate it.

Progressive Summarization Risks

A common approach to context management is progressive summarization: periodically replacing older messages with a summary. This sounds elegant but has a critical failure mode that the exam tests:

What Progressive Summarization Loses

  • Numerical values: "The customer's account balance is $4,237.89" gets summarized to "Customer discussed account balance" -- the exact amount is lost
  • Dates and deadlines: "Payment is due by March 15th" becomes "Customer asked about payment deadline"
  • Specific identifiers: "Order #ORD-2024-78341" becomes "Customer referenced an order"
  • Amounts and quantities: "3 items at $29.99 each" becomes "Customer ordered items"
  • Conditional terms: "If the claim exceeds $10,000, require supervisor approval" loses the threshold
⚠

Exam answer pattern: When asked about progressive summarization, the correct answer acknowledges that it loses transactional specifics. The mitigation is to extract critical facts (numbers, dates, IDs, amounts) into a persistent structured block that is never summarized.

The "Lost in the Middle" Effect

Research has shown that Claude (and all large language models) pay more attention to information at the beginning and end of the context, with reduced attention to content in the middle. This is called the "lost in the middle" effect.

The practical implication: if you place a critical instruction or fact in the middle of a long context, Claude may not apply it consistently. The exam tests two mitigation strategies:

  • Place summaries and critical facts at the beginning of the input (system prompt or early in the user message), not buried in conversation history.
  • Repeat key instructions in both the system prompt and the final user message for critical behaviors like escalation rules or output format requirements.

Extracting "Case Facts" into a Persistent Block

The recommended architecture for long-running conversations:

Case Facts Block Pattern # System prompt includes a structured facts block # that is updated after each turn but never summarized: <case_facts> customer_name: Jane Doe account_id: ACC-2024-55912 issue_type: billing_dispute disputed_amount: $147.50 disputed_date: 2024-03-01 order_id: ORD-2024-78341 resolution_offered: full_refund resolution_status: pending_approval escalation_history: none turns_elapsed: 12 </case_facts> # Conversation history is summarized, but case_facts # block is preserved verbatim across all turns.

Trimming Verbose Tool Outputs

Another context management technique: when tool calls return large amounts of data (e.g., a database query returning 500 rows, or a file read returning 1000 lines), trim the output before including it in the conversation. The exam tests your ability to recognize when verbose tool output is the primary cause of context exhaustion.

Requiring Subagent Structured Outputs

When using subagents (via the Agent tool or separate API calls) for research tasks, require them to return structured summaries with metadata rather than raw conversation transcripts. This compresses the information before it enters the main conversation's context:

Subagent Output Format { "findings": [ { "claim": "Authentication uses JWT with 24h expiry", "source_file": "src/auth/jwt_handler.py", "source_line": 42, "confidence": "high" }, { "claim": "Rate limiting is 100 req/min per API key", "source_file": "src/middleware/rate_limit.py", "source_line": 18, "confidence": "high" } ], "files_examined": 23, "gaps": ["Could not determine session storage backend"] }
💡

Key pattern: The subagent reads 23 files and produces a compact summary. Only the summary enters the main conversation's context. Without this pattern, the 23 file reads would consume most of the main context window.

3. Escalation & Ambiguity

Escalation -- knowing when to hand a conversation to a human -- is one of the highest-stakes decisions in a customer-facing AI system. The exam tests whether you understand the correct escalation triggers and common anti-patterns.

Explicit Escalation Criteria

The exam presents scenarios where you must identify the correct trigger for escalation. The key principle: escalation should be based on explicit, observable conditions, not on sentiment analysis or frustration detection.

Trigger Type Example Reliability
Explicit request "I want to speak to a human" / "Transfer me to a manager" Always honor immediately
Policy gap Customer asks about a scenario not covered by documented policies High reliability -- clear decision boundary
Unable to make progress 3 failed attempts to resolve the same issue High reliability -- measurable condition
Identity ambiguity Multiple accounts match the customer's description High reliability -- requires human judgment
Sentiment-based "Customer seems frustrated" based on tone analysis LOW reliability -- tone is culturally variable and easy to misread

Honor Explicit Human Requests Immediately

When a customer explicitly asks for a human, the only correct response is to initiate the handoff. Do not attempt to resolve the issue first, ask "Are you sure?", or explain that the AI can help. This is a trust issue: if the customer has to ask twice, they lose trust in the entire system.

Correct Handoff Pattern

Agent Response "I understand you would like to speak with a human agent. Let me transfer you now. I will include a summary of our conversation so you will not need to repeat yourself. Summary for the agent: - Customer: Jane Doe (ACC-2024-55912) - Issue: Billing dispute for $147.50 on order ORD-2024-78341 - What we tried: Verified the charge, offered full refund - Status: Customer declined refund, requesting supervisor review - Customer sentiment: Professional but firm Transferring now..."

Acknowledge Frustration But Offer Resolution

When a customer expresses frustration but does not request a human, the correct response is to acknowledge their frustration and continue working toward resolution. Escalation is only appropriate if you cannot make further progress or the customer explicitly requests it.

Ask for Identifiers When Multiple Matches

When a customer provides ambiguous identifying information (e.g., "I placed an order last week" but has multiple recent orders), ask for specific identifiers rather than guessing:

  • "I can see several recent orders on your account. Could you provide the order number? It starts with ORD- and was included in your confirmation email."
  • Do not pick the most recent order and assume it is the right one
  • If the customer cannot provide an identifier, describe each option briefly and ask them to confirm

Escalate on Policy Ambiguity

When the customer's situation falls outside documented policies, escalate rather than inventing a policy. Common examples: unusual refund amounts, cross-border regulatory questions, requests involving legal liability. The correct response acknowledges the limitation:

Policy Gap Escalation "This situation involves a cross-border return which requires review of specific regulatory requirements that are outside my documented guidelines. I want to make sure you get accurate information, so I am going to connect you with a specialist who handles international returns. One moment please."
⚠

Exam focus: The exam will present a frustrated customer and ask whether to escalate. The correct answer depends on whether the customer explicitly requests a human, not on how frustrated they seem. Frustration alone is not an escalation trigger -- inability to resolve the issue is.

4. Error Propagation Strategies

In multi-tool, multi-step AI systems, errors are inevitable. A database call times out. A file is not found. An API returns unexpected data. The exam tests how you handle these errors -- not whether they occur, but how they propagate through the system and how the final output communicates what happened.

Structured Error Context

When a tool call fails, capture structured information about the failure rather than just "an error occurred." This enables downstream processing to make informed decisions:

Structured Error Object { "error_type": "database_timeout", "attempted_action": "fetch customer order history", "partial_results": "Retrieved 3 of estimated 12 orders before timeout", "alternatives": [ "Retry with narrower date range", "Use cached results from last successful query (2 hours old)", "Proceed with partial results and note the gap" ], "impact_on_response": "Cannot provide complete order history" }

Distinguish Access Failures from Valid Empty Results

This is a critical distinction that the exam tests. Consider two scenarios:

  • Access failure: The database query timed out, returning no results. This means we do not know whether results exist. The correct annotation is "unable to retrieve" -- not "no results found."
  • Valid empty result: The database query succeeded and returned zero rows. This means we know there are no matching records. The correct annotation is "confirmed: no results."
⚠

Exam trap: A question may describe a system that reports "No orders found for this customer" when the actual failure was a database timeout. The correct answer identifies this as a dangerous conflation of "access failure" with "valid empty result" that could cause the agent to make incorrect decisions (e.g., telling a customer they have no orders when in fact the query just failed).

Local Recovery for Transient Failures

Not every failure needs to propagate to the user. Transient failures (network timeouts, rate limits, temporary service unavailability) should be retried locally before surfacing to the user:

  • Retry with backoff for timeout errors (1s, 2s, 4s)
  • Use cached data when fresh data is unavailable, annotating the staleness
  • Narrow the query if a broad query times out (e.g., reduce date range)
  • Surface clearly only after recovery attempts fail

Coverage Annotations in Synthesis

When synthesizing information from multiple sources, annotate which parts of the response are well-supported and which have gaps. This is especially important when some sources succeeded and others failed:

Coverage-Annotated Response "Based on available data: Account summary [COMPLETE - all 3 sources available]: - Name: Jane Doe - Balance: $4,237.89 - Status: Active since 2019 Recent orders [PARTIAL - 3 of 12 orders retrieved]: - ORD-2024-78341: $147.50 (March 1) - ORD-2024-76902: $89.00 (Feb 15) - ORD-2024-75180: $212.30 (Feb 2) Note: Order history may be incomplete due to a temporary database issue. The remaining orders could not be retrieved. Loyalty points [UNAVAILABLE - service timeout]: - Unable to retrieve loyalty points at this time. This does not mean you have zero points."

Anti-Patterns

  • Silent suppression: Swallowing errors and presenting incomplete data as if it were complete. The user has no idea they are seeing partial results.
  • Terminating on single failure: If 3 out of 4 data sources succeed, the correct response presents the available data with a note about the gap -- not "I could not process your request."
  • Cascading failure inflation: Reporting "multiple system errors" when only one service failed and the others succeeded normally.

5. Codebase Exploration & Confidence

Context Degradation in Extended Sessions

A phenomenon the exam tests: after many turns of codebase exploration (reading files, running searches, analyzing results), Claude's responses can degrade from specific, well-sourced answers to vague "typical patterns" answers. This happens because:

  • Earlier file reads have been pushed out of the context window or compressed by summarization
  • Claude begins relying on its training data knowledge of "typical" patterns rather than the actual code
  • Specific function names, line numbers, and parameter names get replaced by generic descriptions

The telltale sign: Claude starts saying things like "this likely uses a repository pattern" instead of "the UserRepository class at src/repos/user.py:34 implements get_by_id() with a SQLAlchemy query." When answers become generic, the context has degraded.

Scratchpad Files for Persisting Findings

A practical mitigation for context degradation: write findings to a scratchpad file as you discover them. This creates a persistent record that survives context window turnover:

Scratchpad Pattern # .claude/scratchpad.md # Written by Claude during exploration, read back when needed ## Authentication Flow - Entry point: src/auth/router.py:login() (line 45) - Calls: src/auth/service.py:authenticate(email, password) - JWT creation: src/auth/jwt.py:create_token() with 24h expiry - Token stored: HttpOnly cookie named "session_token" - Refresh: src/auth/router.py:refresh() (line 78) ## Database Access - ORM: SQLAlchemy with async sessions - Session factory: src/db/session.py:get_session() - Migrations: Alembic in migrations/ directory - Models: src/models/ (User, Order, Product, Payment) ## Open Questions - [ ] Where is rate limiting configured? - [ ] How are background jobs scheduled?

Subagent Delegation for Verbose Research

When a research task requires reading many files, delegate to a subagent rather than reading everything in the main conversation. The subagent:

  1. Gets its own context window (not competing with the main conversation)
  2. Performs all the verbose file reads and searches
  3. Returns a compact structured summary to the main conversation
  4. The main conversation preserves its context for decision-making

Using /compact for Long Sessions

The /compact command in Claude Code triggers an explicit context compression. The exam tests when this is appropriate:

  • Use /compact after completing a discrete subtask (e.g., finished exploring auth, moving to database layer)
  • Use /compact when Claude's answers become noticeably generic or when it starts hallucinating file paths
  • Do not use /compact in the middle of a multi-step operation that depends on recent context
  • Before using /compact, ensure critical findings are written to a scratchpad file

Structured State for Crash Recovery

In long-running tasks, periodically save structured state that allows the task to be resumed after a crash or session timeout. This is especially important for multi-file refactoring or migration tasks:

Recovery State { "task": "migrate_auth_to_oauth2", "completed_files": ["src/auth/basic.py", "src/auth/session.py"], "pending_files": ["src/auth/middleware.py", "src/routes/protected.py"], "decisions_made": { "token_storage": "HttpOnly cookie (not localStorage)", "refresh_strategy": "sliding window, 7-day max" }, "last_checkpoint": "2024-03-15T14:30:00Z" }

Field-Level Confidence Calibration

Rather than providing a single confidence score for an entire extraction, provide field-level confidence annotations. This allows downstream systems to handle high-confidence and low-confidence fields differently:

  • High confidence: Value was explicitly stated in the source with no ambiguity
  • Medium confidence: Value was inferred from context or partially stated
  • Low confidence: Value is a best guess based on patterns in the source
  • Not found: Value could not be determined from the source at all

Stratified Sampling for Error Detection

When processing large batches of documents, you cannot manually review every result. Stratified sampling -- reviewing a proportional sample from each confidence level -- is more effective than random sampling:

  • Review 100% of "low confidence" results (small number, high error rate)
  • Review 20% of "medium confidence" results
  • Review 5% of "high confidence" results
  • This catches more errors per review hour than random sampling across all results

Temporal Data Handling

A subtle exam topic: when extracting dates from documents, distinguish between:

  • Document date: When the document was created or published
  • Event date: When the event described in the document occurred
  • Effective date: When a policy, contract, or rule takes effect
  • Filing date: When the document was officially filed or submitted

Conflating these dates is a common extraction error. The exam may present a scenario where Claude extracts a "date" without specifying which type, leading to downstream errors.

💡

Provenance pattern: For every factual claim in a synthesized response, maintain a claim-source mapping. Each claim should trace back to a specific source document, page, or section. This enables verification and builds trust with users who need to audit the AI's reasoning.
