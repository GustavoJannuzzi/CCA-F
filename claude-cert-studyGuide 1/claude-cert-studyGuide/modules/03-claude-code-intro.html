Claude Code -- Your AI Pair Programmer

Learn how to use Claude Code, an AI coding assistant that lives in your terminal. It reads files, writes code, runs commands, and executes multi-step tasks autonomously. This module covers installation, the core tools, effective workflows, and hands-on practice with a CLI simulator.

⏱ ~75 min 📚 Beginner 🎓 Everyone 🎯 4 Interactive Exercises

1. Getting Started with Claude Code

Claude Code is an AI coding assistant that lives in your terminal. It is not a chatbot and it is not an autocomplete plugin. It is an autonomous agent that can read your files, write code, run commands, search your codebase, and manage complex multi-step tasks -- all while reasoning about the best approach.

Think of it as a highly skilled pair programmer who sits inside your terminal, has deep knowledge of software engineering, and can actually touch your files and run your tools. You describe what you want, and Claude figures out how to accomplish it.

What Makes Claude Code Different?

Unlike browser-based AI chat tools where you copy-paste code, or IDE extensions that offer line-level suggestions, Claude Code operates on your actual project. It has full access to your filesystem, can run any command in your terminal, and executes multi-step workflows autonomously.

Capability Browser AI Chat IDE Autocomplete Claude Code
Where it runs Browser tab IDE extension Your terminal (local machine)
Read your files Only if you paste them Current file + neighbors Any file on your filesystem
Write/edit files No -- you copy/paste Inline suggestions only Yes -- creates, edits, overwrites
Run commands No No Yes -- full shell access
Multi-step tasks Manual back-and-forth Single completions Autonomous multi-step execution
Codebase search No Limited Grep, Glob, full regex

Installation

Getting started requires Node.js (v18 or later). Install Claude Code globally via npm:

Terminal # Install Claude Code globally npm install -g @anthropic-ai/claude-code # Verify installation claude --version

First Launch

Navigate to any project directory and type claude. That is it. Claude Code will start, read your project's CLAUDE.md file (if one exists), and present an interactive prompt where you type natural-language requests.

Terminal # Start Claude Code in your project cd ~/my-project claude # Or run a one-shot command claude "What does this project do?"

The interactive prompt is where you type requests and watch Claude work. You can ask it to explore the project, fix bugs, add features, run tests, create files, and more.

💡

Quick demo: Try running claude "What does this project do?" in any project directory. Claude will read key files (package.json, README, entry points) and give you a comprehensive overview -- without you needing to specify which files to read.

ⓘ

Note: Claude Code requires an Anthropic API key set as the environment variable ANTHROPIC_API_KEY. Your organization may provide this through a centralized configuration. Check with your team lead if you are unsure.

2. The Agentic Loop -- How Claude Code Thinks

Understanding how Claude Code works under the hood is essential to using it effectively. Claude Code operates through a tool-use loop -- a cycle of observing, thinking, acting, and observing results that repeats until the task is complete.

The Core Loop: Observe, Think, Act, Repeat

Claude does not just answer your question and stop. It plans, executes, observes results, and adjusts. Each interaction follows this pattern:

  1. Observe -- Claude receives your request or the output from a previous tool call.
  2. Think -- Claude analyzes the situation and decides what needs to happen next.
  3. Act -- Claude picks the right tool and executes it with the appropriate parameters.
  4. Observe results -- Claude reads the tool's output (file contents, command output, search results).
  5. Repeat or respond -- If the task is not done, loop back to step 2. If done, present the result.

Example Walkthrough

Suppose you ask: "Add error handling to the login function." Here is what the agentic loop looks like:

You ask Claude

"Add error handling to the login function."

Claude uses Glob to find login files

Glob: **/*login* -- Finds src/auth/login.ts, src/auth/login.test.ts

Claude uses Read to examine the function

Read: src/auth/login.ts -- Reads the file and understands the current code structure.

Claude uses Edit to add try/catch blocks

Edit: src/auth/login.ts -- Adds error handling with proper error types and logging.

Claude uses Bash to run tests and verify

Bash: npm test -- --testPathPattern login -- Runs tests to ensure nothing broke.

Claude reports results to you

"Added error handling to the login function in src/auth/login.ts. All 8 tests pass."

The Permission System

Claude Code does not execute tools silently. Every tool call requires your permission unless it has been pre-approved in your settings. When Claude wants to use a tool, you will see the tool name and parameters, and you can approve or reject:

  • Approve the action (press Enter or type "y")
  • Reject the action (type "n") -- Claude will re-plan
  • Auto-approve certain tools via settings for faster workflows
⚠

You stay in control: Even though Claude is powerful, always review tool calls before approving -- especially Bash commands that could modify files, install packages, or interact with production systems. Trust, but verify.

3. The Core Tools Arsenal

Claude Code has eight built-in tools. Each one is purpose-built for specific operations. Understanding when and how to use each tool is the foundation of working effectively with Claude Code. Here is your complete reference.

📄

Read

File I/O

Reads a file from the local filesystem and returns its contents with line numbers. This is Claude's primary way to understand existing code. It can read text files, images (Claude is multimodal), PDFs, and Jupyter notebooks.

When to use it

  • Understanding an existing file's structure and contents
  • Reviewing code before making edits (required before Write on existing files)
  • Reading configuration files, logs, or documentation
  • Viewing images, screenshots, PDFs, or notebooks

Key parameters

  • file_path (required) -- absolute path to the file
  • offset -- start reading from this line number (for large files)
  • limit -- only read this many lines
  • pages -- page range for PDFs (e.g., "1-5")

Example

# Read an entire file Read: file_path=/src/auth/login.ts 1 import { hash, compare } from 'bcrypt'; 2 import { signToken } from './jwt'; 3 4 export async function login(email: string, password: string) { 5 const user = await findUserByEmail(email); 6 // ... # Read only lines 50-80 of a large file Read: file_path=/src/engine.ts offset=50 limit=30
💡

Pro tip: For large files, use offset and limit to read only the section you need. This saves tokens in your context window.

✍

Write

File I/O

Creates a new file or completely overwrites an existing file. This is the go-to tool for creating brand-new files from scratch. For modifying existing files, the Edit tool is preferred.

When to use it

  • Creating a new file that does not exist yet
  • Completely rewriting a file (rare -- prefer Edit for partial changes)
  • Generating configuration files, new components, or boilerplate

Example

# Creating a new test file Write: file_path=/src/auth/login.test.ts content="import { login } from './login'; import { describe, it, expect } from 'vitest'; describe('login', () => { it('should return a token for valid credentials', async () => { const result = await login('user@test.com', 'password123'); expect(result.token).toBeDefined(); }); });"
⚠

Important: Claude must Read an existing file before it can Write to it. This safeguard ensures Claude understands the current contents before overwriting.

🔧

Edit

File I/O

Makes targeted, surgical edits to existing files. Instead of rewriting the whole file, Edit replaces a specific string with a new string. This is the preferred tool for modifying existing code because it minimizes the chance of accidentally changing unrelated code.

When to use it

  • Fixing a bug in a specific function
  • Adding a new import statement
  • Renaming a variable across a file (with replace_all)
  • Any targeted change to existing code

Key parameters

  • file_path (required) -- the file to modify
  • old_string (required) -- the exact text to find and replace
  • new_string (required) -- the replacement text
  • replace_all -- if true, replaces all occurrences (default: false)

Example

# Rename a function Edit: file_path=/src/services/user.ts old_string="function getUserData(" new_string="function fetchUserProfile(" Applied edit to /src/services/user.ts # Rename a variable everywhere in a file Edit: file_path=/src/models.ts old_string="user_name" new_string="username" replace_all=true Replaced 7 occurrences
ⓘ

How it works: The old_string must be unique in the file (unless using replace_all). If it is not unique, the edit will fail. To fix this, include more surrounding context in the old_string.

💻

Bash

System

Executes shell commands and returns their output. This is Claude's interface to the operating system -- it can run tests, install packages, manage git, start servers, and anything else you would normally do in a terminal.

When to use it

  • Running tests: npm test, pytest, cargo test
  • Installing dependencies: npm install, pip install
  • Git operations: git status, git diff, git commit
  • Building projects: npm run build, make
  • Running scripts: python3 script.py, node index.js

Example

# Run the test suite Bash: command="npm test" PASS src/tests/auth.test.ts (1.2s) PASS src/tests/users.test.ts (0.8s) FAIL src/tests/tasks.test.ts (1.5s) Tests: 1 failed, 2 passed, 3 total # Check git status Bash: command="git status"
⚠

Be mindful: Bash commands can do anything on your system. Always review commands before approving, especially those involving rm, git push --force, or anything that touches production systems.

🔍

Glob

Search

Fast file pattern matching -- finds files by their names and paths. Think of it as a more intuitive find command. It supports glob patterns and returns matching file paths sorted by modification time.

When to use it

  • Finding all files of a certain type: **/*.py
  • Locating test files: **/*.test.tsx
  • Finding config files: **/tsconfig*.json
  • Understanding project structure before diving into code

Example

# Find all TypeScript React components Glob: pattern="src/**/*.tsx" src/App.tsx src/components/Header.tsx src/components/Sidebar.tsx src/pages/Dashboard.tsx # Find all test files Glob: pattern="**/test_*.py" tests/test_auth.py tests/test_api.py tests/test_models.py
🔎

Grep

Search

Powerful content search built on ripgrep. Searches inside files for patterns -- text, function names, API endpoints, error messages, or any regular expression. This is one of Claude's most-used tools for understanding unfamiliar codebases.

When to use it

  • Finding where a function or variable is defined or used
  • Locating all API endpoint definitions
  • Searching for TODO/FIXME comments
  • Finding configuration values or environment variables
  • Tracing imports and dependencies

Key parameters

  • pattern (required) -- regex pattern to search for
  • type -- file type filter (e.g., "py", "js", "ts")
  • output_mode -- "files_with_matches" (default), "content", or "count"
  • -i -- case-insensitive search
  • -C -- context lines around matches

Example

# Find all route definitions Grep: pattern="@app\.(get|post|put|delete)" type="py" output_mode="content" main.py:12: @app.get("/") main.py:17: @app.get("/health") main.py:22: @app.post("/users") # Find files importing a specific module Grep: pattern="from database import" output_mode="files_with_matches" backend/main.py backend/services/user_service.py
🤖

Agent

Advanced

Launches a sub-agent with its own fresh context window. The Agent tool is used for complex research tasks that require reading many files, or tasks that can be parallelized. Because each agent gets a clean context, it will not crowd your main conversation's context window.

When to use it

  • Researching how a feature works across dozens of files
  • Running multiple independent investigations in parallel
  • Tasks that require deep exploration but whose details you do not need in your main context
  • Complex refactoring analysis before making changes

Example

# Research authentication flow across the codebase Agent: prompt="Analyze how user authentication works in this project. Find the login endpoint, trace the token generation, identify where tokens are validated, and summarize the complete auth flow with file paths and line numbers." [Sub-agent reads ~30 files, follows imports, traces the flow] Returns a comprehensive summary of the auth system
💡

Why this matters: The Agent tool is your secret weapon for staying within context limits. Instead of reading 30 files yourself and filling up the context window, delegate the research to a sub-agent. You only get back the summary.

🌐

WebSearch / WebFetch

Internet

Access the web for documentation, API references, examples, and up-to-date information. WebSearch performs web searches to find relevant pages, while WebFetch retrieves the content of a specific URL.

When to use it

  • Looking up library documentation or API references
  • Finding examples for unfamiliar APIs
  • Checking for the latest version of a package
  • Researching error messages or stack traces

Example

# Search for documentation WebSearch: query="FastAPI middleware authentication tutorial" # Fetch a specific docs page WebFetch: url="https://fastapi.tiangolo.com/tutorial/middleware/"

4. Codebase Exploration Patterns

One of the most common uses for Claude Code is exploring and understanding a codebase you have never seen before. The most effective approach follows a three-phase pattern: Orient, Navigate, Read.

The "Orient, Navigate, Read" Workflow

Orient: See the big picture

Use Glob to understand the project structure. What files are there? What directories exist?

Glob: pattern="**/*.ts" # See all TypeScript files and their directory structure

Navigate: Find key code

Use Grep to find specific patterns, function definitions, or entry points.

Grep: pattern="export function" type="ts" output_mode="content" # Find all exported functions -- the public API of each module

Read: Understand specifics

Use Read to examine the files you identified as important.

Read: file_path=/src/core/engine.ts # Deep-dive into the core logic

Following the Import Chain

Another powerful pattern is starting at the entry point and following imports. This is especially useful for understanding how data flows through the application:

  1. Read the entry point (e.g., src/index.ts or main.py)
  2. Identify the key imports
  3. Read those imported modules
  4. Continue until you understand the full picture
💡

5-minute codebase understanding: Ask Claude: "What does this project do? Give me the architecture, tech stack, and main entry points." Claude will use Orient-Navigate-Read automatically and give you a comprehensive summary.

5. Making Changes and Verifying

The most productive workflow in Claude Code follows a three-step cycle: Read, Edit, Test. This ensures every change is understood, precise, and verified.

The "Read, Edit, Test" Cycle

Read the file to understand current state

Claude always reads a file before modifying it. This ensures the edit is based on the actual code, not assumptions.

Edit with precise old_string to new_string

Surgical changes. Only the lines that need to change are modified. Everything else stays exactly the same.

Bash to run tests and verify

After every change, run the test suite. If tests fail, Claude can read the output and fix the issue.

When to Use Edit vs. Write

Situation Use Edit Use Write
Fix a bug in an existing file Yes -- targeted change No
Create a brand new file No Yes -- file does not exist yet
Add an import to a file Yes -- add one line No
Rewrite a file from scratch No Yes -- complete replacement
Rename a variable Yes -- with replace_all No
Add a new function to a file Yes -- insert at specific location No

Git Integration

Claude Code has full git integration through the Bash tool. You can ask it to:

  • Check status and diffs: "Show me what changed"
  • Create commits: "Commit these changes with an appropriate message"
  • Create branches: "Create a feature branch for this change"
  • Create pull requests: "Create a PR for these changes"
  • Review code: "Review the changes in this PR"
ⓘ

Git safety: Claude Code is careful with git operations. It prefers creating new commits over amending, never force-pushes unless explicitly asked, and always shows you the diff before committing.

6. Context Window Management

Every conversation with Claude Code has a context window -- a limited amount of "memory" (roughly 200,000 tokens, with Opus capable of up to 1M) that holds everything: your messages, Claude's responses, tool call results, file contents, and command outputs.

When the context window fills up, older messages get compressed or dropped. Understanding this constraint is key to working effectively.

What Consumes Tokens?

  • Conversation turns: Each prompt + reply uses ~400 tokens on average
  • File reads: A typical 100-line file uses ~800 tokens
  • Large file reads: Files with 500+ lines use ~4,000 tokens
  • Bash output: Test results, build logs use ~1,500 tokens
  • Search results: Grep/Glob results use ~300 tokens
  • Verbose output: Full test suites or webpack builds can use ~8,000 tokens

Strategies for Staying Efficient

  1. Read only what you need. Use offset and limit for large files instead of reading the entire thing.
  2. Use Agent for research. Delegate file-heavy research to sub-agents. They get their own context window, keeping yours lean.
  3. Start new sessions for new topics. If you are switching to a completely different task, start a fresh Claude Code session.
  4. Use /compact to compress context. This command summarizes the conversation so far, freeing up space for new work.
  5. Be specific in your requests. "Fix the TypeError in user_service.py line 42" consumes fewer turns than "fix the bug" which requires Claude to search for it.

Signs You Are Running Low

  • Claude's responses get shorter and less detailed
  • Claude forgets earlier context from the conversation
  • Claude asks you to repeat information you already provided
  • The /status command shows high context usage
⚠

Pro tip: Run /status periodically to check your context usage. If you are above 70%, consider using /compact or starting a new session.

7. CLAUDE.md -- Your Project's AI Config File

CLAUDE.md is a special file that lives at the root of your project. Claude Code reads it automatically at the start of every conversation. Think of it as a "briefing document" that tells Claude about your project's conventions, rules, and context.

A good CLAUDE.md saves you from repeating the same instructions every session.

Where to Put It

Place it in your project root directory, right next to package.json or pyproject.toml. Claude Code automatically detects and reads it when you start a session in that directory.

What to Include

  • Tech stack and framework versions: "Python 3.11 + FastAPI + PostgreSQL"
  • Build and test commands: "Run tests: npm test"
  • Coding conventions: "Use functional components, prefer const over let"
  • Testing requirements: "All new functions need unit tests"
  • Important file locations: "Routes are in src/routes/"
  • Things to avoid: "Never use any types in TypeScript"

Example CLAUDE.md

CLAUDE.md # Project: Task Management API ## Tech Stack - Backend: TypeScript + Express.js - Database: PostgreSQL with Prisma ORM - Testing: Vitest with fixtures - Deployment: Docker + Kubernetes ## Commands - Run tests: `npm test` - Start dev server: `npm run dev` - Build: `npm run build` - Lint: `npm run lint` ## Coding Rules - ALWAYS use TypeScript strict mode - NEVER use `any` types -- use `unknown` and type guards - Prefer functional components with hooks - All API routes must have input validation with Zod - Use `const` over `let` wherever possible ## Important Directories - src/routes/ -- API endpoint handlers - src/services/ -- Business logic - src/models/ -- Prisma models and types - src/middleware/ -- Auth, validation, error handling - tests/ -- Test files mirror src/ structure ## Context - src/legacy/ is being deprecated. Do not add new code there. - Use `node` (not `bun`) for all commands.
💡

Start simple: You do not need a 50-line CLAUDE.md on day one. Start with your tech stack, build commands, and one or two coding rules. Add more as you discover patterns that Claude keeps getting wrong.
