Claude API Fundamentals

Master the Claude Messages API from first principles. Learn authentication, request/response anatomy, streaming, tool use (function calling), error handling, and cost optimization strategies for production workloads.

⏱ ~90 min 📚 Intermediate 🎓 Developers, Architects 🎯 4 Interactive Exercises

1. The Messages API

All interactions with Claude happen through a single endpoint: the Messages API. Whether you are building a chatbot, a data pipeline, or an autonomous agent, every request goes to the same place.

The Endpoint

HTTP POST https://api.anthropic.com/v1/messages

Authentication

Every request requires an API key. Pass it in the x-api-key header (or the equivalent anthropic-api-key header). You also need to specify the API version with the anthropic-version header.

Headers x-api-key: sk-ant-api03-... anthropic-version: 2023-06-01 content-type: application/json
⚠

Never commit your API key to source control. Use environment variables (ANTHROPIC_API_KEY) or a secrets manager. If a key is leaked, rotate it immediately in the Anthropic Console.

Request Body Anatomy

The request body is JSON with a handful of required and optional fields. Here is the minimum viable request:

JSON Request { "model": "claude-sonnet-4-6-20250514", "max_tokens": 1024, "system": "You are a helpful assistant.", "messages": [ {"role": "user", "content": "What is the capital of France?"} ] }

Response Structure

The API returns a structured JSON response that includes the model's output, metadata, and token usage:

JSON Response { "id": "msg_01XFDUDYJgAACzvnptvVoYEL", "type": "message", "role": "assistant", "content": [ {"type": "text", "text": "The capital of France is Paris."} ], "model": "claude-sonnet-4-6-20250514", "stop_reason": "end_turn", "usage": { "input_tokens": 25, "output_tokens": 15 } }

Key Parameters

Parameter Required Description
model Yes Model identifier (e.g. claude-sonnet-4-6-20250514)
max_tokens Yes Maximum tokens in the response. Set this intentionally.
messages Yes Array of message objects with role and content
system No System prompt that guides Claude's behavior
temperature No Randomness (0.0 = deterministic, 1.0 = creative). Default: 1.0
top_p No Nucleus sampling. Use temperature or top_p, not both.
tools No Array of tool definitions for function calling
stream No Set to true for streaming responses via SSE

Multi-Turn Conversations

Claude is stateless -- it does not remember previous requests. To have a multi-turn conversation, you send all previous messages (both user and assistant) in each request. Messages must alternate between user and assistant roles.

Multi-Turn Example { "model": "claude-sonnet-4-6-20250514", "max_tokens": 1024, "messages": [ {"role": "user", "content": "What is the capital of France?"}, {"role": "assistant", "content": "The capital of France is Paris."}, {"role": "user", "content": "What is its population?"} ] }

The stop_reason Field

The stop_reason field tells you why Claude stopped generating. This is critical for building reliable applications:

  • "end_turn" -- Claude finished its response naturally.
  • "max_tokens" -- Claude hit the max_tokens limit. The response is likely truncated.
  • "tool_use" -- Claude wants to call a tool. You must execute the tool and send the result back.
💡

Always check stop_reason. If it is "max_tokens", your response was cut off. If it is "tool_use", Claude is waiting for you to execute the tool and return results. Ignoring these signals leads to broken applications.

2. Using the SDKs

While you can call the Messages API with raw HTTP, the official SDKs provide typed interfaces, automatic retries, and streaming helpers that make development much smoother.

Python SDK

Install pip install anthropic
Python import anthropic # The SDK reads ANTHROPIC_API_KEY from your environment client = anthropic.Anthropic() message = client.messages.create( model="claude-sonnet-4-6-20250514", max_tokens=1024, messages=[ {"role": "user", "content": "Hello, Claude!"} ] ) print(message.content[0].text)

TypeScript / Node.js SDK

Install npm install @anthropic-ai/sdk
TypeScript import Anthropic from '@anthropic-ai/sdk'; // The SDK reads ANTHROPIC_API_KEY from your environment const client = new Anthropic(); const message = await client.messages.create({ model: "claude-sonnet-4-6-20250514", max_tokens: 1024, messages: [ { role: "user", content: "Hello, Claude!" } ] }); console.log(message.content[0].text);

Error Handling

The SDKs throw typed exceptions that make error handling straightforward:

Python import anthropic client = anthropic.Anthropic() try: message = client.messages.create( model="claude-sonnet-4-6-20250514", max_tokens=1024, messages=[{"role": "user", "content": "Hello!"}] ) except anthropic.AuthenticationError: print("Invalid API key") except anthropic.RateLimitError: print("Rate limited -- back off and retry") except anthropic.APIError as e: print(f"API error: {e.status_code}")
ⓘ

Environment variable: Both SDKs look for ANTHROPIC_API_KEY automatically. You can also pass the key explicitly: anthropic.Anthropic(api_key="sk-...") in Python or new Anthropic({ apiKey: "sk-..." }) in TypeScript.

3. Streaming

Without streaming, your application waits for the entire response before showing anything to the user. With streaming, tokens appear in real-time as Claude generates them, creating a dramatically better user experience.

Why Streaming Matters

A 500-token response from Claude Sonnet might take 3-5 seconds to generate. Without streaming, the user stares at a blank screen for that entire duration. With streaming, they see the first word within ~200 milliseconds.

Server-Sent Events (SSE)

Streaming uses the SSE protocol. When you set "stream": true in your request, the API sends back a stream of events rather than a single JSON response.

Event Types

Event When Contains
message_start First event Message ID, model, role
content_block_start New content block begins Block type (text or tool_use)
content_block_delta Each chunk of text Incremental text addition (1-3 words)
content_block_stop Content block finishes Block index
message_delta Near end Stop reason, final usage stats
message_stop Last event Stream complete signal

Python Streaming

Python with client.messages.stream( model="claude-sonnet-4-6-20250514", max_tokens=1024, messages=[{"role": "user", "content": "Write a poem about APIs."}] ) as stream: for text in stream.text_stream: print(text, end="", flush=True)

TypeScript Streaming

TypeScript const stream = client.messages.stream({ model: "claude-sonnet-4-6-20250514", max_tokens: 1024, messages: [{ role: "user", content: "Write a poem about APIs." }] }); for await (const event of stream) { if (event.type === "content_block_delta") { process.stdout.write(event.delta.text); } }
💡

Always stream in user-facing applications. The perceived latency improvement is dramatic. Even when processing the stream programmatically, streaming lets you start work on partial results sooner.

4. Tool Use (Function Calling)


      

        Tool use is the most powerful capability of the Claude API.
        It allows Claude to call functions that you define and control. This is the foundation
        of agentic applications -- Claude can search databases, call external APIs, run calculations,
        and take real-world actions through your code.
      


      
How It Works


      

        You define tools as JSON Schema descriptions in your API request. When Claude determines it
        needs to use a tool, it responds with a structured tool_use content block instead
        of plain text. Your code then executes the function and sends the result back.
      


      
Defining Tools


      

        Each tool needs a name, a description (which Claude reads to decide
        when to use it), and an input_schema that defines the parameters:
      


      

        Tool Definition
        
{
  "name": "get_weather",
  "description": "Get current weather for a location. Returns temperature, conditions, and humidity.",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "City and country, e.g. 'Paris, France'"
      },
      "units": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "Temperature units"
      }
    },
    "required": ["location"]
  }
}


      

        💡
        

          
Write great descriptions. Claude uses the tool description to decide
          when to call it. A vague description like "Gets data" will lead to poor tool selection.
          Be specific about what the tool does and what it returns.

        

      


      
The Tool Use Loop (The Agentic Pattern)


      

        This is the core agentic loop that powers sophisticated AI applications. Understand this pattern
        deeply:
      


      

        

          

            
Send message with tool definitions

            
Include the tools array in your API request along with the user message.
            Claude now knows what tools are available.

          

        


        

          

            
Claude responds with tool_use

            
If Claude decides a tool is needed, it responds with stop_reason: "tool_use"
            and includes a tool_use content block with the tool name and input arguments.

          

        


        

          

            
Your code executes the function

            
You extract the tool name and input from Claude's response, call the actual function in your
            code, and get the result.

          

        


        

          

            
Send tool_result back

            
You add Claude's response to the messages array, then add a user message
            containing a tool_result content block with the function output.

          

        


        

          

            
Claude responds with final text

            
Claude reads the tool result and responds with natural language. It might also call
            another tool -- in which case you repeat steps 2-4.

          

        

      


      
tool_choice Parameter


      
    
        
  • "auto" (default) -- Claude decides whether to use a tool or respond with text.
    • 
        
  • "any" -- Forces Claude to call at least one tool. It will not respond with text only.
    • 
        
  • {"type": "tool", "name": "get_weather"} -- Forces Claude to call a specific tool.
    • 
      


      
Multiple Tool Calls


      

        Claude can call multiple tools in a single response. For example, if a user asks "What's the weather
        in Paris and London?", Claude might respond with two tool_use blocks. You need to execute
        both and send both results back.
      


      
Error Handling in Tool Results


      

        If a tool execution fails, send the error back as a tool_result with
        "is_error": true. Claude will gracefully handle the error and either retry
        or explain the issue to the user.
      


      

        Error Result
        
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01A...",
      "is_error": true,
      "content": "Error: Location 'Narnia' not found in weather database."
    }
  ]
}

5. Error Handling & Retries


      

        Production applications must handle API errors gracefully. The Claude API uses standard HTTP
        status codes, and the SDKs provide typed exceptions for each error category.
      


      
Error Types


      

        

          
            

              Code
              Meaning
              Action
            

          
          
            

              400
              Invalid request (bad parameters)
              Fix the request. Do not retry.
            

            

              401
              Authentication failed
              Check your API key. Do not retry.
            

            

              429
              Rate limited
              Exponential backoff and retry.
            

            

              500
              Internal server error
              Retry with backoff. Usually transient.
            

            

              529
              API overloaded
              Retry with longer backoff. The system is under heavy load.
            

          
        

      


      
Retry Strategy with Exponential Backoff


      

        Python
        
import time
import anthropic

client = anthropic.Anthropic()

def call_with_retry(messages, max_retries=3):
    for attempt in range(max_retries):
        try:
            return client.messages.create(
                model="claude-sonnet-4-6-20250514",
                max_tokens=1024,
                messages=messages
            )
        except anthropic.RateLimitError:
            wait = 2 ** attempt  # 1s, 2s, 4s
            print(f"Rate limited. Retrying in {wait}s...")
            time.sleep(wait)
        except anthropic.APIStatusError as e:
            if e.status_code == 529:
                wait = 2 ** (attempt + 1)  # Longer backoff for overload
                time.sleep(wait)
            else:
                raise
    raise Exception("Max retries exceeded")


      
Rate Limits


      
    
        
  • Requests per minute (RPM) -- How many API calls you can make per minute.
    • 
        
  • Tokens per minute (TPM) -- Combined input and output tokens per minute.
    • 
        
  • Tokens per day (TPD) -- Total tokens across a 24-hour window.
    • 
      


      

        Rate limits vary by model and usage tier. Check the x-ratelimit-* response headers
        to see your current limits and remaining quota.
      


      

        ⓘ
        

          
SDK built-in retries: The official SDKs include automatic retry logic
          for 429 and 5xx errors. You can configure the retry count:
          anthropic.Anthropic(max_retries=5). For most applications, the built-in retries
          are sufficient.

        

      

    


    
    

      
6. Cost Optimization


      

        API costs are directly proportional to token usage. Understanding the pricing model and applying
        the right optimization strategies can reduce your costs by 80% or more without sacrificing quality.
      


      
Model Selection by Task


      

        Not every task requires the most powerful model. Match the model to the task complexity:
      


      
    
        
  • Haiku 4.5 -- Classification, extraction, formatting, simple Q&A. Fast and cheap.
    • 
        
  • Sonnet 4.6 -- Most development tasks, code generation, analysis, writing. Best balance.
    • 
        
  • Opus 4.6 -- Complex reasoning, novel problem-solving, research. Use when quality is paramount.
    •

Prompt Caching


      

        If you send the same system prompt or tool definitions across many requests, enable
        prompt caching. Cached input tokens are billed at 90% less
        than regular input tokens. The first request pays full price; subsequent requests within the
        cache TTL get the discount.
      


      
Message Batches API


      

        For non-urgent workloads -- bulk classification, report generation, data processing -- use the
        Message Batches API. You submit up to 100,000 requests in a batch,
        and the API processes them within 24 hours at a 50% discount.
      


      
Token Budgeting Best Practices


      
    
        
  • Set max_tokens appropriately. If you expect a 200-token response, don't set it to 4096. While you only pay for tokens actually generated, lower limits prevent runaway costs from unexpectedly long outputs.
    • 
        
  • Keep system prompts concise. Every token in the system prompt is billed on every request.
    • 
        
  • Trim conversation history. In multi-turn chats, summarize or truncate old messages rather than sending the entire history.
    • 
        
  • Monitor usage by feature. Tag requests with metadata and track costs per feature, per user, or per model.
    • 
      


      

        ⚠
        

          
Output tokens cost 5x more than input tokens. This is true across all models.
          Techniques that reduce output length (structured JSON responses, concise formatting instructions)
          have outsized cost impact.
