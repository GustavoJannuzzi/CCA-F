Claude Code Advanced: Subagents, Plugins, and Hooks

Master the advanced patterns that unlock the full power of Claude Code -- spawning independent subagents for parallel research, building reusable plugins for your team, and orchestrating automated workflows with lifecycle hooks.

⏱ ~90 min 📚 Advanced 🎯 4 Interactive Exercises 👥 Developers & Architects

1. The Subagent Pattern

A subagent is an independent Claude instance spawned by the main (parent) agent. When the parent encounters a complex research task or needs to isolate verbose work, it delegates to a subagent that gets its own fresh context window. The parent continues only with the concise summary the subagent returns -- not the full transcript of everything the subagent did.

This is one of the most powerful patterns in Claude Code because it lets you keep your main context lean while still doing deep, thorough analysis across large codebases.

How Subagents Work

The subagent lifecycle follows a clear sequence:

  1. Parent spawns subagent -- the parent provides a detailed prompt describing the task, but the subagent receives no parent history.
  2. Subagent gets a fresh context window -- it starts clean, with access to all standard tools (Read, Grep, Glob, Bash, etc.) but none of the parent's conversation.
  3. Subagent works independently -- it reads files, searches code, runs commands, and reasons through the problem. This may consume thousands of tokens.
  4. Subagent returns a summary -- when complete, it produces a concise result that flows back to the parent.
  5. Parent continues with only the summary -- all the verbose tool calls, file contents, and intermediate reasoning from the subagent stay in the subagent's context, not the parent's.
💡

Key insight: Think of subagents as "disposable researchers." You send them on a mission, they come back with findings, and then their context is discarded. This is exactly what you want for tasks that would otherwise fill your main context with noise.

When to Use Subagents

Subagents shine in specific scenarios where context isolation provides real value:

Exploring a Large Codebase

When you need to understand how authentication works across 50+ files, a subagent can read, search, and map the entire auth flow without polluting your main context with every file it reads.

Parallel Tasks

Launch 3 subagents simultaneously -- one to analyze the database layer, one to map API endpoints, and one to review test coverage. Each works independently, and you get three focused summaries instead of a massive serial exploration.

Specialized Analysis

Architecture reviews, security audits, and test generation are all perfect subagent tasks. Give the subagent a focused mandate ("Review all SQL queries for injection vulnerabilities") and get back a structured report.

Verbose Operations

Build output, large file analysis, log parsing -- anything that generates huge amounts of text that you need processed but not stored in your main conversation.

Anti-Patterns: When NOT to Use Subagents

⚠

Avoid these common mistakes:

  • Simple file reads -- If you just need to read one or two files, using a subagent is overkill. The overhead of spawning a new context is not worth it for trivial tasks.
  • Insufficient context in the prompt -- The subagent has no access to your conversation history. If you send it on a mission without enough detail ("go look at the auth code"), it will waste time figuring out what you already know.
  • Too many parallel agents -- While 2-3 parallel subagents can be very effective, launching 10+ leads to diminishing returns. Each agent needs to independently orient itself, and the coordination overhead grows.
  • Tasks requiring conversation continuity -- If the task requires back-and-forth dialogue or iterative refinement with the user, a subagent cannot do that. It runs to completion and returns.

Example: Spawning a Subagent

When you ask Claude Code to research something complex, it may use the Agent tool like this:

Agent Tool Call // Parent's internal tool call: Agent("Research the authentication flow in this codebase. Map out every file involved in the auth process: - Where are credentials validated? - How are sessions managed? - What middleware handles token verification? - Are there any security concerns? Return a structured summary with file paths and findings.") // Subagent works independently -- reads 30+ files, greps for patterns, // traces imports -- all in its own context window. // Parent receives only this summary: Authentication Flow Summary: - Entry point: src/middleware/auth.ts (verifyToken middleware) - Token validation: src/services/jwt.ts (RS256 with key rotation) - Session store: src/services/session.ts (Redis-backed, 24h TTL) - User lookup: src/models/user.ts -> src/db/queries/auth.sql - 3 security concerns identified: 1. No rate limiting on /api/login endpoint 2. Refresh tokens stored in localStorage (XSS risk) 3. Missing CSRF protection on session cookie

The parent now has a clean, actionable summary. It never saw the 30+ file reads, the grep searches, or the intermediate reasoning. That is the power of subagents.

2. Plugin Architecture

Plugins are packaged extensions that add commands, agents, skills, and hooks to Claude Code. They let you encapsulate domain-specific knowledge and workflows into shareable, versioned packages that your entire team can use.

A plugin is just a directory with a specific structure. There is no build step, no compilation, no package manager required. You create the files, point Claude Code at the directory, and it picks up everything automatically.

Plugin Directory Structure

Directory Structure my-plugin/ .claude-plugin/ plugin.json # Plugin metadata and configuration commands/ deploy.md # Slash command: /deploy rollback.md # Slash command: /rollback agents/ code-reviewer.md # Custom agent definition skills/ SKILL.md # Domain knowledge and patterns hooks/ pre-commit-check.sh # Lifecycle hook script

plugin.json Anatomy

The plugin.json file is the heart of your plugin. It declares metadata, lists all components, and configures how the plugin integrates with Claude Code.

plugin.json { "name": "deployment-toolkit", "version": "1.0.0", "description": "Production deployment commands and safety checks", "author": "Platform Team", "commands": [ { "name": "deploy", "description": "Deploy to staging or production", "file": "commands/deploy.md" }, { "name": "rollback", "description": "Rollback to a previous version", "file": "commands/rollback.md" } ], "agents": [ { "name": "deploy-validator", "description": "Validates deployment readiness", "file": "agents/deploy-validator.md" } ] }

How to Install Plugins

Plugins can be loaded in several ways:

  • Local development: Use the --plugin-dir flag when launching Claude Code:
    claude --plugin-dir ./my-plugin
  • Persistent installation: Copy the plugin directory to ~/.claude/plugins/
  • Project-level: Add a .claude/plugins/ directory in your repository root
🛈

Pro tip: During development, use --plugin-dir so you can iterate quickly. Once stable, commit the plugin to your repo's .claude/plugins/ directory so every team member gets it automatically.

3. Building Your First Plugin

Let us walk through building a plugin from scratch. We will create a code-quality plugin that adds a /lint command and a code review agent.

Create the directory structure

Set up the standard plugin scaffold with all component directories:

Terminal mkdir -p code-quality/.claude-plugin mkdir -p code-quality/commands mkdir -p code-quality/agents mkdir -p code-quality/skills

Write plugin.json with metadata

Define your plugin's identity and list its components:

code-quality/.claude-plugin/plugin.json { "name": "code-quality", "version": "1.0.0", "description": "Code quality tools: linting, review, and best practices", "commands": [ { "name": "lint", "description": "Run linters and report issues", "file": "commands/lint.md" } ], "agents": [ { "name": "code-reviewer", "description": "Reviews code for quality and best practices", "file": "agents/code-reviewer.md" } ] }

Create a command (markdown file with instructions)

Commands are markdown files that instruct Claude on what to do when the user types the slash command:

code-quality/commands/lint.md # /lint Command When the user runs /lint, perform these steps: 1. Detect the project type by checking for: - package.json (Node.js) -> run `npx eslint .` - pyproject.toml or setup.py (Python) -> run `ruff check .` - Cargo.toml (Rust) -> run `cargo clippy` - go.mod (Go) -> run `golangci-lint run` 2. Parse the linter output and categorize issues: - **Errors**: Must be fixed before merge - **Warnings**: Should be addressed - **Style**: Nice to have 3. Present a summary table with file, line, severity, and description for each issue. 4. Offer to auto-fix what can be auto-fixed.

Create a skill (SKILL.md with domain knowledge)

Skills provide persistent domain knowledge that Claude can reference:

code-quality/skills/SKILL.md --- trigger: "when reviewing code quality or discussing best practices" context: "fork" --- # Code Quality Best Practices ## Principles - Functions should do one thing and do it well - Prefer explicit over implicit - Names should reveal intent - Comments explain WHY, not WHAT - Tests should be independent and deterministic ## Review Checklist - [ ] No unused imports or variables - [ ] Error handling for all async operations - [ ] Input validation on all public APIs - [ ] No hardcoded secrets or credentials - [ ] Consistent naming conventions - [ ] No console.log / print statements in production code

Test locally with --plugin-dir

Point Claude Code at your plugin directory to test it:

Terminal # Launch Claude Code with your plugin loaded claude --plugin-dir ./code-quality # Now you can use the command: > /lint # The code-reviewer agent is also available

Share with your team

Once your plugin works, share it by committing to your repo:

Terminal # Move to your project's plugin directory cp -r code-quality/ .claude/plugins/code-quality/ # Commit so all team members get it git add .claude/plugins/code-quality/ git commit -m "Add code-quality plugin for linting and review"

4. Hook Lifecycle

Hooks are shell commands that run at specific lifecycle points during a Claude Code session. They let you intercept, validate, or modify Claude's behavior without changing the AI's instructions.

Think of hooks like git hooks but for AI interactions. They are your safety net and your automation layer.

Hook Types

Hook Type When It Runs Use Case
PreToolUse Before Claude uses any tool Block dangerous operations, validate parameters
PostToolUse After a tool completes Validate output, transform results, log actions
Stop When Claude wants to end the session Check if work is actually complete, force continuation
Notification On specific events Monitoring, Slack alerts, audit logging

Hook Configuration

Hooks are configured in your settings.json file. Each hook specifies a matcher (which tool or event to intercept) and a command to run:

settings.json { "hooks": { "PreToolUse": [ { "matcher": "Bash", "command": "./hooks/check-safety.sh $TOOL_INPUT" }, { "matcher": "Write", "command": "./hooks/check-write-target.sh $TOOL_INPUT" } ], "PostToolUse": [ { "matcher": "Edit", "command": "./hooks/validate-format.sh $TOOL_INPUT" } ], "Stop": [ { "command": "./hooks/check-completion.sh" } ] } }

Hook Exit Codes

The exit code of your hook script determines what happens next:

  • Exit code 0 -- Allow the action. The tool call proceeds (PreToolUse) or the session ends (Stop).
  • Non-zero exit code -- Block the action. The tool call is prevented (PreToolUse) or Claude is forced to continue (Stop).

Real-World Hook Examples

PreToolUse: Prevent Deletion of Production Files

Block any Bash command that attempts to rm files in protected directories:

hooks/check-safety.sh #!/bin/bash # Block dangerous delete operations TOOL_INPUT="$1" if echo "$TOOL_INPUT" | grep -qE 'rm\s+(-rf?\s+)?(/prod|/production|/live)'; then echo "BLOCKED: Cannot delete production files" >&2 exit 1 fi exit 0

PostToolUse: Validate Code Formatting After Edits

After every Edit tool call, run the formatter to check if the result is properly formatted:

hooks/validate-format.sh #!/bin/bash # Check formatting after each edit FILE_PATH="$1" if [[ "$FILE_PATH" == *.py ]]; then ruff format --check "$FILE_PATH" 2>/dev/null elif [[ "$FILE_PATH" == *.ts ]] || [[ "$FILE_PATH" == *.tsx ]]; then npx prettier --check "$FILE_PATH" 2>/dev/null fi

Stop: Check If All Tests Pass

Prevent the session from ending until the test suite is green:

hooks/check-tests.sh #!/bin/bash # Don't let Claude stop until tests pass npm test --silent 2>/dev/null if [ $? -ne 0 ]; then echo "Tests are still failing. Keep working." >&2 exit 1 fi exit 0

5. The Stop Hook Pattern

The Stop Hook is arguably the most powerful pattern in Claude Code's hook system. It transforms Claude from a single-turn tool into a multi-phase workflow engine.

How It Works

The pattern is elegantly simple:

  1. Claude finishes a phase of work and signals that it wants to stop.
  2. The Stop hook reads a state file (e.g., .jumpstarter-state.json) to check the workflow progress.
  3. If more phases remain, the hook exits non-zero -- Claude continues working on the next phase.
  4. If all phases are complete, the hook exits 0 -- Claude stops normally.

State File Anatomy

The state file tracks where Claude is in a multi-phase workflow:

.jumpstarter-state.json { "current_phase": 3, "total_phases": 7, "completed": ["scaffold", "backend", "frontend"], "remaining": ["api", "tests", "qa", "deploy"], "max_iterations": 50, "iteration_count": 12 }

The Stop Hook Script

hooks/check-completion.sh #!/bin/bash # Stop hook: check if all phases are complete STATE_FILE=".jumpstarter-state.json" # If no state file, allow stop if [ ! -f "$STATE_FILE" ]; then exit 0 fi # Read state values CURRENT=$(python3 -c "import json; print(json.load(open('$STATE_FILE'))['current_phase'])") TOTAL=$(python3 -c "import json; print(json.load(open('$STATE_FILE'))['total_phases'])") ITERATION=$(python3 -c "import json; print(json.load(open('$STATE_FILE'))['iteration_count'])") MAX_ITER=$(python3 -c "import json; print(json.load(open('$STATE_FILE'))['max_iterations'])") # Safety: always stop if max iterations reached if [ "$ITERATION" -ge "$MAX_ITER" ]; then echo "Max iterations reached ($ITERATION/$MAX_ITER). Allowing stop." exit 0 fi # If more phases remain, keep going if [ "$CURRENT" -lt "$TOTAL" ]; then echo "Phase $CURRENT of $TOTAL complete. Continue to next phase." >&2 exit 1 fi # All phases complete echo "All $TOTAL phases complete!" exit 0
⚠

Critical safety measure: Always include a max_iterations check. Without it, a bug in your phase logic could create an infinite loop where Claude never stops. The max_iterations guard ensures the session will eventually end regardless of the state.

Use Cases for the Stop Hook Pattern

  • Multi-phase code generation -- Scaffold, then implement backend, then frontend, then tests, then QA, then docs
  • Batch processing -- Process 100 files one by one, checking progress in the state file
  • Iterative refinement -- Keep refining code until quality metrics pass or max iterations are reached
  • Pipeline execution -- Run a sequence of CI-like stages, each building on the previous one

6. Skills Packaging and Sharing

Skills are reusable knowledge files that give Claude specialized expertise. Unlike commands (which define actions) or agents (which define personalities), skills define what Claude knows about a specific domain.

SKILL.md Best Practices

A well-written skill file has these key characteristics:

Clear Trigger Description

The frontmatter trigger field tells Claude when to apply this skill. Be specific:

  • Good: "when writing database migration scripts for PostgreSQL"
  • Bad: "when working with databases" (too broad)

Specific Instructions

The skill body should contain concrete patterns, templates, and rules -- not vague guidelines. Include:

  • Code templates with placeholders
  • Naming conventions specific to your team
  • Error handling patterns you expect
  • Testing requirements

Context Management

Use context: fork in the frontmatter to run the skill in an isolated subagent context. This is essential for skills that might consume lots of tokens during execution -- it keeps your main conversation clean.

Operations Directory

For skills that need supporting files (SQL templates, config examples, reference docs), use an operations/ directory alongside the SKILL.md:

Directory Structure skills/ SKILL.md # Main skill definition operations/ migration-template.sql # SQL template for migrations rollback-template.sql # SQL template for rollbacks naming-conventions.md # Reference doc for naming rules

Example: Database Migration Skill

skills/SKILL.md --- trigger: "when creating database migrations or modifying schema" context: "fork" --- # Database Migration Skill ## Rules 1. Every migration MUST have a corresponding rollback 2. Use timestamped filenames: YYYYMMDD_HHMMSS_description.sql 3. Never use DROP TABLE in production -- use soft deletes 4. Always add NOT NULL constraints with DEFAULT values 5. Create indexes for any column used in WHERE or JOIN ## Migration Template See operations/migration-template.sql for the standard format. ## Testing Requirements - Run migration forward AND backward before committing - Verify with: `psql -f migration.sql && psql -f rollback.sql` - Check for data loss with: SELECT COUNT(*) before and after

Sharing Skills

Skills can be shared through several channels:

  • Git repository -- Commit skills to a shared repo. Team members clone and point their config at the skills directory.
  • Plugin bundling -- Package skills inside a plugin for one-click installation.
  • npm/pip packages -- For larger organizations, publish skills as packages with versioning and dependency management.
💡

Team strategy: Start with a shared git repo of skills. As patterns stabilize, graduate them into versioned plugins. This gives you the fastest iteration cycle while building toward a mature plugin ecosystem.
