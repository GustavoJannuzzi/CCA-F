1. Domain Overview

Domain 4 of the Claude AI certification exam focuses on Prompt Engineering & Structured Output and carries a 20% weight -- making it the second-highest weighted domain on the exam. This domain tests your ability to design prompts that produce reliable, structured, and validated outputs in production systems.

Exam Weight: Domain 4
20%
Explicit Criteria Few-Shot Prompting Structured Output (tool_use) Validation-Retry Loops Batch Processing Multi-Pass Review

Why This Domain Matters

In production AI systems, the difference between a prototype and a reliable product often comes down to how well you engineer prompts and validate outputs. A clever prompt that works 90% of the time is useless if the other 10% produces malformed JSON, hallucinated data, or confident-sounding errors. This domain tests your ability to build systems that are reliably correct, not just usually correct.

The Six Task Areas

  • Explicit Criteria: Converting vague requirements into concrete, categorical evaluation criteria that Claude can apply consistently. The exam tests whether you understand why explicit criteria outperform confidence-score filtering.
  • Few-Shot Prompting: Using 2-4 targeted examples to calibrate Claude's behavior for ambiguous classification and extraction tasks. The exam focuses on example selection strategy, not just the mechanical placement of examples.
  • Structured Output with tool_use: Using the tool_use mechanism to guarantee schema-compliant JSON output. The exam tests tool_choice modes and schema design decisions (required vs. optional, enum extensibility).
  • Validation-Retry Loops: Building loops that detect output errors, provide specific error feedback, and know when retries are futile. The exam focuses on effective error messages and recognizing data-absent situations.
  • Batch Processing: Using the Message Batches API for cost-efficient bulk processing. The exam tests your understanding of the 50% cost savings, 24-hour window, limitations (no multi-turn tool calling), and custom_id correlation.
  • Multi-Pass Review: Architectures where separate Claude instances review each other's work. The exam focuses on why same-session self-review is unreliable and how to design independent review passes.
ⓘ

Exam tip: This domain heavily rewards understanding of why techniques work, not just how to use them. Expect scenario-based questions where you must choose between multiple valid-looking approaches and identify the one that produces the most reliable results.

2. Explicit Criteria for the Exam

One of the most common mistakes in production AI systems is relying on vague instructions like "be conservative" or "flag important issues." These instructions feel intuitive to humans but give Claude no concrete decision boundary. The exam tests whether you can convert vague requirements into explicit categorical criteria that produce consistent results.

Converting Vague to Explicit

Consider a code review system where the requirement is "be conservative about flagging issues." This is ambiguous. Conservative could mean "only flag critical bugs" or "flag everything but with low confidence scores." The exam expects you to understand the correct approach:

Vague Criteria (What Not to Do)

Prompt (Vague) "Review this code and flag issues. Be conservative -- only flag things you are very confident about. Rate your confidence 1-10 for each finding."

This approach fails because Claude's confidence scores are not calibrated. A "confidence: 8" from one prompt might correspond to "confidence: 5" from a slightly reworded prompt. Filtering by self-reported confidence is unreliable.

Explicit Criteria (The Correct Approach)

Prompt (Explicit) "Review this code. Flag ONLY findings in these categories: 1. NULL_DEREF: Variable used without null check after a function that can return null (per its type signature or documentation) 2. RESOURCE_LEAK: File handle, connection, or lock acquired but not released in all code paths (including exceptions) 3. SQL_INJECTION: String concatenation in SQL query construction without parameterized queries 4. TYPE_MISMATCH: Function called with argument type that does not match the declared parameter type Do NOT flag: style issues, naming conventions, missing comments, potential performance improvements, or deprecated API usage."

This approach works because each category has a concrete, observable condition that either exists in the code or does not. There is no subjective judgment involved.

Impact of False Positive Rates on Trust

The exam tests your understanding of a critical production dynamic: false positives erode user trust faster than false negatives. If a code review system flags 50 issues and 40 of them are wrong, developers will stop reading the reports entirely -- even if the remaining 10 findings are genuinely critical bugs.

  • High false positive rate → Users stop trusting the system → Genuine findings are ignored → The system provides negative value
  • Temporarily disabling high-FP categories is often better than keeping them enabled. If "POTENTIAL_NULL_DEREF" has a 70% false positive rate, disable it until you can improve the detection criteria.
  • Explicit criteria over confidence filtering: Instead of "flag everything and filter by confidence > 7," define categories where the false positive rate is naturally low because the criteria are unambiguous.

Defining Severity with Concrete Code Examples

When you need Claude to assess severity, provide concrete examples rather than abstract definitions:

Prompt (Severity Definitions) "Severity definitions: CRITICAL: Will cause data loss, security breach, or system crash in production under normal usage. Example: writing user password to application log in plaintext. HIGH: Will cause incorrect behavior that users will notice. Example: off-by-one error in pagination causing items to be skipped when browsing results. MEDIUM: Could cause issues under specific conditions that are plausible but not guaranteed. Example: race condition in concurrent cache update that requires exact timing. Do NOT use LOW severity -- if it is not at least MEDIUM, do not include it in your findings."
⚠

Exam focus: The exam will present scenarios where you must choose between confidence-based filtering and explicit criteria. The answer is always explicit criteria. Confidence scores are a property of the model's internal state, not of the actual finding quality.

3. Few-Shot Engineering

Few-shot prompting gives Claude 2-4 examples of the desired input-output behavior before presenting the actual task. The exam focuses not on the mechanics of few-shot prompting (everyone knows how to include examples) but on the strategy of example selection -- which examples to include, why, and how many.

The 2-4 Rule: Targeted Examples for Ambiguous Scenarios

Research and practice show that 2-4 well-chosen examples outperform larger sets. Why? Because the purpose of few-shot examples is not to teach Claude the task (it already knows most tasks) but to calibrate its behavior for ambiguous edge cases. More examples beyond 4 typically add redundancy without improving calibration.

What makes a good few-shot example set?

  • Diversity of categories: Each example should demonstrate a different category or outcome. If you have 3 examples and all show the "positive" class, you have taught Claude nothing about how to handle the "negative" class.
  • Ambiguous edge cases: At least one example should be a genuinely ambiguous case where the correct answer is not obvious. Show Claude how you want it to reason through borderline situations.
  • Reasoning shown: Examples should include the reasoning process, not just the final answer. This calibrates Claude's thinking, not just its output format.
  • Format consistency: All examples must follow the exact same output format. If your output schema has 5 fields, every example must populate all 5 fields, even if some are null.

Showing Reasoning for Choices

Few-Shot with Reasoning // Example 1: Clear positive case Input: "The patient presents with chest pain radiating to left arm, diaphoresis, and ST elevation in leads II, III, aVF." Reasoning: ST elevation in contiguous inferior leads with classic symptoms (radiating chest pain, diaphoresis) strongly indicates acute inferior STEMI. This is unambiguous. Classification: EMERGENT Confidence_rationale: All three criteria met -- symptoms, ECG, distribution. // Example 2: Ambiguous case (critical for calibration) Input: "Patient reports intermittent chest tightness for 2 weeks, worse with exertion, ECG shows nonspecific ST changes." Reasoning: Nonspecific ST changes could indicate ischemia or could be benign (medication effects, electrolyte imbalance). Exertional component is concerning but 2-week history argues against acute event. This does NOT meet criteria for EMERGENT because ECG changes are nonspecific, not diagnostic. Classification: URGENT Confidence_rationale: Exertional symptoms warrant urgent evaluation but nonspecific ECG prevents EMERGENT classification. // Example 3: Clear negative case Input: "Patient requests refill of atorvastatin 40mg, no new symptoms, last lipid panel 3 months ago within target." Reasoning: Routine medication management with no acute symptoms. Classification: ROUTINE Confidence_rationale: No symptoms, no abnormal findings, scheduled care.

Distinguishing Acceptable Patterns from Genuine Issues

In classification tasks, one of the hardest challenges is teaching Claude to distinguish between patterns that look like issues but are actually acceptable, and genuine problems. This is especially important in code review, document analysis, and compliance checking.

Example: Code Review Pattern Calibration

Include an example where a pattern that often indicates a bug is actually correct:

Input: "try { result = api.fetch(url); } catch (e) { /* ignore */ }" Reasoning: Empty catch blocks are usually a code smell, but this is in a retry loop wrapper where the outer function handles the final failure. The empty catch here is intentional -- it allows the loop to continue to the next retry attempt. Classification: NOT_AN_ISSUE Note: The surrounding context (retry loop) makes this acceptable.

Handling Varied Document Structures

When extracting data from documents (invoices, contracts, medical records), include examples that show different document formats and layouts. This teaches Claude that the meaning matters, not the specific formatting:

  • One example with a clean, structured invoice with labeled fields
  • One example with a narrative-style invoice where amounts are embedded in paragraphs
  • One example with missing optional fields (showing null/absent handling)

Reducing Hallucination in Extraction Tasks

A key exam topic: few-shot examples can reduce hallucination by explicitly showing Claude that it is acceptable to output null or "not found" when information is absent from the source. Without such an example, Claude may infer or fabricate values to fill required fields.

💡

Key insight: Include at least one few-shot example where a field is legitimately absent from the source and the correct output is null. This gives Claude explicit permission to say "I don't know" rather than fabricating an answer.

4. Structured Output & Validation

The tool_use mechanism is Claude's primary method for producing guaranteed schema-compliant structured output. Unlike asking Claude to "return JSON," which can produce syntax errors, missing fields, or extra commentary, tool_use forces the output through a JSON Schema validator at the API level.

How tool_use Guarantees Schema Compliance

When you define a tool with an input_schema, Claude's output for that tool call is validated against the schema before being returned. This means:

  • No JSON syntax errors: The output is always valid JSON
  • Required fields are present: Every field marked as required will exist
  • Types are correct: Strings are strings, numbers are numbers, enums match
  • No extra commentary: The output contains only the schema-defined fields, not freeform text mixed with JSON

tool_choice Settings

The tool_choice parameter controls whether and which tool Claude must call:

Setting Behavior Use Case
"auto" Claude decides whether to use a tool or respond with text When tool use is optional and Claude may legitimately respond without it
"any" Claude must call one of the available tools but chooses which one When you have multiple output schemas and Claude picks the appropriate one
{"type": "tool", "name": "extract_invoice"} Claude must call this specific tool When you want guaranteed structured output in a specific schema
⚠

Exam alert: The exam will test the difference between "auto" and "any". With "auto", Claude can choose to not use a tool at all and respond with plain text. With "any", Claude must call a tool but gets to choose which one. Use a specific tool name when you need guaranteed structured output.

Required vs. Optional Fields (Nullable for Missing Data)

Schema design is a critical exam topic. The key distinction:

  • Required fields must always be present in the output. If a field is required but the data might not exist in the source, make it nullable: "type": ["string", "null"]. This forces Claude to explicitly output null rather than omitting the field or hallucinating a value.
  • Optional fields can be omitted entirely. Use these for fields that only apply to certain document types (e.g., a "tax_id" field that only applies to business invoices, not personal ones).
JSON Schema Design { "name": "extract_invoice", "description": "Extract structured data from an invoice document", "input_schema": { "type": "object", "required": ["vendor_name", "invoice_date", "total_amount", "line_items"], "properties": { "vendor_name": { "type": "string", "description": "Company name of the vendor" }, "invoice_date": { "type": ["string", "null"], "description": "Invoice date in ISO 8601 format, null if not found" }, "total_amount": { "type": "number", "description": "Total invoice amount in the document currency" }, "currency": { "type": "string", "enum": ["USD", "EUR", "GBP", "other"], "description": "Currency code; use 'other' for unlisted currencies" }, "currency_detail": { "type": ["string", "null"], "description": "Actual currency name when 'other' is selected, null otherwise" }, "line_items": { "type": "array", "items": { "type": "object", "required": ["description", "amount"], "properties": { "description": { "type": "string" }, "quantity": { "type": ["number", "null"] }, "unit_price": { "type": ["number", "null"] }, "amount": { "type": "number" } } } } } } }

Enum Values with "other" + Detail for Extensibility

A best practice that the exam tests: when using enum values, always include an "other" option paired with a detail field. This prevents Claude from forcing data into the wrong category when the real value does not match any predefined enum option.

Validation-Retry Loops

Even with schema validation, the semantic content of the output may be wrong. A validation-retry loop catches these issues:

  1. Call Claude with the initial prompt
  2. Validate the output against business rules (not just schema)
  3. If validation fails, append the specific error to the conversation and retry
  4. Limit retries (typically 2-3 attempts)

Effective Error Feedback (Not Just "Try Again")

Good Retry Message "Your extraction had the following errors: 1. total_amount (450.00) does not equal sum of line_item amounts (150.00 + 200.00 = 350.00). Re-read the invoice and verify. 2. invoice_date '2024-13-01' is not a valid date (month 13). Check the date format in the source document. Please re-extract with corrections."

Ineffective Retry Message (Common Mistake)

Bad Retry Message "The output was incorrect. Please try again and be more careful."

This gives Claude no information about what was wrong. It will likely produce the same errors.

Identifying When Retries Are Ineffective

A critical exam topic: not all errors are fixable by retrying. If the data is simply absent from the source document, no amount of retrying will make Claude produce it correctly. Effective systems distinguish between:

  • Extraction errors (Claude misread something that is present) → Retry is effective
  • Computation errors (Claude did the math wrong) → Retry is effective
  • Data absence (the information is not in the source) → Retry is futile; accept null
  • Ambiguous source (the source contradicts itself) → Retry is futile; escalate to human
💡

Production pattern: After 2 failed retries for the same field, check whether the field's value exists anywhere in the source. If not, set it to null with a coverage annotation like "invoice_date_source": "not_found_in_document" rather than continuing to retry.

5. Batch Processing & Multi-Pass Review

Message Batches API

The Message Batches API allows you to submit up to 100,000 requests in a single batch, receiving a 50% cost reduction compared to real-time API calls. The trade-off is latency: batches are processed within a 24-hour window (though most complete much sooner).

Feature Real-Time API Message Batches API
Cost Full price 50% discount
Latency Seconds Up to 24 hours (usually faster)
Multi-turn tool calling Supported Not supported
Max requests per batch N/A (individual calls) 100,000
Result correlation Direct response Via custom_id field
Best for Interactive, low-latency needs Bulk processing, overnight jobs

Matching API to Latency Requirements

The exam tests your ability to choose the right API for the task:

  • User-facing chat: Real-time API (latency matters)
  • Nightly document processing: Batch API (50% savings, latency acceptable)
  • Code review on PR submission: Real-time API (developer is waiting)
  • Monthly compliance audit of 10,000 documents: Batch API (overnight processing)
  • Tool-dependent extraction requiring multiple API calls: Real-time API (Batch API does not support multi-turn tool calling)

custom_id for Correlation

Each request in a batch includes a custom_id field that is echoed back in the result. This is your mechanism for correlating results to input records:

Batch Request with custom_id { "custom_id": "invoice-2024-00847", "params": { "model": "claude-sonnet-4-20250514", "max_tokens": 1024, "messages": [ {"role": "user", "content": "Extract data from: ..."} ], "tools": [/* extraction tool definition */], "tool_choice": {"type": "tool", "name": "extract_invoice"} } }

Self-Review Limitations

A critical exam topic: same-session self-review retains reasoning bias. When you ask Claude to review its own output within the same conversation, it has access to its own reasoning chain and is biased toward confirming its original answer. This is analogous to asking someone to proofread their own essay immediately after writing it -- they tend to read what they intended to write rather than what they actually wrote.

⚠

Exam answer: When asked about self-review reliability, the correct answer is that same-session review is unreliable because the model retains its reasoning context. Independent review requires a separate API call where the reviewing instance sees only the output, not the original reasoning.

Multi-Pass Architecture: Per-File + Cross-File

The recommended architecture for high-reliability document processing:

  1. Pass 1 -- Per-document extraction: Each document is processed independently by its own API call. This produces structured output for each document.
  2. Pass 2 -- Per-document review: A separate API call reviews each extraction result against the source document. This reviewer sees the source and the extraction but NOT the original reasoning.
  3. Pass 3 -- Cross-document synthesis: A final pass aggregates all reviewed extractions, checks for cross-document consistency (e.g., do all invoices from the same vendor use the same tax ID?), and flags anomalies.

This architecture is more expensive (3 API calls per document) but dramatically more reliable than single-pass extraction. The exam expects you to understand when the additional cost is justified.

ⓘ

Cost optimization: Use the Batch API for Pass 1 and Pass 2 (50% savings each). Only Pass 3 (cross-document synthesis) needs the real-time API if results are needed quickly. This reduces the total cost of the 3-pass architecture by approximately 33%.
