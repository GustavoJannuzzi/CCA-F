Cert Domain 3: Claude Code Config & Workflows

Master the 20% of the certification exam covering CLAUDE.md hierarchy, custom commands and skills, path-specific rules, plan mode, iterative refinement, and CI/CD integration. This is the largest single domain on the exam.

⏱ ~75 min ★ Expert 🎯 3 Interactive Exercises 📋 8-Question Practice Exam

1. Domain Overview

20% of Exam Weight

Domain 3 is the single largest domain on the certification exam, accounting for 20% of your total score. It tests your mastery of Claude Code's configuration system, workflow modes, and integration with CI/CD pipelines. This domain rewards practitioners who have spent time configuring real projects, not just reading documentation.

The domain is divided into 6 task areas. The CLAUDE.md hierarchy and commands/skills areas together account for the majority of questions.

The 6 Task Areas

1 CLAUDE.md Hierarchy
2 Custom Commands/Skills
3 Path-Specific Rules
4 Plan Mode
5 Iterative Refinement
6 CI/CD Integration
ⓘ

Exam strategy: CLAUDE.md hierarchy questions are the most common, and they are also the most frequently missed. The exam loves to present scenarios where a rule "should" be working but is not, and asks you to diagnose the scoping issue. Practice the "Resolution Preview" exercise below until you can trace the inheritance chain instantly.

What Makes This Domain Challenging

This domain tests system-level thinking. Individual features (plan mode, commands, path rules) are simple in isolation, but the exam combines them into scenarios where multiple features interact. For example: "A team uses plan mode for architecture decisions, has path-specific rules for Terraform files, and runs Claude in CI for code review. Which configuration ensures the CI pipeline respects both the plan mode output and the Terraform rules?"

⚠

Common trap: Confusing .claude/rules/ path-scoped rules with directory-level CLAUDE.md files. They solve similar problems but have different scoping mechanisms. The exam specifically tests when to use which.

2. CLAUDE.md Architecture for the Exam

The CLAUDE.md system is a hierarchical configuration cascade that determines what instructions Claude follows for any given file or task. Understanding the resolution order is essential for both the exam and real-world usage.

The Three-Level Hierarchy

🌐

Global: ~/.claude/CLAUDE.md

User-level

Personal preferences that apply to all projects. Things like preferred code style, personal shortcuts, or default behaviors you always want.

  • Applies to every project this user works on
  • Not version-controlled (personal to this machine)
  • Lowest priority -- overridden by project and directory rules
  • Exam trap: Rules here are invisible to teammates
📁

Project: ./CLAUDE.md

Repo root

Project-wide conventions shared by the entire team. This is the most important level for team collaboration.

  • Applies to all files in this repository
  • Version-controlled -- committed to the repository
  • Overrides global settings; overridden by directory-level
  • Can use @import to pull in modular config files
📂

Directory: subdir/CLAUDE.md

Subdirectory

Subdirectory-specific rules that apply only when working within that directory tree. Highest priority in the cascade.

  • Applies to files in this directory and below
  • Version-controlled alongside the directory's code
  • Highest priority -- overrides both global and project rules
  • Use for framework-specific conventions (e.g., React patterns in frontend/)

The @import Pattern

For large projects, a single CLAUDE.md file becomes unwieldy. The @import directive lets you split configuration into modular files.

CLAUDE.md # Project conventions Use TypeScript strict mode for all new files. Run `npm run lint` before committing. @import .claude/rules/testing.md @import .claude/rules/security.md @import .claude/rules/api-conventions.md

Path-Scoped Rules: .claude/rules/

The .claude/rules/ directory supports YAML frontmatter with a paths field that scopes the rule to specific file patterns. This is a more surgical alternative to directory-level CLAUDE.md files.

.claude/rules/terraform.md --- paths: ["terraform/**/*", "*.tf"] --- # Terraform Conventions - Always use `terraform fmt` before committing - Module sources must use version pins - State backend must be S3, never local - Tag all resources with team and environment

Diagnosing Hierarchy Issues

The exam frequently presents scenarios where configuration is not working as expected. Here is the diagnostic checklist:

Symptom Likely Cause Fix
New team member does not get project coding standards Rules are in ~/.claude/CLAUDE.md (user-level) instead of project-level Move rules to ./CLAUDE.md and commit to repo
Directory CLAUDE.md rules are being overridden Directory-level should have highest priority -- check if the user has conflicting global rules with !important-style overrides Review the full hierarchy chain; directory wins
Path-specific rules not applying to expected files The paths glob pattern does not match the target files Test the glob pattern; ensure **/* is used for recursive matching
Imported file rules not taking effect The @import path is wrong or the file does not exist Verify the relative path from the CLAUDE.md file's location

3. Commands, Skills & Path Rules

Custom Commands

Custom commands extend Claude Code with project-specific or personal workflows. They live as Markdown files in a .claude/commands/ directory.

Aspect Project Commands Personal Commands
Location .claude/commands/ (project root) ~/.claude/commands/ (home dir)
Invocation /project:command-name /user:command-name
Shared with team Yes -- committed to repo No -- personal only
Best for Project-specific workflows (deploy, review, test) Personal productivity shortcuts
.claude/commands/review.md Review the changes in the current branch against main. Focus on: 1. Type safety violations 2. Missing error handling 3. Security concerns (SQL injection, XSS) 4. Test coverage for new code paths Use `git diff main...HEAD` to see the changes. Provide a structured review with severity levels.

Skills: SKILL.md Frontmatter

Skills are enhanced commands with metadata that controls execution context. The SKILL.md frontmatter defines critical behavior.

SKILL.md Frontmatter --- name: deploy-staging context: fork allowed-tools: [Bash, Read, Glob] argument-hint: "branch name to deploy" --- # Deploy to Staging Deploy the specified branch to the staging environment. 1. Verify the branch exists 2. Run the test suite 3. Build the Docker image 4. Push to staging registry 5. Trigger deployment via kubectl

Key frontmatter fields for the exam:

  • context: fork -- Runs the skill in a separate context window (sub-agent), so it does not pollute the main conversation. Use for heavy research or long-running tasks.
  • allowed-tools -- Restricts which tools the skill can use. Critical for security-sensitive skills that should not have shell access.
  • argument-hint -- Tells the user what argument to pass when invoking the skill.
💡

Exam tip: The context: fork field is heavily tested. The exam asks when to use it (answer: when the skill involves extensive exploration or research that would fill the main context window) and what happens when it is missing (answer: the skill runs in the main context, potentially crowding out the user's conversation history).

Path-Specific Rules vs Directory CLAUDE.md

Both mechanisms scope rules to specific parts of the codebase, but they solve different problems. The exam specifically tests your ability to choose the right one.

Factor Path-Specific Rules Directory CLAUDE.md
Scoping mechanism Glob patterns in YAML frontmatter Physical directory location
Can span directories Yes -- e.g., ["*.tf", "modules/**/*"] No -- only the directory and its children
Best for Rules that apply to a file type regardless of where it lives Rules specific to a coherent subsystem (e.g., frontend/)
Location .claude/rules/ in project root Any subdirectory
Example use "All Terraform files must use version pins" "Frontend code uses React 18 patterns"
⚠

Decision rule for the exam: If the convention applies to files scattered across multiple directories (like all *.tf files or all *.test.ts files), use path-specific rules. If the convention applies to a cohesive subsystem that lives in one directory tree, use a directory CLAUDE.md.

4. Plan Mode vs Direct Execution

Claude Code offers two execution modes, and the exam tests your ability to choose the right one for a given scenario. This is not about personal preference -- there are objectively better choices depending on the task characteristics.

When to Use Plan Mode

📑

Plan Mode

Think First

Claude analyzes the problem, proposes a plan of action, and waits for your approval before making changes. Use when the cost of a wrong action is high.

  • Complex multi-file refactors: Changing an interface affects 20+ files
  • Architecture decisions: Choosing between approaches with trade-offs
  • Unfamiliar codebases: Claude needs to explore before acting
  • Breaking changes: Modifications that could impact other teams
  • Database migrations: Schema changes that are hard to reverse

When to Use Direct Execution

⚡

Direct Execution

Act Now

Claude makes changes immediately without waiting for approval. Use when the task is well-scoped and the risk is low.

  • Simple bug fixes: Off-by-one errors, typos, missing null checks
  • Well-defined additions: "Add a new endpoint that follows the existing pattern"
  • Test writing: Adding tests for existing code
  • Documentation updates: Low-risk content changes
  • Code formatting: Running linters, fixing style issues

The Explore Sub-Agent

The Explore sub-agent runs in a separate context window optimized for verbose discovery. It can read many files, trace call chains, and build understanding without filling your main context. The exam tests when to use it.

Combining Modes // Step 1: Plan mode -- understand the architecture "Plan how to migrate the authentication system from sessions to JWT tokens." // Step 2: Explore sub-agent -- research details "Use the Agent tool to find all session-related code across the codebase and map the dependencies." // Step 3: Direct execution -- implement "Now implement the JWT middleware as planned."
ⓘ

Exam pattern: "A developer needs to refactor the payment processing module, which touches 15 files across 3 services. What is the recommended workflow?" The answer is always: plan the architecture first, use explore to map dependencies, then execute implementation directly for each well-defined step.

5. CI/CD and Iterative Refinement

Claude Code can run in non-interactive mode for CI/CD pipelines, automated code review, and structured output generation. The exam tests your ability to configure these integrations correctly.

The -p Flag: Non-Interactive Mode

The -p (or --prompt) flag runs Claude Code as a one-shot command without interactive input. This is the foundation of all CI/CD integration.

Basic CI Usage # Non-interactive code review claude -p "Review the changes in this PR for security issues. \ Focus on SQL injection, XSS, and auth bypasses." # With structured output claude -p "Analyze this codebase for type safety issues" \ --output-format json \ --json-schema '{"type":"object","properties":{"issues":{"type":"array"}}}'

Structured Output: --output-format json

For CI pipelines that need machine-readable output, combine --output-format json with --json-schema to get structured, validated responses.

Structured Review Schema { "type": "object", "properties": { "summary": { "type": "string" }, "issues": { "type": "array", "items": { "type": "object", "properties": { "severity": { "type": "string", "enum": ["critical", "high", "medium", "low"] }, "file": { "type": "string" }, "line": { "type": "integer" }, "description": { "type": "string" }, "suggestion": { "type": "string" } } } }, "approved": { "type": "boolean" } } }

Including Prior Review Findings

A critical exam topic: when running Claude for iterative review, you must include the previous review's findings in the prompt to avoid duplicates. Without this, Claude will report the same issues on every run.

Iterative Review Pattern # First review claude -p "Review this PR" --output-format json > review-1.json # After fixes, include prior findings claude -p "Review this PR. These issues were already found and fixed in a previous review: $(cat review-1.json). Do NOT report these again. Focus on new issues only." \ --output-format json > review-2.json

Test-Driven Iteration

The interview pattern uses Claude in a loop: write code, run tests, feed failures back, iterate until all tests pass. This is especially powerful in CI.

  1. Define concrete input/output examples

    Write test cases first. Give Claude specific examples of expected behavior -- not vague descriptions, but actual input strings and expected output values.

  2. Generate implementation

    Use Claude to write the code, with the test cases as context. Claude can see exactly what "correct" looks like.

  3. Run tests, feed back failures

    Execute the test suite. If tests fail, pass the failure output back to Claude with the instruction to fix only the failing cases.

  4. Repeat until green

    Continue the loop. Each iteration should fix failures without introducing new ones. Set a maximum iteration count (typically 3-5) to prevent infinite loops.

💡

Exam tip: Questions about "concrete input/output examples" always refer to this pattern. The exam tests whether you know that vague instructions ("make it work") produce worse results than specific examples ("given input 'abc', return 'ABC'").
