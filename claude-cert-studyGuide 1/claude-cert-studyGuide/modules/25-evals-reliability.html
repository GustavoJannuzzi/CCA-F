Evaluations, Context Management & Reliability

Master the practices that separate hobby projects from production AI systems: measuring output quality with evaluations, managing your most precious resource (context), building confidence calibration, and making multi-agent pipelines resilient to failure.

⏱ ~90 min 📚 Advanced 🎓 Developers & Architects 🎯 4 Interactive Exercises

1. Why Evaluations Matter

"If you can't measure it, you can't improve it." This principle, fundamental to every engineering discipline, is doubly true for AI systems. Without evaluations, you are flying blind -- making prompt changes based on vibes, shipping updates without knowing if they help or hurt, and discovering regressions only when users complain.

Evaluations (evals) are the foundation of a systematic approach to AI quality. They transform prompt engineering from an art into a science by providing repeatable, quantifiable measurements of your system's performance.

The Eval Loop

Every production AI system should follow this continuous improvement cycle:

Build

Design your prompt, agent, or pipeline. Make your best educated guess at what will work based on the requirements.

Measure

Run your system against a curated evaluation set. Collect scores across multiple dimensions (accuracy, completeness, tone, safety).

Improve

Analyze where scores are weak. Revise prompts, add examples, adjust system architecture. Make targeted changes based on evidence.

Repeat

Run the updated system against the same eval set. Verify improvements did not introduce regressions. Continue iterating.

Three Types of Evaluation

Type How It Works Strengths Weaknesses
Human evaluation Expert reviewers rate outputs against criteria Gold standard; catches nuance and context Expensive, slow, doesn't scale
Automated / deterministic Regex, schema validation, exact match, keyword checks Fast, cheap, perfectly reproducible Only catches surface-level issues; brittle
Model-graded (LLM-as-judge) Use Claude to evaluate Claude's output against a rubric Scalable, nuanced, can evaluate open-ended quality Potential bias; needs calibration against humans

When to Use Each Type

  • Human evaluation -- Use during initial development and for periodic "ground truth" checks. Essential for high-stakes domains (medical, legal, financial).
  • Automated/deterministic -- Use as a first line of defense for every output. Check JSON schema validity, required field presence, forbidden content patterns, response length bounds.
  • Model-graded -- Use as your primary scaling mechanism. Run after every prompt change against your full eval set. Supplement with periodic human review to keep the judge calibrated.
⚡

Best practice: Use all three types in combination. Deterministic checks catch structural failures instantly. LLM-as-judge catches quality issues at scale. Human review calibrates the judge and catches what machines miss.

2. LLM-as-Judge

The most powerful innovation in AI evaluation is using a separate Claude instance to grade outputs against a structured rubric. This approach combines the nuance of human judgment with the scalability of automated evaluation.

Why It Works

Claude can assess qualities that are impossible to capture with regex or keyword matching: Is the response relevant to the user's actual intent? Is it complete without being unnecessarily verbose? Does the tone match the context? Is it actionable -- can the user actually do something with this information?

The Judge Prompt Anatomy

A well-structured judge prompt has four components: role definition, evaluation criteria with explicit scoring rubric, the response to evaluate, and output format instructions.

Judge Prompt Template You are evaluating an AI assistant's response. Criteria: - Accuracy (1-5): Are all facts correct? 1 = Contains factual errors 3 = Mostly correct with minor imprecisions 5 = All facts verified and precise - Completeness (1-5): Does it address all parts of the question? 1 = Misses major aspects of the question 3 = Addresses the main point but misses secondary aspects 5 = Thoroughly addresses every part - Clarity (1-5): Is it well-organized and easy to understand? 1 = Confusing, disorganized, or overly technical 3 = Understandable but could be better structured 5 = Crystal clear with logical flow - Actionability (1-5): Can the user act on this information? 1 = Vague advice with no concrete next steps 3 = Some actionable items but missing specifics 5 = Clear, specific steps the user can take immediately Response to evaluate: [response] Provide scores and brief justification for each criterion. Format as JSON: { "accuracy": { "score": N, "justification": "..." }, "completeness": { "score": N, "justification": "..." }, "clarity": { "score": N, "justification": "..." }, "actionability": { "score": N, "justification": "..." } }

Best Practices

Use a Different Model Instance

The generating instance retains its own reasoning context, making it biased toward its own output. Always use a separate instance (or a different model entirely) for evaluation. This is analogous to the principle that you should not grade your own exam.

Include Clear Rubric Definitions

Don't just say "rate accuracy 1-5." Define what each score level means with concrete examples. What does a "3" look like? How is it different from a "4"? Without this, the judge's scores will drift over time.

Calibrate Against Human Judgments

Run the judge on a sample set where you have human scores. Compare the results. If the judge consistently scores 1 point higher than humans on "completeness," adjust the rubric. Track this calibration over time.

Track Inter-Rater Reliability

Run the same evaluation multiple times. If the judge gives the same response a 3 on one run and a 5 on another, your rubric is too vague. Consistency (low variance across runs) is as important as accuracy (matching human scores).

⚠

Watch out for self-serving bias: Claude tends to rate Claude's outputs more favorably than human experts would. Always validate your judge against a human baseline before trusting it for production decisions.

3. Building Evaluation Sets

An evaluation is only as good as the data you evaluate against. Your golden dataset is the collection of representative examples that define what "good" looks like for your system.

Size and Composition

A golden dataset should contain 20-50 representative examples with expected outputs. This is large enough to catch common failure modes but small enough to manually curate and maintain. Each example should include the input, the expected output (or acceptable output range), and metadata about what makes this case important.

Diversity Matters

Your eval set must cover three categories:

  • Easy cases (40%) -- Common, straightforward inputs that your system should handle flawlessly. These establish your baseline accuracy.
  • Hard cases (30%) -- Complex inputs that require multi-step reasoning, disambiguation, or domain expertise. These test your system's ceiling.
  • Edge cases and adversarial examples (30%) -- Tricky inputs that commonly fool AI: ambiguous phrasing, contradictory context, requests that look benign but require refusal, inputs with subtle errors.

Eval Set Discipline

Version Your Eval Sets

Never modify eval examples in place. Instead, create new versions (eval-set-v1, eval-set-v2) so you can track how your system's performance changes over time against a stable benchmark. If you change both the prompt and the eval set simultaneously, you cannot attribute improvements or regressions to either change.

Regression Testing

Every prompt change, every system architecture change, every model upgrade should be tested against your eval set before deployment. Treat your eval set like a test suite: if a change causes a score regression on 3 of 50 examples, investigate before shipping.

A/B Testing

When comparing two prompts, run both against the exact same eval set. Compare scores dimension by dimension. A prompt that improves accuracy by 0.5 points but drops actionability by 1.2 points is not an improvement -- it's a tradeoff you need to evaluate consciously.

ⓘ

Pro tip: Include adversarial examples that test specific failure modes you have discovered. If you once found that your agent gives medical advice when it shouldn't, add that exact scenario to your eval set as a regression guard.

4. Context Management

The context window is your most precious resource. Every token you put into the context competes with every other token for the model's attention. Overloaded context leads to missed instructions, forgotten facts, and degraded reasoning. Managing context effectively is what separates amateur prompt engineering from production-grade systems.

The "Lost in the Middle" Effect

Research has demonstrated that language models process information at the beginning and end of long inputs more reliably than information buried in the middle. This is not a minor quirk -- it is a fundamental characteristic of transformer attention mechanisms. When your context is 50,000+ tokens, information in positions 20,000-30,000 may receive significantly less attention than information in the first or last 5,000 tokens.

⚠

Critical implication: Always place your most important instructions and facts at the beginning or end of your prompt. Never bury critical business rules in the middle of a long context.

Progressive Summarization Risks

A common technique for long conversations is to summarize earlier messages. But summarization inherently loses information. Numerical values, specific dates, exact quotes, and conditional logic are particularly vulnerable to being generalized or dropped during summarization. If your workflow depends on specific facts from early in the conversation, use structured context blocks rather than relying on summaries.

Strategies for Context Management

Selective Injection

Only include context that is directly relevant to the current task. Resist the temptation to dump everything into the prompt "just in case." If your agent is answering a question about shipping policies, it does not need the full product catalog in context.

Structured Context Blocks

For critical information that must persist across a conversation, create a dedicated "case facts" section at the top of every prompt. Format it as a clearly delimited block with labeled fields. This prevents important facts from getting lost as the conversation grows.

Structured Context Block <case_facts> Customer: Jane Smith (ID: C-4829) Account tier: Enterprise Issue: Invoice #INV-2024-0312 overcharged by $1,247.50 Resolution authority: Up to $2,000 without escalation Previous contacts: 2 (both unresolved) </case_facts>

Tool Output Trimming

Verbose tool results are context killers. A full pytest -v output might consume 8,000 tokens when all you need is "3 tests passed, 1 failed: test_auth_flow." Trim tool outputs to the relevant fields before they accumulate in context.

The /compact Command

In Claude Code, the /compact command compresses your conversation context. Use it when you notice the conversation getting long, especially before starting a new phase of work. It summarizes earlier exchanges while preserving key decisions and facts.

Subagent Delegation

For research-heavy tasks, delegate to a subagent with its own context window. The subagent reads 50 files, synthesizes findings, and returns a concise 300-token summary. Your main conversation never sees the 50 files, preserving context for the actual work.

Token Budgeting

Treat context like a financial budget. Allocate token quotas to each component: system prompt (2,000), persistent context (1,000), conversation history (variable), tool outputs (capped). 5. Escalation and Confidence

One of the most important skills for an AI system is knowing when not to answer. An AI that confidently gives wrong advice is far more dangerous than one that says "I don't know" and escalates to a human.

When Claude Should Say "I Don't Know"

Not every question requires an AI-generated answer. The mark of a well-designed system is that it can recognize the boundaries of its competence and hand off gracefully.

Escalation Triggers

  • Customer explicitly asks for a human -- Always honor this request immediately. No amount of "But I can help with that!" is acceptable. The customer's autonomy comes first.
  • Policy gaps -- When Claude encounters a situation that the system prompt and available documentation do not address. It is better to say "I need to check with a colleague" than to invent a policy.
  • Ambiguous situations with no clear right answer -- When two reasonable interpretations of a policy would lead to different outcomes, escalate rather than guess.
  • High-stakes decisions requiring human accountability -- Medical diagnoses, legal advice, financial commitments, termination of service. These require a human in the loop regardless of Claude's confidence.

What NOT to Escalate

Over-escalation is as much a problem as under-escalation. If Claude escalates every question it is slightly unsure about, human reviewers get overwhelmed and the AI adds no value. Things Claude can and should handle, even with some uncertainty:

  • Answering factual questions from provided documentation
  • Standard troubleshooting steps for known issues
  • Generating code that follows established patterns in the codebase
  • Summarizing content that has been provided in context

Confidence Calibration

⚠

Critical insight: Claude's self-reported confidence is poorly calibrated. When Claude says "I'm fairly confident," this does not correlate well with actual accuracy. Do not use "Are you confident?" as a reliability filter.

Instead of relying on Claude's self-assessment, build confidence into your system design:

  • Use explicit criteria -- Define specific conditions under which escalation should happen, not vague "if you're unsure" instructions.
  • Provide few-shot examples -- Show Claude examples of situations that should and should not be escalated. Concrete examples are more reliable than abstract rules.
  • Source-based confidence -- An answer backed by a specific document citation is more reliable than one from general knowledge. Build this distinction into your system.

Handoff Patterns

When escalating, provide the human reviewer with a structured summary:

Escalation Handoff Template <escalation_summary> Reason: Policy gap -- customer requesting exception not covered in guidelines Customer: Jane Smith (Enterprise tier, 3-year relationship) Request: Waive annual renewal fee ($4,800) due to service outage Relevant context: - 47-hour outage affected this customer's production environment - SLA guarantees 99.9% uptime (this month: 93.4%) - No existing policy for fee waiver vs. service credit Attempted resolution: Offered standard 10% service credit ($480) Customer response: Rejected, requesting full fee waiver My assessment: Customer request seems reasonable given SLA breach severity </escalation_summary>

6. Error Propagation in Multi-Agent Systems

In a single-agent system, errors are straightforward: something fails, you retry or report. In multi-agent pipelines, errors cascade. One agent's failure becomes corrupted input for the next agent, which produces bad output for the next, and so on. Without proper error handling, a minor failure in step 1 becomes a catastrophic failure by step 4.

Error Boundaries

Borrow from React's error boundary pattern: each agent in your pipeline should catch and handle errors at its own level. An error in the "Enrich" agent should not crash the "Extract" agent. Each agent should be able to report: what happened, what was attempted, and what partial results are available.

The Structured Error + Partial Results Pattern

Error Response Structure { "status": "partial_failure", "completed_steps": ["extract", "validate"], "failed_step": "enrich", "error": { "type": "ServiceTimeout", "message": "Enrichment API timed out after 30s", "attempted": "3 retries with exponential backoff" }, "partial_results": { "extracted_data": { /* ... available data ... */ }, "validation_status": "passed" }, "skipped_steps": ["store"], "suggested_action": "Retry enrichment or proceed with unenriched data" }

Recovery Strategies

  • Retry with backoff -- For transient errors (timeouts, rate limits, temporary service outages). Use exponential backoff: 1s, 2s, 4s, 8s. Set a maximum retry count.
  • Fallback to a simpler approach -- If the enrichment API fails, use the data you already have rather than failing entirely. A response with unenriched data is better than no response.
  • Continue with partial results -- If 4 of 5 data sources are available, produce a result with a disclaimer about missing information. Let the consumer decide if the partial result is sufficient.
  • Escalate to human -- For unrecoverable errors that affect business-critical decisions. Include the full error context so the human reviewer can diagnose and resolve.

Cascade Prevention

Design each agent to accept degraded input gracefully. If Agent 2 expects enriched data but receives raw data (because the enrichment step failed), it should still produce a useful result rather than crashing. This requires defensive input validation and fallback behavior at every stage.

⚡

Design principle: Every agent should be able to run independently with degraded input and produce a partial result. If your pipeline requires every step to succeed for any output to be produced, your architecture is fragile.

7. Provenance and Audit Trails

In production AI systems, knowing what the system said is only half the story. You also need to know why it said it -- which sources informed the output, what evidence supports each claim, and whether that evidence is current and reliable.

Information Provenance

Provenance means tracking the lineage of information from source to output. Every assertion in your AI's response should be traceable to a specific source: a document, a database record, a tool output, or the model's training data (in which case it should be marked as general knowledge rather than cited as fact).

Claim-Source Mappings

The gold standard for provenance is a claim-source mapping: each assertion in the output is linked to the specific evidence that supports it.

Claim-Source Mapping Example Output: "The customer's contract expires March 15, 2025 [1]. Their account has been active for 3 years [2]. Based on current pricing, renewal would be $48,000/year [3]." Sources: [1] Contract document, Section 4.2, paragraph 1 [2] CRM record, account_created_at: 2022-03-15 [3] Pricing API response, enterprise_annual tier, retrieved 2025-01-10

Handling Conflicting Sources

When sources disagree, annotate the conflict rather than silently choosing one. If the contract says the renewal price is $48,000 but the current pricing page shows $52,000, the system should surface both values and explain the discrepancy rather than presenting one as fact.

Temporal Data

Always include publication dates when citing sources. A market analysis from 2021 should not be presented as current data in 2025. When using tool outputs, include the retrieval timestamp. This prevents misinterpreting stale data as current.

Audit Trail Requirements

Production AI systems, especially in regulated industries, need comprehensive audit trails:

  • Who -- Which user or system triggered the interaction
  • What -- The complete input (prompt + context) and output
  • When -- Timestamps for each step of the interaction
  • Why -- The sources and reasoning chain that produced the output
  • How -- The model version, prompt version, and system configuration used

Compliance Frameworks

Framework Key Requirements AI-Specific Considerations
SOC 2 Access controls, change management, audit logs Log all prompt changes; track who approved model updates; record all user interactions
HIPAA PHI access logging, minimum necessary, breach notification Ensure PHI is not stored in model context beyond the session; log all access to patient data via AI
GDPR Right to explanation, data minimization, consent Be able to explain how AI reached a decision; do not retain personal data in training or context
ⓘ

Remember: Audit trails are not just about compliance -- they are invaluable for debugging. When a user reports that the AI gave wrong advice two weeks ago, the audit trail lets you reconstruct exactly what happened: what context was available, what prompt version was active, and what the model produced.
