🤖 Domínio 1: Arquitetura Agêntica & Orquestração

27% do exame · ~16 questões · o maior peso

Este domínio é sobre como um agente decide o que fazer sozinho e como vários agentes se coordenam. Quase tudo aqui vive no 🧩 Agent SDK (você constrói a lógica em código) — com paralelos no 🖥️ Claude Code, onde o agente já vem pronto.

1.1O loop agêntico
🧩 Agent SDK 🔌 Claude API 🖥️ Claude Code 🌐 MCP
Em uma frase: o loop agêntico é o ciclo "mando pro Claude → ele pede uma ferramenta → eu executo → devolvo o resultado → ele decide o próximo passo", repetindo até ele terminar. Quem controla esse ciclo é o campo stop_reason.
📦 O cenário
Você está construindo o Customer Support Resolution Agent (Cenário 1) com o Agent SDK. O cliente escreve "meu pedido veio errado". Você não sabe de antemão quantas ferramentas o agente vai precisar chamar — ele decide na hora: buscar o cliente, depois o pedido, depois avaliar reembolso. O loop é o que permite essa autonomia.
⚙️ Como funciona

A cada resposta da API você inspeciona stop_reason. Se for "tool_use", o Claude quer usar uma ferramenta — você executa, anexa o resultado ao histórico e chama de novo. Se for "end_turn", ele terminou — você mostra a resposta. É a API que decide, não uma árvore de decisão sua.

Por que reenviar tudo? A Claude API é stateless — não guarda memória entre chamadas. Você reenvia o array messages inteiro a cada volta, e cada tool_result carrega o tool_use_id que o casa com a chamada correspondente (mais um is_error opcional). Esse "reenviar o histórico" é o que o Domínio 5 chama de gerenciar contexto.

🐍 Loop agêntico mínimo (Agent SDK / API)
while True:
    resp = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=2048,
        tools=tools, messages=messages,
    )
    if resp.stop_reason == "tool_use":          # Claude quer uma ferramenta
        messages.append({"role": "assistant", "content": resp.content})
        results = [executar(b) for b in resp.content if b.type == "tool_use"]
        messages.append({"role": "user", "content": results})  # devolve p/ o modelo
    elif resp.stop_reason == "end_turn":         # terminou
        break
🔄 Fluxo
  enviar request ──▶ stop_reason?
                       │
        ┌──────────────┴──────────────┐
     "tool_use"                    "end_turn"
        │                              │
   executa ferramenta            mostra resposta
   anexa resultado ao histórico       (FIM)
        └────────▶ (volta ao topo)

No Claude Code esse loop já está pronto — quando você pede "refatora e roda os testes", ele roda esse ciclo internamente. No SDK, você escreve o while.

🎯 Quando usar / quando não
✅ Faça
  • Terminar o loop SÓ por stop_reason == "end_turn"
  • Anexar todo tool_result ao histórico antes da próxima chamada
⚠️ Evite (anti-padrões)
  • Procurar texto tipo "tarefa concluída" para parar — o modelo pode dizer isso e ainda querer chamar ferramentas
  • Usar um limite fixo de iterações como mecanismo primário de parada
⚠️ No exame

Se a questão diz "o loop para cedo / resposta incompleta", a resposta é usar stop_reason, não parsear o texto da resposta. Analisar linguagem natural para decidir parada e caps arbitrários de iteração são as distratoras clássicas.

1.2Orquestração coordenador-subagente
🧩 Agent SDK 🖥️ Claude Code 🔌 Claude API 🌐 MCP
Em uma frase: um agente coordenador decompõe a tarefa e delega pedaços a subagentes especializados (hub-and-spoke). Toda comunicação passa pelo coordenador — subagentes não conversam entre si.
📦 O cenário
Você está construindo o Multi-Agent Research System (Cenário 3): um coordenador delega para um subagente que busca na web, um que analisa documentos, um que sintetiza e um que gera o relatório. O coordenador decide quais subagentes acionar conforme a complexidade da pergunta.
⚙️ Como funciona — e onde isso vive
🖥️ No Claude Code — automático
  • O próprio Claude já é o coordenador
  • Ele spawna subagentes via ferramenta Task quando a tarefa pede
  • Você não escreve a orquestração — só influencia via CLAUDE.md
🧩 No Agent SDK — você constrói
  • Você define o coordenador e cada subagente (AgentDefinition)
  • O coordenador precisa de "Task" em allowedTools para delegar
  • Você decide a decomposição e a agregação dos resultados
🛰️ Hub-and-spoke
              ┌─────────────┐
       ┌──────│ COORDENADOR │──────┐      ⚠ subagentes NÃO
       ▼      └─────────────┘      ▼        falam entre si:
   [busca web]   [análise doc]   [síntese]  tudo passa pelo hub
🎯 Quando usar / quando não
✅ Use multi-agente
  • Subtarefas independentes que cabem ser paralelizadas
  • Especializações distintas (buscar ≠ sintetizar)
⚠️ Cuidado
  • Decomposição estreita demais = cobertura incompleta do tópico
  • Tarefa simples e linear não precisa de coordenador
⚠️ No exame

Questão clássica: "todos os subagentes tiveram sucesso, mas o relatório cobriu só parte do tópico". A causa é decomposição estreita do coordenador — não culpe os subagentes (que fizeram bem o que foi pedido). Olhe o log do coordenador.

1.3Invocação de subagentes, passagem de contexto & paralelismo
🧩 Agent SDK 🖥️ Claude Code 🔌 Claude API 🌐 MCP
Em uma frase: subagentes têm contexto isolado — não herdam o histórico do coordenador. Tudo que precisam tem que ir explícito no prompt da Task. E para rodar em paralelo, o coordenador emite várias Task numa única resposta.
📦 O cenário
No Multi-Agent Research System, o subagente de síntese precisa dos resultados da busca e da análise. Se você só disser "sintetize as descobertas", ele responde "que descobertas?" — porque começou do zero. Você precisa passar os resultados dentro do prompt dele.
⚙️ Como funciona
Passagem de contexto — errado vs certo
❌ ERRADO — subagente não herdou nada:
Task(prompt="Investigue o problema de cobrança do cliente")

✅ CERTO — contexto completo no prompt:
Task(prompt="""Investigue cobrança dupla.
Cliente: CUS-4829 | Pedido: ORD-789
Cobrado 2x em 15/03 | Valor: R$249,90
Ferramentas: get_order_details, process_refund""")

Paralelo vs sequencial: para paralelismo real, o coordenador emite múltiplas chamadas Task na MESMA resposta. Uma Task por turno = execução sequencial.

✅ Paralelo (tarefas independentes)
  • 3 Task numa única resposta: pesquisar UE, EUA e China
  • Tempo total ≈ o da subtarefa mais longa
⏭️ Sequencial (há dependência)
  • Turnos separados quando B precisa do resultado de A
  • Ex.: coletar dados → depois analisar o que foi coletado

Boa prática: passe metadados estruturados (URL da fonte, nome do doc, página) junto do conteúdo, para preservar a atribuição na síntese. E escreva prompts de coordenador com objetivos e critérios de qualidade, não um passo-a-passo rígido — isso deixa o subagente se adaptar.

⚠️ No exame

"Subagente responde: não tenho informações" → passar contexto explícito no prompt da Task. "Como paralelizar?" → múltiplas Task em UMA resposta (não turnos consecutivos, não threads). E lembre: allowedTools precisa incluir "Task" para o coordenador poder spawnar.

1.4Workflows multi-etapa com enforcement & handoff
🧩 Agent SDK 🔌 Claude API 🖥️ Claude Code 🌐 MCP
Em uma frase: quando uma ordem de passos é obrigatória (ex.: verificar identidade antes de reembolsar), prompt não basta — ele tem taxa de falha não-zero. Use enforcement programático (hooks / gates de pré-requisito).
📦 O cenário
No Customer Support Agent, a política diz: nunca processar reembolso sem verificar o cliente primeiro. Em 12% dos casos o agente pula get_customer e chama process_refund direto — risco financeiro real.
⚙️ Como funciona

Um gate programático — um hook PreToolUse, que roda antes da ferramenta — bloqueia process_refund e lookup_order até que get_customer tenha retornado um ID verificado. Isso é código — uma garantia determinística — não uma instrução no prompt que o modelo "deveria" seguir.

🧩 PreToolUse — gate de pré-requisito (Agent SDK)
def antes_da_ferramenta(tool, state):
    if tool.name == "process_refund" and not state.customer_verified:
        return bloquear("Verifique o cliente com get_customer primeiro")
    return permitir()

Handoff estruturado: ao escalar no meio do processo para um humano (que não vê o histórico da conversa), compile um resumo: ID do cliente, causa-raiz, valor, ação recomendada.

⚠️ No exame

"Em X% dos casos o agente pula a verificação apesar do system prompt" → pré-requisito programático / hook. As distratoras "reforçar o system prompt" e "adicionar few-shot" dependem de conformidade probabilística — insuficiente quando o erro tem consequência financeira.

1.5Hooks do Agent SDK — PreToolUse (bloqueia) vs PostToolUse (transforma)
🧩 Agent SDK 🖥️ Claude Code 🔌 Claude API 🌐 MCP
Em uma frase: hooks são pontos de interceptação determinísticos ligados a eventos do agente. Os dois que mais caem: PreToolUse roda antes da ferramenta e pode bloquear/negar a ação; PostToolUse roda depois e transforma o resultado antes do modelo ver.
📦 O cenário
No Customer Support Agent, duas regras. (1) Reembolso acima de R$500 não pode ser processado sem aprovação humana — precisa ser barrado antes de executar → PreToolUse. (2) Suas ferramentas MCP retornam datas em formatos diferentes (Unix, ISO 8601, código numérico) — normalize tudo depois que a ferramenta responde → PostToolUse.
⚙️ Como funciona — a linha do tempo importa
⏱️ Onde cada hook entra
  modelo pede a ferramenta
        │
        ▼
   ┌────────────┐  pode NEGAR aqui  (permissionDecision: "deny")
   │ PreToolUse │──✗──▶ ação bloqueada → escala p/ humano
   └────────────┘  ← reembolso > R$500, gate de verificação
        │ (permitido)
        ▼
   FERRAMENTA EXECUTA   (dinheiro sai, query roda…)
        │
        ▼
   ┌────────────┐  transforma/apara o RESULTADO
   │ PostToolUse│──▶ datas normalizadas, 45 campos → 5
   └────────────┘  ← nunca bloqueia: a ação já aconteceu
        │
        ▼
   resultado limpo entra no contexto do modelo
🚫 PreToolUse — bloqueia ANTES (nega a ação)
def pre_tool_use(tool_name, tool_input):
    if tool_name == "process_refund" and tool_input["amount"] > 500:
        return {"hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "deny",   # allow | deny | ask | defer
            "permissionDecisionReason": "> R$500 exige aprovação humana"}}
🧹 PostToolUse — transforma DEPOIS (o resultado)
def post_tool_use(tool_name, result):
    if "date" in result:
        result["date"] = para_iso8601(result["date"])  # formato único
    return result  # resultado limpo p/ o modelo

O ponto-chave do exame: hook = garantia determinística; prompt = conformidade probabilística. E o timing decide qual hook: se precisa impedir algo, é PreToolUse (antes); se é ajustar o que voltou, é PostToolUse (depois). No 🖥️ Claude Code os mesmos hooks vivem no settings.json.

⚠️ No exame

Bloquear uma ação (reembolso acima do limite, gate de verificação) → PreToolUse com permissionDecision: "deny", porque roda antes de executar. PostToolUse roda depois — serve para normalizar/aparar tool results verbosos (ligação com o Domínio 5), nunca para impedir a ação (que já aconteceu).

1.6Estratégias de decomposição de tarefas
🧩 Agent SDK 🖥️ Claude Code 🔌 Claude API 🌐 MCP
Em uma frase: escolha entre pipeline fixo (prompt chaining — passos previsíveis) e decomposição dinâmica (subtarefas geradas conforme as descobertas) com base em quão aberto é o problema.
📦 O cenário
No Developer Productivity (Cenário 4): "adicione testes abrangentes a um codebase legado" é aberto — você mapeia a estrutura, identifica áreas de alto impacto e cria um plano que se adapta. Já uma revisão de PR multi-arquivo é previsível — analise cada arquivo, depois faça uma passagem de integração entre arquivos.
🎯 Quando usar / quando não
🔗 Prompt chaining (fixo)
  • Revisões multi-aspecto previsíveis
  • Dividir revisão grande em: por-arquivo + integração
🌱 Decomposição dinâmica
  • Investigação aberta (explorar legado)
  • Gera subtarefas conforme dependências aparecem
⚠️ No exame

Revisão de muitos arquivos numa passagem só → resultado inconsistente por diluição de atenção. A resposta é dividir em passagens (por-arquivo + integração), não "usar uma janela de contexto maior" (ver 4.6).

1.7Estado de sessão: resume & fork
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: --resume <nome> (ou -c/--continue) continua uma sessão; --fork-session ao retomar cria um novo id (preservando o original), e /fork abre um subagente que herda a conversa. Mas quando os tool results estão obsoletos, começar uma sessão nova com um resumo estruturado é mais confiável que retomar.
📦 O cenário
No Developer Productivity, você analisou um codebase ontem. Hoje os arquivos mudaram. Retomar a sessão antiga traz resultados de ferramentas obsoletos (o conteúdo já não bate). Forkar é útil para comparar duas abordagens de refatoração a partir da mesma análise base.
🎯 Quando usar / quando não
✅ Resume / fork
  • --resume: contexto anterior ainda majoritariamente válido
  • --fork-session / /fork: explorar caminhos divergentes da mesma base
🆕 Sessão nova + resumo
  • Quando tool results estão obsoletos/contaminados
  • Injeta um resumo estruturado em vez de arrastar lixo
⚠️ No exame

"O contexto degradou / o Claude cita 'padrões típicos' em vez das classes específicas" → nova sessão com resumo estruturado, NÃO --resume. Ao retomar após mudanças, informe explicitamente quais arquivos mudaram para re-análise direcionada.

🔧 Domínio 2: Tool Design & MCP Integration

18% do exame · ~11 questões

Aqui o tema é as ferramentas que o Claude pode chamar: como descrevê-las para ele escolher certo, como expor seu backend via 🌐 MCP, como erros voltam, e quais são as ferramentas nativas do 🖥️ Claude Code.

2.1Descrições de ferramentas claras
🌐 MCP 🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code
Em uma frase: a descrição é o mecanismo PRIMÁRIO que o modelo usa para escolher a ferramenta — não o nome, não o código. Descrição fraca = seleção errada em produção.
📦 O cenário
No Customer Support Agent, get_customer ("Retrieves customer information") e lookup_order ("Retrieves order details") têm descrições mínimas e aceitam IDs parecidos. Quando o usuário diz "check my order #12345", o agente às vezes chama get_customer — bug.
⚙️ Como funciona
Descrição com fronteiras explícitas
lookup_order: "Retorna DETALHES DE PEDIDO por order ID (#12345 ou ORD-XXXXX).
  Use para: status, rastreio, itens, entrega, elegibilidade de reembolso.
  NÃO use para: dados de perfil do cliente — use get_customer.
  Ex.: 'onde está meu pedido', 'status do pedido #12345'"

Regra de ouro: se dois devs discordariam sobre qual ferramenta usar para um input, a descrição está ruim. Adicione formatos de entrada, exemplos, casos extremos e um "NÃO use para → use X". Cuidado também com o system prompt: instruções sensíveis a palavra-chave podem sobrescrever descrições boas.

⚠️ No exame

Misrouting é quando o Claude erra a escolha e chama a ferramenta errada. Acontece quando você tem duas tools parecidas.

Por que acontece? O Claude escolhe qual ferramenta chamar lendo a description dela — não o nome, não o código, só o texto da descrição. Então, se ele erra, a causa raiz quase sempre é: as descrições estão fracas / ambíguas / parecidas demais.

A correção (resposta certa): expandir as descrições — primeiro passo, baixo esforço, alta alavancagem. Deixe explícito:

  • pra que serve (input, formato)
  • exemplos de quando usar
  • quando NÃO usar ("para status de pedido, use lookup_order")

Distratoras (PEGADINHAS):

  • "adicionar few-shot" (dar ao modelo poucos exemplos de entrada→saída) → só adiciona overhead, não conserta a descrição
  • "camada de roteamento por keywords" → over-engineering
  • "consolidar numa tool só" → over-engineering desnecessário
2.2Respostas de erro estruturadas (MCP)
🌐 MCP 🧩 Agent SDK 🔌 Claude API 🖥️ Claude Code
Em uma frase: a flag isError comunica falha ao agente — mas um erro genérico ("Operation failed") impede recuperação inteligente. Retorne metadados estruturados: categoria, se é retryável, e descrição legível. Atenção: só isError é campo do protocolo MCP — o resto é convenção sua, dentro do conteúdo do resultado.
📦 O cenário
Sua ferramenta process_refund pode falhar por motivos muito diferentes: timeout (transitório), input inválido (validação), valor acima da política (negócio), sem permissão. O agente precisa saber qual é para decidir: tentar de novo? explicar ao cliente? escalar?
⚙️ Como funciona
Erro estruturado
{
  "isError": true,              # ← único campo do PROTOCOLO MCP
  "errorCategory": "business",  # daqui p/ baixo = convenção SUA, no conteúdo:
  "isRetryable": false,        #   transient | validation | business | permission
  "message": "Reembolso acima de R$500 exige aprovação humana"
}

Distinga também falha de acesso (timeout — precisa decisão de retry) de resultado vazio válido (consulta funcionou, não achou nada). Em multi-agente: subagentes resolvem falhas transitórias localmente e só propagam ao coordenador o que não conseguem resolver, junto com resultados parciais e o que foi tentado.

⚠️ No exame

"Erro genérico impede o agente de decidir" → metadados estruturados (errorCategory + isRetryable). isRetryable: false + mensagem amigável para violações de regra de negócio evita tentativas inúteis.

2.3Distribuição de ferramentas & tool_choice
🔌 Claude API 🧩 Agent SDK 🌐 MCP 🖥️ Claude Code
Em uma frase: dar ferramentas demais a um agente (18 em vez de 4-5) degrada a seleção. Dê a cada agente só o que o seu papel exige. E use tool_choice para controlar SE/QUAL ferramenta é chamada.
📦 O cenário
No Multi-Agent Research, o agente de síntese tentava fazer buscas na web (fora da especialização dele) e errava. Você restringe o set de ferramentas dele — e dá só uma verify_fact com escopo para os fact-checks simples (85% dos casos), deixando o resto via coordenador.
⚙️ Como funciona — tool_choice
ValorComportamentoQuando
"auto"Pode chamar ferramenta OU responder textoPadrão conversacional
"any"DEVE chamar alguma ferramenta (qualquer)Garantir saída estruturada
{"type":"tool","name":"X"}DEVE chamar a ferramenta XForçar uma etapa primeiro (ex.: extract_metadata)
⚠️ No exame

"85% das verificações são simples, 15% complexas" → ferramenta com escopo (verify_fact) para o caso comum, mantendo o caminho do coordenador para o resto = princípio do menor privilégio. Dar todas as ferramentas (over-provisioning) ou batch de verificações (cria bloqueio) são distratoras.

2.4Integração de servidores MCP
🌐 MCP 🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API
Em uma frase: MCP conecta o Claude ao seu backend. Config do time vai em .mcp.json (raiz do projeto, versionado); config pessoal/experimental em ~/.claude.json. Tokens vêm de variáveis de ambiente, nunca hardcoded.
📦 O cenário
No Developer Productivity, o time integra o servidor MCP do GitHub. Todo dev que clonar o repo deve receber a config automaticamente — por isso ela vai versionada no .mcp.json, com o token vindo do ambiente de cada um.
⚙️ Como funciona
.mcp.json (raiz do projeto, versionado)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }  # do ambiente

}

Resources vs Tools (sempre cai): resource = leitura passiva de catálogo (listar schemas de banco, hierarquia de docs); tool = ação com efeito colateral (rodar query, criar registro). Expor catálogos como resources reduz chamadas exploratórias. Ferramentas de todos os servidores configurados ficam disponíveis simultaneamente. Prefira servidores MCP da comunidade (ex.: Jira) a reimplementar.

⚠️ No exame

Config do time em ~/.claude.json = colegas não recebem (erro clássico). Deve ser .mcp.json na raiz. "Ler catálogo para decidir" = resource; "mudar o estado do sistema" = tool. Se o agente prefere Grep nativo a uma tool MCP melhor, melhore a descrição da tool MCP.

Em uma frase: Grep busca por conteúdo; Glob busca por nome/caminho; Read/Write são arquivo inteiro; Edit é alteração pontual; Bash é o único que executa comandos.
📦 O cenário

⚙️ Domínio 3: Claude Code Configuration & Workflows

20% do exame · ~12 questões

Tudo aqui vive no 🖥️ Claude Code: os arquivos que você coloca no repo para configurar o agente pronto, os comandos e skills do time, e como rodar o Claude no CI/CD. A regra-mãe: config do time é versionada; config pessoal não.

3.1CLAUDE.md — hierarquia, escopo & organização modular
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: o CLAUDE.md dá contexto/instruções ao Claude Code, em 3 níveis: usuário (~/.claude/CLAUDE.md, só seu), projeto (CLAUDE.md na raiz ou .claude/CLAUDE.md, do time) e diretório (subpastas). Sempre carregado.
📦 O cenário
No Code Generation (Cenário 2), um novo dev não recebe as convenções do time. Causa: alguém colocou as instruções em ~/.claude/CLAUDE.md (escopo pessoal, não versionado). Instruções do time têm que estar no nível de projeto.
⚙️ Como funciona
📁 Onde cada coisa vai
  ~/.claude/CLAUDE.md          ← PESSOAL (não versionado, só você)
  CLAUDE.md  (raiz)            ← TIME (versionado ✓)
  .claude/CLAUDE.md            ← TIME (versionado ✓)
  backend/CLAUDE.md            ← específico do diretório
       └─ @../docs/standards.md   ← importa padrão compartilhado

A sintaxe de import é @caminho (ex.: @../docs/standards.md) — não existe uma palavra-chave @import. Ela mantém o arquivo modular (uma fonte de verdade compartilhada). Para arquivos enormes, prefira dividir em .claude/rules/ (ver 3.3). Use /memory para ver quais arquivos de memória estão carregados e diagnosticar comportamento inconsistente.

⚠️ No exame

"Membro novo do time não recebe as instruções" → estão no nível de usuário, deviam estar no nível de projeto (versionado). Esse é o diagnóstico de hierarquia mais cobrado.

3.2Slash commands & Skills
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: commands (.claude/commands/) são atalhos de prompt; skills (.claude/skills/ com SKILL.md) são workflows com frontmatter (context: fork, allowed-tools, argument-hint). Projeto = versionado; ~/.claude/ = pessoal.
📦 O cenário
No Code Generation, você cria um /review com o checklist de code review do time. Colocando em .claude/commands/review.md, todo dev que clonar o repo já tem o comando.
⚙️ Como funciona
SKILL.md com frontmatter
---
name: analyze-codebase
context: fork          # roda isolado, não polui a conversa principal
allowed-tools: [Read, Grep, Glob]   # restringe ferramentas
argument-hint: "<módulo a analisar>"
---
Analise o módulo indicado e retorne só um resumo estruturado.

context: fork é o ponto-chave: a skill roda num sub-contexto isolado, então output verboso (análise, brainstorm) não contamina a sessão principal. Skill = invocação sob demanda para um workflow; CLAUDE.md = padrões universais sempre carregados.

⚠️ No exame

"Comando disponível a todos ao clonar" → .claude/commands/ (versionado), não ~/.claude/commands/ nem CLAUDE.md nem um config.json inexistente. "Isolar output verboso" → context: fork.

3.3Regras com escopo de caminho (.claude/rules/)
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: arquivos em .claude/rules/ com paths: (glob) no frontmatter carregam só quando você edita arquivos que batem com o padrão — economizando contexto e aplicando convenções por tipo de arquivo, independente da pasta.
📦 O cenário
No Code Generation, arquivos de teste ficam ao lado do código (Button.test.tsx junto de Button.tsx), espalhados por toda parte. Você quer que TODOS sigam a mesma convenção, não importa onde estejam.
⚙️ Como funciona
.claude/rules/testing.md
---
description: Convenções de teste
paths:
  - "**/*.test.tsx"
  - "**/*.spec.tsx"
---
Use describe/it. Mocke só dependências externas. Cubra os casos-limite.

Vantagem sobre CLAUDE.md por subdiretório: o glob alcança arquivos do mesmo tipo espalhados pelo codebase — um CLAUDE.md de pasta só vale para aquela pasta.

⚠️ No exame

"Aplicar convenções a arquivos de teste espalhados, automaticamente" → .claude/rules/ com glob. As distratoras: CLAUDE.md raiz (depende de inferência), skills (exigem invocação manual), CLAUDE.md por subpasta (não pega arquivos espalhados).

3.4Plan mode vs execução direta
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: plan mode explora e projeta antes de mexer no código — para tarefas grandes/arquiteturais com múltiplos caminhos válidos. Execução direta é para mudanças simples e bem-definidas.
📦 O cenário
No Code Generation, reestruturar um monolito em microsserviços toca dezenas de arquivos e exige decidir limites de serviço. Plan mode deixa o Claude entender dependências e propor uma abordagem antes de mudar nada — evitando retrabalho caro.
🎯 Quando usar / quando não
TarefaPlan modeExecução direta
Monolito → microsserviços✅ arquitetural, multi-arquivo
Migrar 45+ arquivos para nova API✅ explorar deps antes
Adicionar validação numa função✅ escopo claro
Corrigir bug com stack trace claro✅ sem surpresas

Combine: plan mode para investigar, execução direta para implementar o plano. Use o subagente Explore para isolar a descoberta verbosa e devolver só resumos (preserva o contexto principal).

⚠️ No exame

Tarefa com implicação arquitetural / múltiplas abordagens → plan mode. A distratora "começar direto e mudar para plan mode se aparecer complexidade" ignora que a complexidade já está no enunciado.

3.5Refinamento iterativo
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: quando a prosa não comunica bem, use exemplos input/output concretos; itere por testes (escreva o teste, compartilhe a falha); use o padrão entrevista (peça ao Claude perguntar antes de implementar).
📦 O cenário
No Code Generation, você descreve uma transformação em prosa e o Claude interpreta de formas diferentes a cada vez. Dar 2-3 exemplos de entrada→saída fixa o comportamento. Para um domínio que você não domina, peça que ele te entreviste (estratégias de cache, modos de falha) antes de codar.
🎯 Quando usar / quando não
📥 Tudo numa mensagem
  • Quando os problemas interagem entre si
🔁 Sequencial
  • Quando os problemas são independentes
⚠️ No exame

Prosa gera resultado inconsistente → 2-3 exemplos input/output (não "explicar melhor em texto"). Bug em caso-limite → forneça um caso de teste específico com entrada e saída esperada.

3.6Claude Code em CI/CD
🖥️ Claude Code 🔌 Claude API 🧩 Agent SDK 🌐 MCP
Em uma frase: no pipeline, rode com -p (não-interativo, senão trava esperando input), use --output-format json + --json-schema para saída parseável, e dê contexto via CLAUDE.md.
📦 O cenário
No CI/CD (Cenário 5), seu script roda claude "Analyze this PR" e o job trava — está esperando input interativo. A correção é a flag -p.
⚙️ Como funciona
CI não-interativo
# ❌ trava aguardando input
claude "Analyze this PR for security issues"

# ✅ modo print + saída estruturada
claude -p "Analyze this PR" --output-format json --json-schema schema.json

Isolamento de sessão: a mesma sessão que gerou o código é pior para revisá-lo (carrega o raciocínio da geração). Use uma instância independente para revisar (liga com 4.6). Ao re-rodar revisões após novos commits, inclua os achados anteriores no contexto e peça para reportar só o que é novo/não resolvido — evita comentários duplicados.

⚠️ No exame

Job trava no CI → flag -p. As distratoras (CLAUDE_HEADLESS=true, --batch) são features inexistentes; redirecionar /dev/null é gambiarra que não resolve a sintaxe do Claude Code.

💬 Domínio 4: Prompt Engineering & Structured Output

20% do exame · ~12 questões

Como escrever prompts que produzem comportamento confiável e saída estruturada. Os critérios e few-shot vivem no 🔌 system prompt da sua app (ou no 🖥️ CLAUDE.md); a garantia de schema vem do tool_use na 🔌 API.

4.1Critérios explícitos (reduzir falsos positivos)
🔌 Claude API 🖥️ Claude Code 🧩 Agent SDK 🌐 MCP
Em uma frase: "seja conservador" não melhora precisão — critérios categóricos explícitos sim. Diga exatamente o que reportar e o que ignorar.
📦 O cenário
No CI/CD, a revisão automática de comentários gera muitos falsos positivos. Instrução vaga ("verifique se os comentários são precisos, seja conservador") não diz ao modelo o que é alta confiança no SEU código.
⚙️ Como funciona
Vago → explícito
❌ "Seja conservador ao reportar."

✅ Reporte SOMENTE quando:
   - o comentário contradiz o comportamento do código
   - um parâmetro documentado não existe na assinatura
   NÃO reporte:
   - estilo informal mas semanticamente correto
   - ausência de documentação (ausência ≠ erro)

Categorias com alto falso positivo derrubam a confiança nas categorias boas — vale desativar temporariamente a categoria ruim enquanto melhora o prompt dela. Defina severidade com exemplos de código concretos para cada nível.

⚠️ No exame

Falsos positivos altos → critérios categóricos explícitos. "Peça ao modelo só reportar alta confiança" / "seja conservador" são as distratoras — instruções vagas não calibram precisão.

4.2Few-shot prompting
🔌 Claude API 🖥️ Claude Code 🧩 Agent SDK 🌐 MCP
Em uma frase: few-shot é a técnica mais eficaz quando instruções detalhadas ainda dão saída inconsistente. Mostre 2-4 exemplos input→output (com o raciocínio) e o modelo generaliza para casos novos.
📦 O cenário
No Structured Data Extraction (Cenário 6), documentos vêm em formatos variados (citações inline vs bibliografia; medidas informais como "uns 90kg"). Exemplos mostrando como tratar cada caso ambíguo reduzem alucinação e fixam o formato de saída.
🎯 Quando usar / quando NÃO
Few-shot resolveFew-shot NÃO é a resposta
Saída inconsistente após instruções detalhadasMisrouting entre tools similares → tool descriptions (2.1)
Casos ambíguos que regras não cobremCampo ausente no documento → nullable (4.3)
Formato que a prosa não descreve bemSequência crítica de ferramentas → hooks (1.4)
⚠️ No exame

Few-shot é forte para ambiguidade e formato, e também para calibrar escalação (1.5/5.2). Mas NÃO use few-shot para misrouting de tools (use descrições) nem para campos ausentes (use nullable).

4.3Output estruturado via tool_use & JSON schema
🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code 🌐 MCP
Em uma frase: tool_use com JSON schema é a forma mais confiável de garantir saída estruturada — elimina erros de sintaxe. Mas NÃO previne erros semânticos (line items que não somam ao total, valor no campo errado).
📦 O cenário
No Structured Data Extraction, você define uma ferramenta de extração cujo input é o schema desejado, e lê os dados do bloco tool_use da resposta. Campos que podem não existir no documento devem ser nullable — senão o modelo inventa valor para satisfazer o "required".
⚙️ Como funciona
Schema com nullable + enum extensível
{
  "invoice_total": { "type": "number" },
  "po_number":    { "type": ["string", "null"] },   # null se ausente
  "category":     { "enum": ["goods", "service", "other"] },
  "category_detail": { "type": ["string", "null"] }  # para "other"
}

tool_choice aqui: "any" garante saída estruturada quando há vários schemas e o tipo do doc é desconhecido; forçar {"type":"tool","name":"..."} garante que uma extração rode antes de etapas de enriquecimento.

⚠️ No exame

Campo que pode faltar no documento → nullable (não required, senão fabrica). Schema rígido elimina erro de sintaxe mas não semântico — para semântica, valide (4.4). Enum com "other" + detalhe para categorias extensíveis; "unclear" para ambíguos.

4.4Validação, retry & feedback
🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code 🌐 MCP
Em uma frase: retry-com-feedback (reenviar com o erro de validação específico) corrige erros de formato/estrutura — mas é inútil quando a informação simplesmente não está no documento fonte.
📦 O cenário
No Structured Data Extraction, a validação Pydantic falha porque os line_items não somam ao total. Você reenvia incluindo o documento, a extração que falhou e o erro exato — o modelo se corrige. Mas se o campo que falta não existe no doc, nenhum retry resolve.
🎯 Quando retry funciona / não funciona
✅ Retry resolve
  • Formato errado (DD/MM vs MM/DD)
  • Itens em ordem errada, valor somado errado
❌ Retry NÃO resolve
  • Informação ausente da fonte → use nullable
  • Dado só existe em outro doc não fornecido

Padrões úteis: extrair calculated_total junto de stated_total para flagar divergência; conflict_detected para fonte inconsistente; detected_pattern para analisar falsos positivos depois.

⚠️ No exame

Distinga erro de estrutura (retry resolve) de informação ausente (retry não resolve — use nullable). Esse é o par mais cobrado do 4.4.

4.5Batch processing (Message Batches API)
🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code 🌐 MCP
Em uma frase: a Message Batches API dá 50% de desconto, janela de até 24h e sem SLA de latência. Ótima para cargas tolerantes a atraso; péssima para fluxos bloqueantes.
📦 O cenário
No CI/CD, há dois fluxos: (1) check pré-merge bloqueante (dev espera) e (2) relatório de dívida técnica gerado overnight. O gestor quer migrar os dois para batch pelo desconto. Só o (2) deve ir para batch.
🎯 Quando usar / quando não
✅ Batch
  • Relatórios overnight, auditorias semanais, geração de testes noturna
  • Correlacione request/response por custom_id
❌ Síncrono (não batch)
  • Checks pré-merge bloqueantes
  • Qualquer coisa onde alguém espera o resultado

Batch não suporta tool calling multi-turno num mesmo request. Em falhas, reenvie só os custom_id que falharam (ex.: dividindo docs que estouraram o contexto). Refine o prompt num set de amostra antes de processar o volume todo.

⚠️ No exame

Migrar fluxo bloqueante para batch = errado (sem SLA). "Polling porque geralmente é rápido" não vale para bloqueante. Reprocessar falhas = só os custom_id que falharam, não o lote inteiro.

4.6Arquiteturas de revisão multi-instância & multi-passagem
🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code 🌐 MCP
Em uma frase: um modelo é ruim em revisar o próprio código (carrega o raciocínio da geração). Use uma instância independente para revisar; e divida revisões grandes em passagens (por-arquivo + integração) para evitar diluição de atenção.
📦 O cenário
No CI/CD, um PR de 14 arquivos revisado numa passagem só dá resultado inconsistente: detalhado em uns, superficial em outros, bugs óbvios passam, e feedback contraditório (reprova um padrão num arquivo, aprova idêntico em outro).
🎯 Como reestruturar
✅ Faça
  • Passagem por-arquivo (issues locais) + passagem de integração (fluxo entre arquivos)
  • Revisor independente, sem o contexto de raciocínio do gerador
❌ Não
  • "Janela de contexto maior" — não resolve qualidade de atenção
  • Consenso de 3 runs — suprime bugs detectados intermitentemente
⚠️ No exame

Revisão inconsistente em muitos arquivos → dividir em passagens focadas. "Modelo com janela maior" é a distratora favorita — o problema é diluição de atenção, não tamanho de janela.

🧠 Domínio 5: Context Management & Reliability

15% do exame · ~9 questões

Como manter qualidade e confiabilidade quando as conversas ficam longas, erros acontecem em cadeia e dados precisos precisam sobreviver. Vive na sua app 🧩 SDK (o que entra no contexto a cada chamada) e no 🖥️ Claude Code (/compact, sessões).

5.1Preservar informação crítica em conversas longas
🧩 Agent SDK 🖥️ Claude Code 🔌 Claude API 🌐 MCP
Em uma frase: resumir progressivamente apaga números, datas e valores exatos. Extraia os fatos transacionais para um bloco "case facts" que vai em todo prompt, fora do histórico resumido — e cuide do efeito "lost in the middle".
📦 O cenário
No Customer Support Agent, numa sessão longa, "cobrado R$249,90 em 15/03, pedido ORD-789" vira "o cliente teve problemas recentes". Você perde o que importa para resolver o caso.
⚙️ Como funciona
Bloco CASE FACTS (nunca sumarizado)
---CASE FACTS---
Cliente: CUS-4829 | ORD-789 cobrado 2x em 15/03 | R$249,90
Pontos: 450 | Reclamação: cobrança dupla
---FIM---   # atualizado a cada tool call, incluído em todo prompt

Lost in the middle: o modelo lê bem o início e o fim de inputs longos, mas pode ignorar o meio. Coloque achados críticos no início, com headers explícitos. E apare tool results verbosos (40+ campos quando só 5 importam) com um hook PostToolUse antes que acumulem.

⚠️ No exame

Perda de precisão em sessão longa → bloco de fatos persistente fora do resumo. Achados importantes "no meio" sendo ignorados → reposicionar no início com headers. Tool result gigante → aparar para os campos relevantes.

5.2Escalação & resolução de ambiguidade
🧩 Agent SDK 🔌 Claude API 🖥️ Claude Code 🌐 MCP
Em uma frase: escale quando o cliente pede humano, quando a política é omissa/ambígua, ou quando não há progresso possível. Score de confiança e análise de sentimento NÃO são bons proxies de complexidade.
📦 O cenário
No Customer Support Agent, o agente atinge só 55% de resolução (meta 80%): escala casos simples (troca com foto) e tenta resolver sozinho casos que exigem exceção de política. O limite de decisão está mal definido.
⚙️ Como funciona

A correção é adicionar critérios explícitos de escalação com few-shot ao system prompt, mostrando quando escalar vs resolver. Honre pedido explícito de humano imediatamente (sem investigar antes). Quando a busca retorna múltiplos clientes, peça um identificador adicional em vez de chutar por heurística.

⚠️ No exame

Calibrar escalação → critérios explícitos + few-shot. As distratoras: score de confiança (o LLM já está confiante errado nos casos difíceis), classificador treinado (over-engineering antes de tentar prompt) e sentimento (não correlaciona com complexidade).

5.3Propagação de erros em multi-agente
🧩 Agent SDK 🌐 MCP 🔌 Claude API 🖥️ Claude Code
Em uma frase: ao subir um erro ao coordenador, mande contexto estruturado (tipo de falha, query tentada, resultados parciais, alternativas). Status genérico esconde informação; suprimir o erro ou matar o workflow inteiro são anti-padrões.
📦 O cenário
No Multi-Agent Research, o subagente de busca dá timeout. Se ele só retornar "search unavailable", o coordenador não sabe se vale tentar outra query, seguir com parciais ou trocar de abordagem.
🎯 Como fazer / não fazer
✅ Faça
  • Erro estruturado: falha + tentativa + parciais + alternativas
  • Subagente resolve transitório localmente; só propaga o que não resolve
  • Anote cobertura na síntese (o que está bem suportado vs lacunas)
❌ Não
  • Status genérico ("search unavailable")
  • Marcar falha como sucesso (vazio = ok)
  • Terminar todo o workflow por uma falha
⚠️ No exame

Timeout de subagente → contexto de erro estruturado ao coordenador. Distinga falha de acesso (timeout, precisa decisão) de resultado vazio válido (consulta ok, sem matches).

5.4Contexto em exploração de codebase grande
🖥️ Claude Code 🧩 Agent SDK 🔌 Claude API 🌐 MCP
Em uma frase: em sessões longas o contexto degrada (o modelo passa a citar "padrões típicos" em vez das classes específicas). Use scratchpad files, delegue exploração verbosa a subagentes, e persista estado para recuperação de crash.
📦 O cenário
No Developer Productivity, explorando um sistema legado por horas, o Claude começa a dar respostas genéricas. Você faz ele gravar achados num scratchpad e delega "ache todos os testes" / "trace o fluxo de reembolso" a subagentes, mantendo o agente principal na coordenação de alto nível.
⚙️ Ferramentas
  • Scratchpad: arquivo que persiste achados-chave entre fronteiras de contexto
  • Subagentes: isolam output verboso; resumo volta ao principal
  • Manifesto de estado: cada agente exporta estado para um local conhecido; o coordenador recarrega no resume (recuperação de crash)
  • /compact: comprime o contexto quando enche de descoberta verbosa
⚠️ No exame

"Sistema crasha após 8 de 20 arquivos convertidos" → manifesto JSON de progresso (cada agente escreve estado após cada item). Degradação em sessão longa → scratchpad + delegação a subagentes + /compact.

5.5Revisão humana & calibração de confiança
🔌 Claude API 🧩 Agent SDK 🖥️ Claude Code 🌐 MCP
Em uma frase: precisão agregada (97%) pode esconder desempenho ruim num tipo de documento ou campo específico. Valide por tipo de documento E por campo antes de automatizar; use amostragem estratificada.
📦 O cenário
No Structured Data Extraction, a extração tem 97% de precisão global — mas em faturas manuscritas o campo invoice_total erra 40%. Remover a revisão humana confiando no número agregado seria um desastre nesse segmento.
⚙️ Como funciona
  • Amostragem estratificada de extrações de alta confiança (amostra puramente aleatória sub-representa categorias raras de alto erro)
  • Analisar precisão por tipo de doc e por campo antes de reduzir revisão
  • Modelo emite confiança por campo; calibre os limiares com um set rotulado
  • Roteie baixa confiança / fonte contraditória para revisão humana
⚠️ No exame

Ver 97% agregado e automatizar = a armadilha. Sempre segmente por tipo de documento e por campo. Score de confiança aqui é útil se calibrado com dados rotulados (diferente da escalação em 5.2, onde confiança auto-reportada não serve).

5.6Proveniência & incerteza em síntese multi-fonte
🧩 Agent SDK 🔌 Claude API 🖥️ Claude Code 🌐 MCP
Em uma frase: a atribuição da fonte se perde quando os achados são comprimidos sem manter o mapeamento claim→fonte. Exija mapeamentos estruturados que a síntese preserva; anote conflitos com a fonte em vez de escolher um valor arbitrário.
📦 O cenário
No Multi-Agent Research, duas fontes credíveis dão estatísticas diferentes. A síntese não deve escolher uma — deve manter ambas com atribuição. E datas: censo de 2018 vs estimativa de 2023 não é contradição, é evolução temporal.
⚙️ Como funciona
  • Subagentes emitem claim + trecho de evidência + URL/nome do doc + data de publicação
  • Estruture o relatório separando achados bem estabelecidos dos contestados
  • Análise entrega valores conflitantes anotados; o coordenador decide como reconciliar
  • Renderize cada tipo de conteúdo no formato certo (financeiro = tabela, notícia = prosa)
⚠️ No exame

Estatísticas conflitantes de fontes credíveis → preservar ambas com atribuição (não escolher uma). Exigir datas nos outputs evita que diferença temporal seja lida como contradição.

Domínio 1 — Agentic Architecture & Orchestration (27%)

1.1Projetar e implementar agentic loops para execução autônoma de tarefas
  • O ciclo de vida do loop: enviar request → inspecionar stop_reason ("tool_use" vs "end_turn") → executar a ferramenta pedida → devolver o resultado para a próxima iteração.
  • Resultados de ferramentas são anexados ao histórico da conversa para o modelo raciocinar sobre a próxima ação; a decisão é model-driven, não uma árvore de decisão pré-configurada.
  • Anti-padrões: parsear linguagem natural para decidir a parada, usar cap de iterações como mecanismo primário, ou checar o texto do assistant como indicador de conclusão.
1.2Orquestrar sistemas multi-agente com o padrão coordinator-subagent
  • Arquitetura hub-and-spoke: o coordinator gerencia toda a comunicação entre subagents, o tratamento de erros e o roteamento de informação.
  • Subagents têm contexto isolado — não herdam o histórico do coordinator automaticamente.
  • Papel do coordinator: decomposição de tarefas, delegação, agregação de resultados e escolha de quais subagents invocar. Risco: decomposição estreita demais → cobertura incompleta.
1.3Configurar invocação de subagents, passagem de contexto e spawning
  • A Task tool é o mecanismo para spawnar subagents; allowedTools precisa incluir "Task" para o coordinator conseguir delegar.
  • Contexto do subagent tem que vir explícito no prompt — nada é herdado nem compartilhado entre invocações. Use formatos estruturados para separar conteúdo de metadados (URLs de fonte, nomes de documento, páginas).
  • Paralelismo: emita várias Task calls numa única resposta do coordinator (não em turnos separados). AgentDefinition define descrição, system prompt e restrição de tools por subagent.
1.4Implementar workflows multi-etapa com enforcement e handoff
  • Diferença entre enforcement programático (hooks, prerequisite gates) e orientação por prompt para ordenar o workflow.
  • Quando é preciso compliance determinístico (ex: verificar identidade antes de operação financeira), instruções de prompt têm taxa de falha diferente de zero — use um gate que bloqueia process_refund até get_customer retornar um ID verificado.
  • Protocolos de handoff estruturado para escalação no meio do processo: incluir dados do cliente, root cause e ação recomendada.
1.5Aplicar Agent SDK hooks para interceptar tool calls e normalizar dados
  • PostToolUse hooks interceptam resultados de ferramentas para transformá-los antes do modelo processar (ex: normalizar Unix timestamp, ISO 8601 e códigos numéricos vindos de MCP tools diferentes).
  • Para bloquear uma ação que viola regra de negócio (ex: refund acima de $500), use PreToolUse — roda antes da ferramenta e pode negar (permissionDecision: "deny"). PostToolUse roda depois, então não serve para bloquear.
  • Escolha hooks quando a regra exige garantia determinística; prompt serve para compliance apenas probabilístico.
1.6Projetar estratégias de decomposição de tarefas para workflows complexos
  • Quando usar pipeline sequencial fixo (prompt chaining) vs decomposição dinâmica e adaptativa baseada nos achados intermediários.
  • Prompt chaining para revisões previsíveis (analisar cada arquivo, depois um passe de integração cross-file).
  • Planos de investigação adaptativos que geram subtarefas conforme as dependências vão sendo descobertas (ex: "adicionar testes a um codebase legado").
1.7Gerenciar estado de sessão, retomada e forking
  • --resume <session-name> (ou -c/--continue) retoma uma conversa; --fork-session cria um novo id ao retomar (preservando o original) e /fork abre um subagente que herda a conversa — para explorar abordagens divergentes.
  • Ao retomar após mudanças em arquivos, avise o agente sobre o que mudou para re-análise direcionada.
  • Começar uma sessão nova com um resumo estruturado é mais confiável do que dar --resume arrastando tool results desatualizados.

Domínio 2 — Tool Design & MCP Integration (18%)

2.1Projetar interfaces de tools com descrições e limites claros
  • A description da tool é o mecanismo primário que o LLM usa para escolher qual chamar; descrição mínima = seleção não confiável entre tools parecidas.
  • Inclua na description: formatos de input, exemplos de query, edge cases e limites (quando usar esta vs a alternativa). Descrições sobrepostas causam misrouting.
  • Renomeie tools para eliminar sobreposição e divida uma tool genérica em específicas (ex: analyze_document → extract_data_points, summarize_content, verify_claim_against_source).
2.2Implementar respostas de erro estruturadas para MCP tools
  • O flag isError comunica falhas ao agente. Distinga erros transient (timeout, indisponibilidade), de validation (input inválido), de business (violação de política) e de permission.
  • Retorne metadados dentro do conteúdo do resultado — só isError é campo do protocolo MCP; errorCategory, isRetryable boolean e descrição legível são convenção da sua app. Erro genérico ("Operation failed") impede o agente de decidir a recuperação.
  • Subagents fazem recuperação local para falhas transitórias e só propagam ao coordinator o que não conseguem resolver. Distinga access failure (precisa decidir retry) de resultado vazio válido (query com zero matches).
2.3Distribuir tools entre agents e configurar tool choice
  • Dar tools demais a um agent (ex: 18 em vez de 4-5) degrada a confiabilidade da seleção. Dê a cada agent só as tools do seu papel (scoped access).
  • Substitua tools genéricas por versões restritas (ex: fetch_url → load_document que valida URLs).
  • tool_choice: "auto" (pode responder texto), "any" (obrigado a chamar alguma tool), e forçado ({"type":"tool","name":"..."}) para garantir uma tool específica primeiro.
2.4Integrar MCP servers no Claude Code e nos workflows de agentes
  • Escopo: .mcp.json (projeto, compartilhado com o time via version control) vs ~/.claude.json (pessoal/experimental).
  • Expansão de variável de ambiente em .mcp.json (ex: ${GITHUB_TOKEN}) para não commitar segredos. Tools de todos os servers configurados ficam disponíveis simultaneamente.
  • MCP resources expõem catálogos de conteúdo (issues, hierarquia de docs, schema de banco) para reduzir tool calls exploratórias. Prefira MCP servers da comunidade a implementações custom para integrações padrão (ex: Jira).
2.5Selecionar e usar built-in tools (Read, Write, Edit, Bash, Grep, Glob)
  • Grep = busca por conteúdo (nomes de função, mensagens de erro, imports). Glob = busca por nome/caminho (ex: **/*.test.tsx).
  • Read/Write para arquivo inteiro; Edit para mudança pontual com texto único. Se Edit falha por texto não-único, use Read + Write.
  • Entenda o codebase incrementalmente: Grep para achar entry points, Read para seguir imports e rastrear fluxos — em vez de ler tudo de uma vez.

Domínio 3 — Claude Code Configuration & Workflows (20%)

3.1Configurar CLAUDE.md com hierarquia, escopo e organização modular
  • Hierarquia: user-level (~/.claude/CLAUDE.md), project-level (.claude/CLAUDE.md ou CLAUDE.md na raiz) e directory-level (subpastas). O user-level NÃO é compartilhado com o time via version control.
  • @caminho (ex.: @../docs/standards.md, sem a palavra "import") referencia arquivos externos para manter o CLAUDE.md modular. .claude/rules/ organiza regras por tópico em vez de um CLAUDE.md monolítico.
  • Use /memory para ver quais arquivos de memória estão carregados e diagnosticar comportamento inconsistente.
3.2Criar e configurar slash commands e skills customizados
  • Commands em .claude/commands/ (projeto, versionado) vs ~/.claude/commands/ (pessoal). Skills em .claude/skills/ com SKILL.md.
  • Frontmatter da skill: context: fork (roda em sub-agente isolado, sem poluir a conversa principal), allowed-tools (restringe tools na execução) e argument-hint.
  • Escolha entre skill (invocação sob demanda para workflow específico) e CLAUDE.md (padrões universais sempre carregados).
3.3Aplicar path-specific rules para carregamento condicional de convenções
  • Arquivos em .claude/rules/ com frontmatter YAML paths (glob) carregam só quando você edita arquivos que casam com o padrão — reduz contexto irrelevante e tokens.
  • Vantagem do glob sobre CLAUDE.md por subdiretório: aplica convenções a arquivos por tipo, espalhados pelo codebase (ex: **/*.test.tsx para todos os testes).
3.4Decidir entre plan mode e execução direta
  • Plan mode: tarefas complexas com mudanças em larga escala, múltiplas abordagens válidas, decisões arquiteturais e edição multi-arquivo (ex: migração de 45+ arquivos, monolito → microsserviços).
  • Execução direta: mudanças simples e bem-escopadas (ex: um bug fix de um arquivo, adicionar uma validação).
  • O subagent Explore isola output verboso de descoberta e devolve resumos, preservando o contexto da conversa principal.
3.5Aplicar técnicas de refinamento iterativo
  • Exemplos concretos de input/output são a forma mais eficaz de comunicar transformações quando a prosa é interpretada de forma inconsistente.
  • Iteração test-driven: escrever a suíte de testes primeiro e iterar compartilhando as falhas. Interview pattern: fazer o Claude perguntar antes de implementar.
  • Vários problemas que interagem entre si → tudo numa mensagem só; problemas independentes → correção sequencial.
3.6Integrar Claude Code em pipelines de CI/CD
  • -p (--print) roda em modo não-interativo (evita travar aguardando input). --output-format json + --json-schema para saída estruturada e parseável.
  • CLAUDE.md fornece o contexto do projeto (padrões de teste, fixtures, critérios de review) ao Claude Code invocado no CI.
  • Isolamento de sessão: a mesma sessão que gerou o código é pior para revisá-lo — use uma instância de review independente. Ao re-rodar review, passe os achados anteriores e peça só issues novos.

Domínio 4 — Prompt Engineering & Structured Output (20%)

4.1Projetar prompts com critérios explícitos para reduzir falsos positivos
  • Critérios explícitos batem instruções vagas (ex: "flag um comentário só quando o comportamento descrito contradiz o código real" > "verifique se os comentários estão corretos").
  • "Seja conservador" ou "só reporte alta confiança" não melhoram precisão — critérios categóricos específicos, sim.
  • Categorias com muito falso positivo minam a confiança do dev nas categorias corretas; defina severidade com exemplos de código concretos.
4.2Aplicar few-shot prompting para consistência e qualidade
  • Few-shot é a técnica mais eficaz quando instruções detalhadas ainda produzem output inconsistente; use 2-4 exemplos alvo mostrando o raciocínio da escolha.
  • Demonstre o handling de casos ambíguos e o formato de saída desejado (location, issue, severity, suggested fix).
  • Reduz alucinação em extração (medidas informais, estruturas de documento variadas) e permite generalizar para padrões novos, não só casar os pré-especificados.
4.3Forçar output estruturado com tool use e JSON schemas
  • tool_use com JSON schema é a forma mais confiável de garantir output válido — elimina erro de sintaxe JSON, mas NÃO evita erro semântico (ex: line items que não somam ao total).
  • tool_choice: "auto" / "any" / forçado. Campos que podem não existir no documento devem ser nullable — senão o modelo fabrica valor.
  • Use enum com "unclear" e "other" + campo de detalhe para categorias extensíveis; inclua regras de normalização de formato no prompt.
4.4Implementar validação, retry e feedback loops
  • Retry-with-error-feedback: no retry, anexe o documento, a extração falha e o erro de validação específico para guiar a correção.
  • Retry é inútil quando a informação simplesmente não existe na fonte (≠ erro de formato/estrutura, esses o retry resolve).
  • Fluxos de auto-correção: extrair calculated_total junto de stated_total para flagar divergência; campo detected_pattern para analisar falsos positivos.
4.5Projetar estratégias eficientes de batch processing
  • Message Batches API: 50% de economia, janela de até 24h, sem SLA de latência garantido. Boa para workloads tolerantes a latência (relatórios noturnos), ruim para bloqueantes (pre-merge check).
  • Não suporta multi-turn tool calling num único request. Use custom_id para correlacionar request/response.
  • Ao falhar, reenvie só os documentos que falharam (por custom_id) com ajustes (ex: chunking dos que estouraram o contexto).
4.6Projetar arquiteturas de review multi-instância e multi-pass
  • Auto-review é limitado: a mesma instância retém o raciocínio da geração e questiona menos as próprias decisões. Uma instância independente pega mais issues sutis.
  • Multi-pass: divida reviews grandes em passes por arquivo (issues locais) + um passe de integração cross-file, evitando diluição de atenção e feedback contraditório.

Domínio 5 — Context Management & Reliability (15%)

5.1Gerenciar contexto para preservar informação crítica em interações longas
  • Risco da sumarização progressiva: condensar valores, percentuais, datas e expectativas do cliente em resumos vagos. Extraia isso para um bloco "case facts" persistente, fora do histórico sumarizado.
  • Efeito "lost in the middle": o modelo processa bem o início e o fim de inputs longos, mas pode omitir achados do meio — coloque o crítico no começo, com headers explícitos.
  • Tool results acumulam e consomem tokens desproporcionalmente (40+ campos quando só 5 importam) — apare para os campos relevantes antes de acumular.
5.2Projetar padrões de escalação e resolução de ambiguidade
  • Gatilhos legítimos de escalação: o cliente pede um humano, a política é omissa/ambígua (não só "caso complexo"), ou o agente não consegue progredir.
  • Sentimento e self-reported confidence score são proxies ruins de complexidade real — o agente já erra "confiante" nos casos difíceis. Use critérios explícitos + few-shot.
  • Múltiplos matches de cliente → peça identificador adicional em vez de escolher por heurística.
5.3Implementar propagação de erros em sistemas multi-agente
  • Contexto de erro estruturado (tipo de falha, query tentada, resultados parciais, alternativas) permite ao coordinator decidir a recuperação de forma inteligente.
  • Status genérico ("search unavailable") esconde contexto valioso. Distinga access failure de resultado vazio válido.
  • Anti-padrões: suprimir o erro silenciosamente (retornar vazio como sucesso) e derrubar o workflow inteiro numa única falha.
5.4Gerenciar contexto na exploração de codebases grandes
  • Degradação em sessões longas: o modelo passa a dar respostas inconsistentes e a citar "padrões típicos" em vez das classes específicas descobertas antes.
  • Use scratchpad files para persistir achados-chave; delegue a subagents a exploração verbosa enquanto o agente principal coordena.
  • State persistence estruturado (manifest) para crash recovery; /compact para reduzir contexto em sessões longas.
5.5Projetar workflows de revisão humana e calibração de confiança
  • Precisão agregada (ex: 97%) pode esconder desempenho ruim em tipos de documento ou campos específicos — valide por tipo E por campo antes de automatizar.
  • Amostragem aleatória estratificada para medir erro em extrações de alta confiança e detectar padrões novos.
  • Confidence score por campo calibrado com validation set rotulado; roteie baixa confiança / fontes ambíguas para revisão humana.
5.6Preservar provenance e lidar com incerteza na síntese multi-fonte
  • A atribuição de fonte se perde na sumarização quando os achados são comprimidos sem preservar o mapeamento claim→source. Exija claim-source mappings estruturados que a síntese preserva.
  • Estatísticas conflitantes de fontes confiáveis: anote o conflito com atribuição em vez de escolher um valor arbitrariamente.
